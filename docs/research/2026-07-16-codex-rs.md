# 레퍼런스 노트 — openai/codex의 codex-rs (M8 Task9)

M8 스펙 §7("레퍼런스 탐구")의 1차 대상으로, codex-rs는 유일하게 유효한 동종
코드 탐구 대상으로 선정됐다(스펙 §1: 같은 언어·같은 OpenAI-호환 프로토콜·로컬
모델 `--oss` 지원, Rust 96.7%·Apache-2.0·소스 공개). 이 노트는 `gh api
repos/openai/codex/contents/...`로 2026-07-17 `main` 브랜치에서 직접 가져온
소스(`codex-rs/core/src/compact.rs`·`session/context_window.rs`·
`session/token_budget.rs`·`apply-patch/src/{parser,seek_sequence}.rs`·
`ollama/src/lib.rs`·`lmstudio/src/lib.rs`·`tools/handlers/*` 등)만 근거로
쓴다. 소스에서 직접 확인한 내용만 적었고, 추측은 명시했다.

## 1. 컨텍스트 압축/이력 관리

**트리거**(`session/context_window.rs::context_window_token_status`,
`session/token_budget.rs::maybe_record`): bytes/4 추정이 아니라 **API 응답의
실사용 토큰 수**(`sess.get_total_token_usage()`)를 근거로 삼는다. 두 축을
동시에 본다 — ① 모델별 `auto_compact_token_limit`(스코프는
`AutoCompactTokenLimitScope::Total`/`BodyAfterPrefix` 설정 가능, 후자는
"프리픽스 이후 추가된 토큰만" 계산) ② 모델의 실제 `full_context_window_limit`
(하드 캡, 스코프 설정과 무관하게 도달 시 강제 압축). 압축 임계값 도달 전에는
`reminder_threshold_tokens` 이하로 남으면 모델에게 "남은 토큰이 얼마다"라는
1회성 리마인더 메시지를 대화에 삽입하고, 남은 토큰이 정확히 0이 되면
`auto_compact_fallback_prompt`를 삽입하는 2단계 경보 체계가 있다(둘 다
`state.claim_*` 플래그로 런당 1회만 발동).

**압축 방식**(`compact.rs::run_compact_task_inner`): loco의 `pack()`처럼
과거 항목을 그냥 지우는 게 아니라 **모델에게 요약을 시킨다** —
`SUMMARIZATION_PROMPT`(`prompts/templates/compact/prompt.md`)로 실제 LLM
호출을 한 번 더 보내 "진행상황·핵심 결정·제약/선호·다음 단계·계속하는 데
필요한 구체 데이터"를 담은 핸드오프 요약을 생성시킨다. 요약 메시지는
`SUMMARY_PREFIX`로 마킹되어(`is_summary_message`) 이후 압축 패스가 이미
요약된 턴을 재요약하지 않게 걸러낸다. 일부 프로바이더는
`should_use_remote_compact_task`(`provider.supports_remote_compaction()`)로
압축 자체를 서버 쪽에 위임하기도 한다(세부는 미조사).

**무엇이 살아남는가**(`build_compacted_history`/
`insert_initial_context_before_last_real_user_or_summary`): (1) 세션의
"초기 컨텍스트"(환경/월드 상태)는 압축 전 이력에서 제거됐다가 정확한
위치(마지막 진짜 사용자 메시지 앞, 없으면 요약 앞)에 재주입된다 — 턴 중간
압축(`BeforeLastUserMessage`)과 턴 시작 전 압축(`DoNotInject`, 다음 정규
턴이 어차피 재주입)이 이 지점에서 갈린다. (2) 가장 최근 **실제 사용자
메시지**들은 원문 그대로 유지하되 `COMPACT_USER_MESSAGE_MAX_TOKENS=20_000`
예산 안에서 최신 것부터 채우고 넘치면 토큰 단위로 잘라낸다
(`truncate_text`+`approx_token_count`). (3) 마지막에 요약 텍스트 1개를
붙인다. 즉 결과 이력은 `[초기 컨텍스트] + [최근 사용자 메시지 N개] +
[요약]` 형태다.

## 2. 도구 표면(apply_patch·검색 설계)

**전체 도구 목록**(`core/src/tools/handlers/`): `apply_patch`, `shell`/
`unified_exec`(세션형 실행, PTY 옵션), `mcp`/`mcp_resource`, `multi_agents`/
`multi_agents_v2`, `tool_search`, `plan`, `request_user_input`, `view_image`,
`get_context_remaining`, `new_context_window`, `sleep`,
`wait_for_environment`, `current_time`, `agent_jobs`,
`list_available_plugins_to_install`/`request_plugin_install`, `dynamic`/
`extension_tools` 등.

**apply_patch 포맷**(`apply-patch/src/parser.rs` 1-25행 Lark 문법 주석):
커스텀 V4A 방언 — `*** Begin Patch` / `*** Add File:` / `*** Delete File:` /
`*** Update File:` / `*** Move to:` / `@@ 컨텍스트줄` / `*** End of File` /
`*** End Patch`. GPT-5급 모델에는 **Freeform 툴**(JSON이 아니라 Lark 문법으로
제약된 커스텀 툴 타입, `apply_patch_spec.rs`)로 노출된다 — 설명 문구가
"do not wrap the patch in JSON"이라고 명시할 만큼 일반 함수 호출과 다른
경로다.

**퍼지 매칭**(`apply-patch/src/seek_sequence.rs::seek_sequence`, 각 청크의
`old_lines`/`change_context`를 원본 파일에서 찾는 함수): **4단계 사다리** —
① 완전 일치 ② 줄 끝 공백만 무시(rstrip) ③ 앞뒤 공백 모두 무시(trim) ④
유니코드 문장부호 정규화(스마트 대시류 7종→`-`, 스마트 따옴표류→`'`/`"`,
넌브레이킹 스페이스 등 11종→일반 공백) — 주석이 "이는 git apply의 퍼지
동작을 모방한다"고 명시. `is_end_of_file` 청크는 파일 끝에서부터 먼저
탐색한다. 호출부(`apply-patch/src/lib.rs` ~720-795행)에는 추가 안전장치가
하나 더 있다 — 매칭 실패 시 `old_lines`/`new_lines`의 마지막 빈 줄(파일
끝 개행을 나타냄)을 제거하고 재시도한다. **모델별 파싱 관용**
(`parser.rs::ParseMode::Lenient`): GPT-4.1이 `apply_patch`를 `local_shell`
함수 호출로 보낼 때 실제로는 셸이 heredoc을 해석해주지 않는데(execvpe라서)
`<<'EOF' ... EOF` 형태로 감싸 보내는 버릇이 알려져 있어, lenient 모드가 이
heredoc 마커를 감지해 벗겨낸다 — 주석은 "특정 모델만 이렇게 하지만 모든
모델에 관용 모드를 허용하기로 했다"고 명시(코드상 `PARSE_IN_STRICT_MODE =
false`로 사실상 상시 적용).

**검색/읽기 설계**: 구조화된 `read_file`/`grep`/`list_files` 툴이 **없다**.
파일 열람·검색은 전부 범용 `shell`/`unified_exec` 툴(자유 `cmd` 문자열,
선택적 PTY, 기본 10000토큰 출력 예산)을 통해 모델이 직접 `cat`/`rg` 등을
호출하는 방식으로 위임된다. 대신 **`tool_search`**(`tools/handlers/
tool_search.rs`)가 있다 — `bm25` 크레이트로 MCP/동적 툴들을 인덱싱해두고
`defer_loading: true`로 표시된 툴은 모델의 툴 목록에 상주하지 않다가
`tool_search` 쿼리로 검색해야 로드되는 구조(namespace 단위로 결과를 묶어
반환, 캐시는 `search_infos` 동일성 기준).

## 3. 소형/로컬 모델(`--oss`) 대응 흔적

전용 크레이트 2개 — `ollama`(`ollama/src/lib.rs`)와
`lmstudio`(`lmstudio/src/lib.rs`) — 각각 `ensure_oss_ready(config)`를
제공: 로컬 서버 접근성 확인 → 대상 모델이 로컬에 있는지 조회 → 없으면
자동 pull/download(LM Studio 쪽은 추가로 백그라운드에서 `load_model`까지
호출). 기본 OSS 모델명은 백엔드별로 다르다 — ollama는
`"gpt-oss:20b"`, LM Studio는 `"openai/gpt-oss-20b"`(같은 모델의 카탈로그
표기 차이). ollama 백엔드는 버전 게이팅이 있다 —
`ensure_responses_supported`가 Ollama ≥0.13.4(Responses API 지원 버전)를
요구하고 구버전이면 하드 에러로 업그레이드를 안내한다. `--oss` 플래그는
`show_raw_agent_reasoning`도 `true`로 켠다(`cli/src/main.rs`) — 로컬/OSS
모델은 추론 과정을 요약·은닉하지 않고 그대로 노출한다는 뜻으로 읽힌다.
**부정 확인**: gpt-oss 계열이 쓰는 OpenAI "Harmony" 응답 포맷에 대한 별도
클라이언트측 파싱/특수 처리는 Rust 소스에서 찾지 못했다(레포 전체
코드검색 결과 `harmony` 문자열이 `.rs` 파일에 전혀 없음) — 있다면 Ollama/
LM Studio 쪽 OpenAI-호환 셔틀이 흡수하는 것으로 보이나 이는 확인하지
못했다. `apply_patch`의 GPT-4.1 heredoc 관용도 "로컬 모델용"이 아니라
"특정 모델 버릇"에 대한 대응이라 이 절의 범위와는 결이 다르다.

## 4. loco에의 시사점

**압축**: codex는 압축 시점마다 LLM 호출을 한 번 더 태워 "요약"을 만드는
반면 loco의 `pack()`은 무료지만 손실적이다(오래된 툴 결과부터 축약, 그다음
오래된 사용자/어시스턴트 쌍을 통째로 버림). loco가 codex처럼 LLM 요약
압축을 도입하는 건 대기시간·소형 모델 요약 품질 불확실성 때문에 그대로
옮기기 어렵지만, **트리거 방식**은 훨씬 저렴하게 빌려올 수 있다 — codex는
실제 토큰 사용량을 능동적으로 추적해 임계값 전에 리마인더를 주입하는 반면
loco는 400 응답 문자열에 "context"가 포함됐는지 보는 **사후 반응형
휴리스틱**이다(요청이 이미 실패한 뒤에야 안다). 이는 M8 스펙 §8 백로그
"① 토큰 추정 보정"과 정확히 맞닿는다 — bytes/4 대신 실사용 토큰(API가
`usage` 필드를 돌려준다면)을 추적해 임계값 이전에 선제 축약하는 경로를
만드는 것이 다음 후보다.

**apply_patch 퍼지 매칭**: codex의 4단계 사다리(정확→줄끝공백→양쪽공백→
유니코드 정규화)는 loco `edit_file`의 3단계 사다리(정확→줄끝공백무시→
들여쓰기 시프트)와 설계 철학이 거의 같다 — 기존 설계를 재확인해주는
방향이며 새로 배울 구조는 아니다. 다만 **유니코드 문장부호 정규화 단계는
loco에 없다** — 모델이 코드를 "예쁘게" 옮겨적으며 ASCII 따옴표/대시를
스마트 따옴표/em-dash로 바꿔버리는 실패는 실제로 발생 가능한 케이스이므로,
`edit_file`의 사다리에 저비용으로 추가할 만한 후보다(신규 의존성 없이
`char` 매핑 테이블 하나로 구현 가능).

**Freeform apply_patch(문법 제약 커스텀 툴)는 이식 대상이 아니다**: LM
Studio급 백엔드의 커스텀/freeform 툴 지원은 OpenAI만큼 표준화돼 있지 않고,
loco는애초에 `protocol.rs`에서 네이티브 함수 호출(tool_calls) 대신
`response_format: json_schema`로 어시스턴트 **본문**에 구조를 강제하는
방식을 쓴다(스펙 §4) — 이는 codex가 전제하는 능력(freeform 툴 채널)
자체가 없다는 뜻이라 애초에 비교 대상이 아니다.

**구조화된 read_file/grep 부재는 loco가 따라 하면 안 되는 단순화다**:
codex는 프런티어 모델이 `rg`/`sed`/`cat`을 알아서 잘 조합한다는 전제로
범용 shell 하나에 검색/읽기를 위임할 수 있지만, loco의 대상 모델(8K 소형
로컬 모델)은 정확히 이 조합 능력이 약한 쪽이다 — loco의 `read_file`/
`grep`/`list_files`가 제공하는 구조화된 실패 피드백(0-match 힌트, 정규식
폴백, 디렉터리 경로 안내 등, CLAUDE.md 참고)은 소형 모델에게 필요한
스캐폴딩이지 codex처럼 걷어낼 대상이 아니다.

**`tool_search`(BM25 온디맨드 툴 로딩)는 m9-candidates §2("도구 목록
온디맨드 로딩")의 실제 참조 구현**이다 — MCP/동적 툴이 늘어날 때 전체
목록을 시스템 프롬프트에 상주시키지 않고 검색으로 로드하는 설계가 실제로
동작 중임을 확인했다. 다만 codex의 필요(MCP/플러그인 툴 폭증)와 loco의
현재 상태(6도구)는 규모가 다르므로 지금 당장 끌어올 것은 아니고, MCP·스킬
도입 시점의 설계 원칙으로 m9-candidates에 이미 기록된 방향을 강화하는
근거로 남긴다.

**`--oss`류 자동 프로비저닝(reachability 체크·자동 pull·버전 게이팅)은
loco에 옮길 필요가 없다** — loco의 폐쇄망 배포 전제는 "모델은 이미 LM
Studio에 로드돼 있다"는 사전조건이지, loco 자신이 인터넷에서 모델을 받아오는
것이 아니다. `ensure_oss_ready`가 요구하는 능력(패키지 저장소 접근)은
북극성(폐쇄망 동료 배포)과 정면으로 충돌한다.
