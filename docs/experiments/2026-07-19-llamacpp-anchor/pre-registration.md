## 0. 성격 — 전환 검증이지 효과 실험이 아니다

M13은 통과율을 올리는 마일스톤이 아니다(스펙 §1). 이 두 배치는 "llama.cpp로
스택을 옮겨도 기존 성능이 깨지지 않았는가"를 확인하는 **동등성 확인 + 회귀
게이트**이며, `docs/experiments/PROTOCOL.md` 1항("사전등록 없이는 GPU 배치를
돌리지 않는다")이 여기에도 그대로 적용된다.

## 가설 (반증 가능한 형태로)

**H1 (앵커)**: llama.cpp 스택(§3-3 핀 4개 적용)에서 `tasks/` 36런 통과 수는
M12 LM Studio 대조 배치(`20260718T222824Z`, 33/36)와 **동등**하다 — 즉
결함 하한(< 27/36)을 넘지 않고, 총합 차이가 ±4런 이내다.
**H1은 실패할 수 있다**: 결함 하한 미달, 또는 ±4 초과 모두 관측 가능한 결과이고
사전에 각각의 처분이 정해져 있다(§3-1·§3-3).

**H2 (회귀 게이트)**: T7의 상태선 수선(케이던스 조밀화 + 무뮤테이션 검증 줄)이
`tasks/` 36런 통과 수를 앵커 대비 유의미하게 떨어뜨리지 않는다 — 총합이
앵커 대비 −4런 이내다. **H2도 실패할 수 있다**: 재측정 1회로도 회복하지
못하면 T7 커밋을 되돌린다(§4-2).

## 1. 대상 커밋 동일성

- **앵커 배치(T6)의 대상 코드**: 커밋 `814d87c`(T4 완료 시점 — 전환 부수물
  일반화·`serve.sh`·`PROTOCOL.md` 항목 4 갱신·`schema_fallback` 기록·
  `parse_fail_first` 기계 검사까지 전부 포함, 요청 경로 코드는 T1~T4 중
  불변). T5(본 문서) 승인 후 T6 배치 착수 전까지 `src/`·`scripts/serve.sh`·
  `tasks/`에 변경이 있으면 안 된다 — 착수 전
  `git diff 814d87c..HEAD -- src/ scripts/serve.sh tasks/`가 **비어 있어야**
  한다(배치 조건을 이루는 요청 경로 코드·기동 스크립트·채점 픽스처 세 축
  전부를 잡는다 — `src/`만 보면 `scripts/serve.sh`나 `tasks/`가 바뀐 채로
  배치가 돌아도 이 동일성 검사가 통과해 버린다). 비어 있지 않으면
  배치를 시작하지 말고 보고한다. **`llama-server` 버전도 배치 조건이다** —
  `llama-server --version` 출력을 그대로 report.md에 기록한다(T5 확인:
  `version: 9960 (a935fbffe)`, `built with AppleClang 21.0.0.21000099 for
  Darwin arm64` — 이 기록을 신뢰하지 말고 배치 착수 시 스스로 재확인할 것).
- **회귀 게이트 배치(T8)의 대상 코드**: T7(상태선 수선 A·B) 완료 시점 커밋.
  이 문서 작성 시점에는 아직 존재하지 않으므로 해시를 지금 못 박을 수 없다 —
  T8 착수 시 `git rev-parse HEAD`로 report.md에 기록한다. 착수 시
  `llama-server --version`도 재확인해 report.md에 기록한다(T6과 다르면
  배치를 시작하지 말고 보고한다).
  **파일-범위 제약(기계 검사) — 내용까지는 보증하지 않는다**:
  `git diff 814d87c..<T7 커밋> -- src/`가 `src/agent/status_note.rs`·
  `src/agent/mod.rs` **외의 파일을 포함하면 안 된다**(스펙 §6 Files 목록과
  일치). 그 외 파일이 diff에 나타나면 배치를 시작하지 말고 보고한다. 또한
  `git diff 814d87c..<T7 커밋> -- scripts/serve.sh tasks/`가 **비어 있어야
  한다** — 이 두 경로는 T7에서 어떤 변경도 허용되지 않는다(위 T6 항목과
  동일 근거 — `src/` 밖에서 조건이 조용히 바뀌는 것을 막는다).
  **이 파일-목록 검사가 실제로 보증하는 것은 "T7 diff가 이 두 파일에만
  있다"는 파일 범위뿐이다 — 그 두 파일 내부의 어떤 내용 변경이든 그대로
  통과시킨다.** `src/agent/mod.rs`는 에이전트 루프 그 자체(디스패치·
  salvage·반복 감지·넛지)이므로, T7과 무관한 동작 변경이 그 파일 안에
  섞여도 이 검사는 잡지 못하고 T7 탓으로 잘못 귀속될 위험이 남는다.
  따라서 파일-목록 검사만으로는 "두 배치 사이에 T7 변경 이외의 코드 변경이
  없음"을 **기계적으로 보증하지 않는다** — 그 문구는 파일-범위 제약일
  뿐이다. 이를 메우기 위해, **T8 착수 전 사람이**
  `git diff 814d87c..<T7 커밋> -- src/agent/status_note.rs src/agent/mod.rs`
  **전체를 읽고**, 모든 변경이 수선 A(케이던스 조밀화)·수선 B(무뮤테이션
  검증 줄 렌더) 범위 안임을 확인한다. 이 확인을 수행했다는 사실(확인자·
  확인 시각 포함)을 report.md에 남긴다(예: "T7 diff 정독 완료 — 수선 A·B
  외 변경 없음 확인, 2026-MM-DD"). **이 정독 기록이 없으면 T8을 시작하지
  않는다.**

## 2. 배치 조건 — 두 배치 공통

### 2-0. 표본 (요약)

- **모델**: ornith-1.0-9b (`--alias ornith`로 기동)
- **컨텍스트**: 8K (`context_tokens = 8192`, `-c 8192`)
- **과제**: `tasks/` 12과제 전부, `--filter` 없음(제외 없음)
- **반복**: `--repeats 3 --seed 0`(§2-3 배치 명령에 둘 다 명시 — clap
  기본값과 값이 같아도 배치 명령에서 시드를 감추지 않는다, M12 전례)
- **총 런 수**: 배치당 36런(12과제 × 3반복) — 앵커 1회 + 회귀 게이트 1회 = 72런

### 2-1. `.loco/config.toml` (배치가 실제로 돌 조건 — 그대로 전재)

**주의: 현재 디스크의 `.loco/config.toml`은 M12 배치 2(tasks-large용,
`command_timeout_secs = 240`, `base_url` 미지정 → LM Studio 기본값)의
잔재이며 이번 배치 조건이 아니다.** 이 문서가 등록하는 조건은 다음과
**정확히 일치**해야 한다:

```toml
context_tokens = 8192
max_output_tokens = 4096
command_timeout_secs = 60
base_url = "http://localhost:8080/v1"
```

- `context_tokens = 8192`·`max_output_tokens = 4096`·`command_timeout_secs = 60`은
  대조 배치 `20260718T222824Z`의 `effective_config`를 그대로 따온 값이다
  (직접 확인: `python3 -c "import json; print(json.load(open('.loco/eval/20260718T222824Z/report.json'))['effective_config'])"`
  → `context_tokens=8192, max_output_tokens=4096, command_timeout_secs=60,
  temperature=0.1, max_turns=25`). `temperature`·`max_turns`는 config에 두지
  않는다 — 대조 배치도 override 없이 코드 기본값(0.1·25)을 썼으므로 그대로
  둔다.
- `base_url = "http://localhost:8080/v1"`이 **이 배치의 유일한 의도된 조건
  변경**이다 — `scripts/serve.sh`가 기본 포트 8080으로 llama-server를
  띄우기 때문이다. 대조 배치의 `base_url`(`http://localhost:1234/v1`,
  LM Studio)과 다른 것은 당연하고 등록 대상이다.
- **`max_output_tokens = 4096`은 사고 토큰 예산 잠식(스펙 §3-2)에 대한
  유일한 실효 레버이므로 여기서는 의도적으로 등록하는 조건이지, 대조 배치를
  베낀 우연이 아니다.** `--reasoning-format`은 이 문제를 해결하지 못하고
  (토큰을 어디 보고할지만 정함, json_schema와 병용 시 전 요청 400 — §3-3
  핀 3), llama-server는 Ornith의 사고 토큰을 `reasoning_content`로 분리하되
  `stream:false` 비스트리밍 응답에서 `ResponseMessage`가 `role`/`content`만
  역직렬화하므로 loco는 이를 보지 못하면서도 토큰 예산은 소비된다. 예산이
  사고로 소진되면 content가 잘린 접두가 아니라 **완전히 빈 문자열**로
  온다(관측된 사고 길이 39~551 토큰, `max_output_tokens=4096`이므로 위험은
  실재하되 완만함 — 스펙 §3-2). 이후 이 값을 바꾸는 배치는 이 문서의 대조
  대상이 될 수 없다.
- 배치 시작 전, 배치 러너는 실제 디스크의 `.loco/config.toml`을 위 내용으로
  **덮어써야 한다**(본 태스크는 그 파일을 건드리지 않는다 — T6/T8의 몫).

### 2-2. `scripts/serve.sh` 기동 (핀 4개 고정, T2 산출물)

**버전 확인(배치 조건 등록)** — T6·T8 착수 시 각각 실행하고 출력을 그대로
report.md에 기록한다:

```bash
llama-server --version
```

T5에서 확인한 값(재현 시 스스로 재확인할 것 — 이 문서의 기록을 신뢰하지
말 것): `version: 9960 (a935fbffe)`, `built with AppleClang 21.0.0.21000099
for Darwin arm64`.

**기동(데몬화 + 로그 캡처)** — §2-4 항목 8의 데몬화 래퍼를 이 배치에 실제로
적용한 명령이다. `<라벨>`은 `T6` 또는 `T8`:

```bash
python3 -c "import os,sys; os.setsid(); os.execvp(sys.argv[1], sys.argv[1:])" \
  sh -c 'LOCO_MODEL_GGUF=/Users/sgj/.lmstudio/models/deepreinforce-ai/Ornith-1.0-9B-GGUF/ornith-1.0-9b-Q4_K_M.gguf \
  LOCO_CTX=8192 scripts/serve.sh > .loco/serve-<라벨>.log 2>&1'
```

(`LOCO_PORT` 기본값 8080, `LOCO_ALIAS` 기본값 `ornith` — 위 config의
`base_url` 포트와 일치해야 한다). 이전 서버가 떠 있으면 먼저
`pkill -f llama-server`로 내린다. 핀 내용(참고, `scripts/serve.sh` 주석이
원본): `-np 1`(컨텍스트 슬롯 분할 방지)·`--top-k 40 --top-p 0.95 --min-p 0.05
--repeat-penalty 1.0`(샘플러 4종 명시)·reasoning 처리 기본값(auto, 분리)
고정·`--alias ornith`.

**이 로그가 기계 검사 3(§3-2)의 증거다.** `.loco/serve-<라벨>.log`에는
llama-server 기동 시 `n_ctx_slot`을 찍는 시작 로그가 남는다. `$STAMP`(정의는
§2-3)는 `cargo run -- eval`을 실행하기 전에는 아직 존재하지 않으므로, 배치가 끝나고
`$STAMP`를 알게 되는 즉시 이 로그를 그 배치 자신의 산출물로 편입한다:

```bash
cp .loco/serve-<라벨>.log $STAMP/server-startup.log
```

**이 로그를 남기지 못했거나(리다이렉션 누락·서버가 로그 없이 죽음 등)
`$STAMP/server-startup.log`에서 `n_ctx_slot`을 읽을 수 없으면, 기계 검사 3은
실패로 처리한다 — "확인 못 함"은 통과가 아니다.** 이 규칙을 우회하는 길도
막는다: 로그를 놓친 뒤 서버를 다시 기동해 갓 찍힌 로그를 복사해 넣으면
겉보기엔 조건을 채우지만 이 배치 자신의 서버 프로세스가 남긴 증거가
아니다 — **로그의 시작 시각이 이 배치의 `$STAMP`보다 앞서야 한다**(구체
명령·주의사항):

```bash
BIRTH=$(stat -f %B .loco/serve-<라벨>.log)
STAMP_EPOCH=$(date -u -j -f "%Y%m%dT%H%M%SZ" "$(basename "$STAMP")" +%s)
[ "$BIRTH" -lt "$STAMP_EPOCH" ] && echo "OK: 로그 시작이 STAMP보다 앞섬" \
  || echo "FAIL: 로그 시작이 STAMP와 같거나 뒤"
```

**`%m`(mtime, 마지막 쓰기 시각)이 아니라 `%B`(birth time, 생성 시각)를
써야 한다** — `scripts/serve.sh` 기동 명령(§2-2)은 로그를 리다이렉션만 할
뿐 버퍼링을 억제하지 않으므로, llama-server는 배치 내내(~45분) 요청마다
로그를 이어 쓴다. mtime은 그 마지막 쓰기 시각이라 배치 종료 시점에 가깝고,
**정직한 배치에서도 항상 `$STAMP`보다 뒤로 나온다** — mtime으로 비교하면
이 규칙이 정직한 배치를 매번 탈락시킨다. birth time은 로그 파일이 최초
생성된 순간(서버 기동 시각)에 고정되며 이후 append로 바뀌지 않는다(T5
확인: 스크래치 파일에 birth time을 찍은 뒤 2초 뒤 내용을 append해도 `%B`
값은 그대로였고 `%m`만 갱신됨을 직접 실행해 확인함 — 재현 시 스스로
재확인할 것, 이 확인은 이 태스크가 실행을 허가받은 `stat`을 스크래치
파일에만 적용한 것이지 서버를 기동한 것이 아니다).

**`$STAMP` 디렉터리명(예: `20260719T101500Z`)은 UTC 표기다** — `date -u`로
UTC epoch로 변환한 뒤 비교해야 하며, 로컬 타임존으로 해석하면 최대 수
시간 오차가 나서 이 비교 자체가 무의미해진다(위 명령의 `date -u -j -f`가
이 변환을 한다).

이 선후 관계가 성립하지 않으면(로그가 `$STAMP`보다 뒤에 시작됐다면), 또는
`%B`를 읽을 수 없으면(파일시스템이 birth time을 지원하지 않는 등), 사후에
재기동해 만든 로그로 간주하거나 확인 불가로 간주해 기계 검사 3은 마찬가지로
실패로 처리한다 — "확인 못 함"은 여기서도 통과가 아니다.

### 2-3. 실행 명령 — 빌드 프로파일 포함

```bash
cargo run -- eval tasks --repeats 3 --seed 0
```

(`--seed 0`은 `src/main.rs`의 `default_value_t = 0`과 값이 같아 생략해도
동작은 같지만, M12 전례를 따라 배치 명령에서 시드를 감추지 않고 명시적으로
적는다. 12과제 × 3반복 = 36런.)

**`$STAMP` 표기**: 이 문서 전체(§2-2·§3·§4·§7)에서 `$STAMP`는 이 명령이 만드는
`.loco/eval/<타임스탬프>/` 디렉토리 경로를 가리키는 셸 변수다 — 배치 실행
후 실제 값으로 채워 넣는다(예: `export STAMP=.loco/eval/20260719T101500Z`).
앵커(T6)와 회귀 게이트(T8)는 서로 다른 `$STAMP`를 갖는다.

**`--release`를 쓰지 않는다 — 반드시 디버그 빌드로 돈다.** 대조 배치
`20260718T222824Z`는 디버그 빌드였고(`cargo run`, `--release` 플래그 없음),
빌드 프로파일은 `report.json`의 `effective_config`(`loco_version`만 기록,
프로파일 필드 없음)에 남지 않는다. `--release`로 돌리면 스택 전환과
빌드 프로파일이라는 **두 변수가 동시에 바뀌는데, 그 사실이 나중에 발견될
수단이 report.json에 없다** — 사전등록 시점에 금지해 둔다.

### 2-4. 배치 전 스모크 (`PROTOCOL.md` 항목 4, T2가 갱신 — 전건 통과 필수)

1. **`--verify` 게이트**(PROTOCOL.md 항목 4 ①): `cargo run -- eval tasks --verify`

   별개의 게이트이며, 매 배치 직전에 판정 가능성 쪽을 재확인하는 것이 이

2. json_schema 요청 1건 → **HTTP 200** (실패 시 배치 시작 금지 — C1 재발 방지)
3. 서버 기동 로그(§2-2에서 등록한 `.loco/serve-<라벨>.log`)의

4. `curl -s localhost:8080/v1/models`의 `data[0].id == "ornith"`


**적용 규칙(귀속 기본값 반전 — 사용자 결정, T5 개정)**: 이 6개 중 하나라도

조건과 동일하다).

세 경우 모두 정독 결과를 report.md에 남긴다.

⚠️ **명시적으로 금지하는 오독**: "판정 = (a) 6/6 만족 AND (b) ±4 이내"라는

(정정 — 이전 판은 여기서 "보류에 들어간다는 것은 (b) 자체는 처음부터

  않는다.** 정독 결과와 보류 사유를 report.md에 남기고 사용자에게 보고해
  추가 판단(재정독·추가 자료 확보 여부 등)을 구하며, **이 보류된 배치
  자체는 재수행하지 않는다**(§3-3·§5 참조). 아직 확립된 실패(수치 미달도
  귀속 확인도)가 없으므로 강제 되돌림 terminal은 두지 않는다.
- **(b)가 이미 미달(< 앵커 − 4)이었던 채로 보류에 들어간 경우**, 그리고
  **정독이 T7 귀속을 적극적으로 확인한 경우(갈래 1, (b)의 최초 수치와
  무관하게 미달로 취급됨)**는 §4-2가 새로 정의하는 terminal로 이어진다 —
  아래 참조.

⚠️ 여기서 (a)를 연언 항으로 두지 않는 것이 앵커보다 **더 중요**하다 —
이 게이트의 미달 결과는 ~55분 GPU 재측정 + T7 작업 되돌림이라는 실비용을
자동 발동시키므로, 우연한 흔들림 하나가 그 비용을 촉발해서는 안 된다.

### 4-2. 미달 시 처분 (사전 공약)

**이 절의 범위는 미달의 발생 원인이 아니라 해제 수단으로 가른다.** 아래
1항의 best-of-two(수치 하나만으로 통과 확정)는 §4-1 갈래 3(보류)의 해제
수단이 **아니다** — 보류는 오직 §4-1이 정한 적극적 귀속 결론(관련됨 또는
무관함)으로만 해제되며, 이는 보류 진입 시점에 (b)가 이미 수치 기준을
충족했었는지와 무관하게 항상 적용된다. **다만** 게이트 총합이 처음부터
`< 앵커 − 4`(순수 수치 미달)이거나 갈래 1(T7 귀속을 적극적으로 확인한
경우)인 경우에는, 아래 2항의 T7 되돌림이 적용된다 — 이 두 경우는 각각
다음과 같이 이 절로 흘러든다:

- **갈래 1(T7 귀속 확인)**: (b)의 최초 수치와 무관하게 미달로 확정된다
  (§4-1). 이 미달은 아래 1항의 통과 채널(수치만으로 통과)을 거치지 않고
  곧바로 2항의 T7 되돌림이 적용된다 — 귀속이 이미 확인된 이상, 재측정의
  수치 결과가 무엇이든 그 확인을 뒤집지 않는다.
- **갈래 3(보류)이 게이트 총합 `< 앵커 − 4`(순수 수치 미달)와 겹친 경우**:
  이후 정독·추가 자료 확보로 **무관함이 적극적으로 확인되면**, 그 시점부터
  이미 확정돼 있던 최초 미달 값에 아래 1항(최초 기록·재측정 1회·
  best-of-two)을 정상 적용한다. **적극적으로 관련으로 확인되거나, 끝내
  어느 쪽도 확인하지 못하면**(귀속 불명이 그대로 유지되면), 재측정의 수치
  결과와 무관하게 2항의 T7 되돌림을 적용한다 — 귀속 불명 그 자체를
  되돌림을 미루는 근거로 쓰지 않는다(§3-3의 원칙을 게이트에도 그대로
  적용한 것이며, 이 문단이 이 원칙을 새로 만드는 것이 아니라 §4-2가 이
  원칙에 대해 정의된 terminal을 갖도록 되돌림 처분을 앞당겨 적용할 뿐이다).

(갈래 3이 게이트 총합 `≥ 앵커 − 4`, 즉 순수 수치 통과와 겹친 경우는 위
어느 쪽에도 해당하지 않는다 — §4-1이 정한 대로 강제 되돌림 terminal 없이
적극적 귀속 결론으로만 해제된다. 아직 확립된 실패가 없는 상태에 이 절이
새로 미달을 만들어내지 않는다.)

1. 미달(게이트 총합 < 앵커 − 4, 또는 위에서 이 절로 흘러든 경우)이면,
   **그 최초(미달) 값을 report.md에 먼저 기록한 뒤** 재측정을 1회
   실행한다. **최초 값과 재측정 값 중 하나라도 기준(≥ 앵커 − 4)을
   충족하면 게이트는 통과로 기록한다** — best-of-two 규칙이며, 미달했던
   최초 값도 폐기하지 않고 함께 남는다. (갈래 1로 흘러든 경우는 위에서
   명시한 대로 이 항의 통과 채널을 거치지 않는다.)
2. 재측정도 미달이면(또는 갈래 1로 흘러든 경우, 또는 보류가 끝내 해제되지
   않은 경우), 귀속 여부와 무관하게 **T7 커밋을 되돌리고**
   (`git revert --no-edit <T7 커밋>`), 파일럿(T9 이후)은 T4까지의 코드로
   진행하며, 사용자에게 보고한다. **추가 재측정은 없다.**

### 4-3. T7 이후 지표 비교가능성 — `verify_*` 4종

회귀 게이트 배치의 `verify_total`/`verify_zero`/`verify_allpass`/
`verify_failed`(`scripts/exp_metrics.py`)는 T7의 수선 B(무뮤테이션 분기에도
검증 줄을 렌더)로 인해 **구조적으로 상향된다** — T7 이전에는 이 문자열이
뮤테이션 분기에서만 나올 수 있었으나, T7 이후에는 무뮤테이션 케이던스
노트마다 실린다. **앵커 배치(T7 이전)의 같은 지표와 직접 비교하지 않는다.**
이 각주는 T7 산출물인 `report.md`에도 기록되고(스펙 §5-2 Task 7 Step 7),
T12에서 `docs/baselines.md`의 M13 절로 승계된다.

## 5. "배치 사망"의 정의 (재수행 트리거 — 유일한 정의)

**배치 사망 = LLM 에러(하네스가 abort) · 하네스 에러(exit 1, 서버 다운·
설정 오류) · 프로세스 사망 · Ctrl+C로 인한 부분 리포트(`interrupted: true`)에
한정한다.**

**정상 종료(`interrupted: false`)했으나 통과 수가 낮은 배치는 사망이 아니다
— 재수행 대상이 아니다.** 그 경우는 §3(앵커) 또는 §4(회귀 게이트)의 판정
규칙만 적용된다. 낮은 통과 수 자체를 "무언가 잘못됐으니 다시 돌리자"는
근거로 쓰지 않는다.

(이 정의가 필요한 이유: M12 사전등록 승인 리뷰가 "배치 사망"의 미정의를
악용 가능한 seam으로 지적한 전례가 있다 — 정상 종료한 낮은 통과 수 배치를
"사망"으로 재해석해 재수행을 정당화할 수 있었다.)

### 5-1. 중단 규칙 (배치 사망 시)

배치 사망 시 **1회 재수행**, 재실패면 정지·보고(M11·M12 전례 승계). Ctrl+C로
중단된 부분 리포트는 폐기하고 해당 배치를 처음부터 재수행한다. 이 규칙은
§4-2의 "미달 시 재측정"과는 **별개 트리거**다 — 미달 재측정은 정상 종료한
배치의 통과 수가 기준에 못 미칠 때, 배치 사망 재수행은 하네스/프로세스/
Ctrl+C 실패일 때 발동한다. 두 트리거가 같은 배치에서 겹칠 이유는 없다(사망한
배치는 통과 수 판정 자체가 성립하지 않는다).

**산정 근거(실측 평균 × 여유분)**: §3-3 안정 집합의 근거가 된 동일한 ornith

앵커 ~55분 + 회귀 게이트 ~55분 ≈ **총 ~110분(≈1.8시간)**(서버 기동·스모크
여유 포함). 32K 없음 — 이 두 배치는 8K 스포트(36런)만 다룬다. 재측정이
발동하면(§4-2) 추가로 최대 ~55분(총 상한 ≈165분/2.75시간).
