## 0. 성격 — 전환 검증이지 효과 실험이 아니다

M13은 통과율을 올리는 마일스톤이 아니다(스펙 §1). 이 두 배치는 "llama.cpp로
스택을 옮겨도 기존 성능이 깨지지 않았는가"를 확인하는 **동등성 확인 + 회귀
게이트**이며, `docs/experiments/PROTOCOL.md` 1항("사전등록 없이는 GPU 배치를
돌리지 않는다")이 여기에도 그대로 적용된다.

## 가설 (반증 가능한 형태로)

**H1 (앵커)**: llama.cpp 스택(§3-3 핀 4개 적용)에서 `tasks/` 36런 통과 수는
M12 LM Studio 대조 배치(`20260718T222824Z`, 33/36)와 **동등**하다 — 즉
결함 하한(< 27/36)을 넘지 않고, 총합 차이가 ±4런 이내다.
**H1은 실패할 수 있다**: 결함 하한 미달, 또는 ±4 초과 모두 관측 가능한 결과이고
사전에 각각의 처분이 정해져 있다(§2·§4).

**H2 (회귀 게이트)**: T7의 상태선 수선(케이던스 조밀화 + 무뮤테이션 검증 줄)이
`tasks/` 36런 통과 수를 앵커 대비 유의미하게 떨어뜨리지 않는다 — 총합이
앵커 대비 −4런 이내다. **H2도 실패할 수 있다**: 재측정 1회로도 회복하지
못하면 T7 커밋을 되돌린다(§5).

  T8 착수 시 `git rev-parse HEAD`로 report.md에 기록한다. **동일성 보증**:
  `git diff 814d87c..<T7 커밋> -- src/`가 `src/agent/status_note.rs`·

## 2. 배치 조건 — 두 배치 공통

### 2-0. 표본 (요약)

- **모델**: ornith-1.0-9b (`--alias ornith`로 기동)
- **컨텍스트**: 8K (`context_tokens = 8192`, `-c 8192`)
- **과제**: `tasks/` 12과제 전부, `--filter` 없음(제외 없음)
- **반복**: `--repeats 3 --seed 0`(생략 가능 — 기본값 0)
- **총 런 수**: 배치당 36런(12과제 × 3반복) — 앵커 1회 + 회귀 게이트 1회 = 72런

### 2-1. `.loco/config.toml` (배치가 실제로 돌 조건 — 그대로 전재)

**주의: 현재 디스크의 `.loco/config.toml`은 M12 배치 2(tasks-large용,
`command_timeout_secs = 240`, `base_url` 미지정 → LM Studio 기본값)의
잔재이며 이번 배치 조건이 아니다.** 이 문서가 등록하는 조건은 다음과
**정확히 일치**해야 한다:

```toml
context_tokens = 8192
max_output_tokens = 4096
command_timeout_secs = 60
base_url = "http://localhost:8080/v1"
```

- `context_tokens = 8192`·`max_output_tokens = 4096`·`command_timeout_secs = 60`은
  대조 배치 `20260718T222824Z`의 `effective_config`를 그대로 따온 값이다
  (직접 확인: `python3 -c "import json; print(json.load(open('.loco/eval/20260718T222824Z/report.json'))['effective_config'])"`
  → `context_tokens=8192, max_output_tokens=4096, command_timeout_secs=60,
  temperature=0.1, max_turns=25`). `temperature`·`max_turns`는 config에 두지
  않는다 — 대조 배치도 override 없이 코드 기본값(0.1·25)을 썼으므로 그대로
  둔다.
- `base_url = "http://localhost:8080/v1"`이 **이 배치의 유일한 의도된 조건
  변경**이다 — `scripts/serve.sh`가 기본 포트 8080으로 llama-server를
  띄우기 때문이다. 대조 배치의 `base_url`(`http://localhost:1234/v1`,
  LM Studio)과 다른 것은 당연하고 등록 대상이다.
- **`max_output_tokens = 4096`은 사고 토큰 예산 잠식(스펙 §3-2)에 대한
  유일한 실효 레버이므로 여기서는 의도적으로 등록하는 조건이지, 대조 배치를
  베낀 우연이 아니다.** `--reasoning-format`은 이 문제를 해결하지 못하고
  (토큰을 어디 보고할지만 정함, json_schema와 병용 시 전 요청 400 — §3-3
  핀 3), llama-server는 Ornith의 사고 토큰을 `reasoning_content`로 분리하되
  `stream:false` 비스트리밍 응답에서 `ResponseMessage`가 `role`/`content`만
  역직렬화하므로 loco는 이를 보지 못하면서도 토큰 예산은 소비된다. 예산이
  사고로 소진되면 content가 잘린 접두가 아니라 **완전히 빈 문자열**로
  온다(관측된 사고 길이 39~551 토큰, `max_output_tokens=4096`이므로 위험은
  실재하되 완만함 — 스펙 §3-2). 이후 이 값을 바꾸는 배치는 이 문서의 대조
  대상이 될 수 없다.
- 배치 시작 전, 배치 러너는 실제 디스크의 `.loco/config.toml`을 위 내용으로
  **덮어써야 한다**(본 태스크는 그 파일을 건드리지 않는다 — T6/T8의 몫).

### 2-2. `scripts/serve.sh` 기동 (핀 4개 고정, T2 산출물)

```bash
LOCO_MODEL_GGUF=/Users/sgj/.lmstudio/models/deepreinforce-ai/Ornith-1.0-9B-GGUF/ornith-1.0-9b-Q4_K_M.gguf \
  LOCO_CTX=8192 scripts/serve.sh
```

(`LOCO_PORT` 기본값 8080, `LOCO_ALIAS` 기본값 `ornith` — 위 config의
`base_url` 포트와 일치해야 한다). 이전 서버가 떠 있으면 먼저
`pkill -f llama-server`로 내린다. 핀 내용(참고, `scripts/serve.sh` 주석이
원본): `-np 1`(컨텍스트 슬롯 분할 방지)·`--top-k 40 --top-p 0.95 --min-p 0.05
--repeat-penalty 1.0`(샘플러 4종 명시)·reasoning 처리 기본값(auto, 분리)
고정·`--alias ornith`.

### 2-3. 실행 명령 — 빌드 프로파일 포함

```bash
cargo run -- eval tasks --repeats 3
```

(`--seed 0`은 `src/main.rs`의 `default_value_t = 0`이라 생략 가능. 12과제 ×
3반복 = 36런.)

**`--release`를 쓰지 않는다 — 반드시 디버그 빌드로 돈다.** 대조 배치
`20260718T222824Z`는 디버그 빌드였고(`cargo run`, `--release` 플래그 없음),
빌드 프로파일은 `report.json`의 `effective_config`(`loco_version`만 기록,
프로파일 필드 없음)에 남지 않는다. `--release`로 돌리면 스택 전환과
빌드 프로파일이라는 **두 변수가 동시에 바뀌는데, 그 사실이 나중에 발견될
수단이 report.json에 없다** — 사전등록 시점에 금지해 둔다.

### 2-4. 배치 전 스모크 (`PROTOCOL.md` 항목 4, T2가 갱신 — 전건 통과 필수)

1. json_schema 요청 1건 → **HTTP 200** (실패 시 배치 시작 금지 — C1 재발 방지)
2. 서버 기동 로그의 `n_ctx_slot == 8192`
3. `curl -s localhost:8080/v1/models`의 `data[0].id == "ornith"`
4. `.loco/config.toml`이 §2-1과 바이트 동일
5. `ls ${TMPDIR}/.cargo` — 존재하면 수동 제거(트립와이어)
6. 측정 중 `cargo build`/`test` 병행 금지
7. 데몬화: `python3 -c "import os,sys; os.setsid(); os.execvp(sys.argv[1], sys.argv[1:])" <cmd>...`
   (macOS에 `setsid` 없음 — 하네스 백그라운드 60분 수명 상한 대응)

## 3. 배치 1 — 앵커 (T6)

- **대조**: 스탬프 `20260718T222824Z` — **33/36, 엄격 32**
  (M12 회귀 게이트 배치, `docs/baselines.md` "M12 실험 — 회귀 게이트" 참조)

### 3-1. 결함 하한 (판정 규칙보다 먼저 적용 — 이것 자체는 판정이 아니라 게이트)

**앵커 배치가 대조보다 6런을 초과해 낮으면(< 27/36), 그 숫자를 앵커로
기록하지 않는다.** 설정 결함(예: C1형 조용한 폴백 영구 발동)으로 간주하고
**마일스톤을 정지해 진단한다** — 이 단계에서는 어떤 동등성 판정도 시도하지
않는다.

### 3-2. 기계 검사 3종 (결함 하한과 별개로, 앵커로 인정되려면 전건 통과 필수)

이 3종은 **동등한 셋이 아니다** — 우선순위가 있다:

1. **주 검사**: `report.json`의 `tasks[].runs[]` 전건에서 `schema_fallback == false`
   (T3 산출물). 런의 최종 상태를 읽으므로 **몇 번째 턴에서 폴백이 발동했든
   잡는다.**
   ```bash
   python3 -c "
   import json
   d = json.load(open('$STAMP/report.json'))
   fb = [f\"{t['name']}-{r['repeat']}\" for t in d['tasks'] for r in t['runs'] if r.get('schema_fallback')]
   print('schema_fallback 발동 런:', fb or '없음')"
   ```
   Expected: `없음`.
2. **보조**: `parse_fail_first(총) == 0`(T4 산출물, `scripts/exp_metrics.py`).
   첫 assistant 메시지만 보고, 거짓 양성을 내지 않는 쪽으로 보수적이라
   (산문·JSON 없음은 전부 0) **단독으로는 불충분** — 1번 주 검사가 놓친
   것을 잡을 목적이 아니라 1번이 통과했다는 것을 독립 경로로 재확인하는
   용도다.
   ```bash
   python3 scripts/exp_metrics.py $STAMP 2>&1 | tail -3   # "parse_fail_first(총): 0" 확인
   ```
3. **환경**: 서버 기동 로그의 `n_ctx_slot == 8192` (§2-4의 스모크 2항과
   동일 — 배치 후에도 재확인한다).

**셋 중 하나라도 실패하면 그 숫자를 앵커로 기록하지 않고, 정지·보고한다.**

### 3-3. 판정 규칙 (결함 하한과 기계 검사를 통과한 뒤에만 적용)

**판정은 (b) 동등성 규칙 단독으로 내린다.**

- **(b) 동등성 판정 규칙**: 앵커 통과 수와 대조(33/36) 차이가 **±4런 이내**면
  동등 성립. (대조가 33이므로 산술 상한 37은 도달 불가 — 총 36런 배치에서
  최댓값은 36이다. 즉 실효 판정 구간은 **29~36**이며, 이 구간을 벗어나는
  경우는 아래로만 발생할 수 있다.)
- **동등 불성립 시**(29 미만이면서 결함 하한 27 이상, 즉 27~28인 경우 —
  또는 이론상 그 이상 벌어진 경우): llama.cpp 앵커를 새로 잡고 이후 비교에
  각주를 단다. 원인 분석(§3-2의 사고 토큰 가설을 우선 검토)은 수행하되
  **수선은 M13 범위 밖**이다.

**(a) 안정 집합 — 판정 항이 아니라 분류(triage) 방아쇠다. 판정식에 대입하지
않는다.** 아래 6개 과제는 ornith 36런 배치 6개 전부(`20260716T062853Z`·
`20260717T041725Z`·`20260717T215729Z`·`20260718T092740Z`·`20260718T115152Z`·
`20260718T222824Z`)에서 예외 없이 3/3이었던 과제다:

```
add-function, chain-edits, count-usages, create-module,
find-definition, fix-off-by-one
```

**적용 규칙**: 이 6개 중 하나라도 앵커 배치에서 3/3 아래로 떨어지면,
**어떤 판정도 기록하기 전에** 그 런의 트랜스크립트를 정독한다. 정독 결과
실패가 스택 차이(llama.cpp 고유의 무엇)로 귀속될 때에만 (b)의 판정을
**불성립으로 뒤집는다**. 귀속되지 않으면(예: 우연한 산문 생성, 기존에도
알려진 변동) (b)의 판정을 그대로 기록하되, 정독 결과를 report.md에 남긴다.

⚠️ **명시적으로 금지하는 오독**: "판정 = (a) 6/6 만족 AND (b) ±4 이내"라는
**연언(conjunctive) 형태로 읽거나 적용하지 않는다.** 그렇게 읽으면 18개
베르누이 시행(6과제 × 3반복)이 전부 성공할 것을 요구하게 되어, per-run
실패 확률이 조금만 있어도(예: p=0.01) 오경보율이 약 17%까지 치솟고 그
추정 자체가 근거 데이터 부족으로 불가능하다(스펙 §3-6-2 "규칙의 근거"
참조 — 안정 집합은 표본을 넓힐 때마다 멤버가 떨어져 나간 이력이 있어
"6개가 마지막"이라는 보장도 없다). (a)가 주는 것은 **진단력**뿐이며,
판정 부담은 전부 (b)로 옮겨져 있다.

### 3-4. 관측 항목 (판정에 영향 없음, 기록 의무)

- **빈-content 턴 수** (`grep -c '"(empty)"' $STAMP/run-*.jsonl`) — 스펙
  §3-2의 사고 토큰 예산 잠식에 대한 **대리 지표**다. `finish_reason`은
  트랜스크립트에 영속되지 않으므로 직접 셀 수 없고, 일부 content가 남은
  절단은 이 지표로 안 잡힌다(다만 §3-2 실측상 예산 소진 시 content가
  부분이 아니라 완전히 비어 지배적 경우는 포착된다).
- `empty_test_note` 발동 수, `status_note` 렌더 수(둘 다
  `scripts/exp_metrics.py`의 `MARKS` 열).

## 4. 배치 2 — 회귀 게이트 (T8)

- **대상 코드**: T7 완료 시점 커밋(§1에서 동일성 보증 방법 등록)
- **조건**: 배치 1과 **바이트 동일** — `.loco/config.toml`(§2-1), `serve.sh`
  인자(§2-2), 실행 명령(§2-3, 디버그 빌드), 배치 전 스모크(§2-4) 전부
  그대로. 유일한 의도된 차이는 T7의 코드 변경이다.
- **기계 검사 3종**: §3-2와 동일 절차·동일 우선순위로 재적용. 실패 시
  정지·보고(앵커와 동일 처분).

### 4-1. 판정 규칙

**게이트: 회귀 게이트 총합 ≥ 앵커 총합 − 4런.** (단방향 — 게이트가 앵커보다
높게 나오는 것은 우려 대상이 아니다.)

**(a) 안정 집합은 여기서도 판정 항이 아니라 분류 방아쇠다** — §3-3과 동일한
6개 목록, 동일한 적용 규칙(3/3 미만이면 판정 전에 정독, T7 변경으로
귀속될 때에만 미달로 취급). ⚠️ 여기서 (a)를 연언 항으로 두지 않는 것이
앵커보다 **더 중요**하다 — 이 게이트의 미달 결과는 ~40분 GPU 재측정 +
T7 작업 되돌림이라는 실비용을 자동 발동시키므로, 우연한 흔들림 하나가
그 비용을 촉발해서는 안 된다.

### 4-2. 미달 시 처분 (사전 공약)

1. 미달(게이트 총합 < 앵커 − 4)이면 **재측정을 1회** 실행한다.
2. 재측정도 미달이면, 귀속 여부와 무관하게 **T7 커밋을 되돌리고**
   (`git revert --no-edit <T7 커밋>`), 파일럿(T9 이후)은 T4까지의 코드로
   진행하며, 사용자에게 보고한다. **추가 재측정은 없다.**

### 4-3. T7 이후 지표 비교가능성 — `verify_*` 4종

회귀 게이트 배치의 `verify_total`/`verify_zero`/`verify_allpass`/
`verify_failed`(`scripts/exp_metrics.py`)는 T7의 수선 B(무뮤테이션 분기에도
검증 줄을 렌더)로 인해 **구조적으로 상향된다** — T7 이전에는 이 문자열이
뮤테이션 분기에서만 나올 수 있었으나, T7 이후에는 무뮤테이션 케이던스
노트마다 실린다. **앵커 배치(T7 이전)의 같은 지표와 직접 비교하지 않는다.**
이 각주는 T7 산출물인 `report.md`에도 기록되고(스펙 §5-2 Task 7 Step 7),
T12에서 `docs/baselines.md`의 M13 절로 승계된다.

## 5. "배치 사망"의 정의 (재수행 트리거 — 유일한 정의)

**배치 사망 = LLM 에러(하네스가 abort) · 하네스 에러(exit 1, 서버 다운·
설정 오류) · 프로세스 사망 · Ctrl+C로 인한 부분 리포트(`interrupted: true`)에
한정한다.**

**정상 종료(`interrupted: false`)했으나 통과 수가 낮은 배치는 사망이 아니다
— 재수행 대상이 아니다.** 그 경우는 §3(앵커) 또는 §4(회귀 게이트)의 판정
규칙만 적용된다. 낮은 통과 수 자체를 "무언가 잘못됐으니 다시 돌리자"는
근거로 쓰지 않는다.

(이 정의가 필요한 이유: M12 사전등록 승인 리뷰가 "배치 사망"의 미정의를
악용 가능한 seam으로 지적한 전례가 있다 — 정상 종료한 낮은 통과 수 배치를
"사망"으로 재해석해 재수행을 정당화할 수 있었다.)

### 5-1. 중단 규칙 (배치 사망 시)

배치 사망 시 **1회 재수행**, 재실패면 정지·보고(M11·M12 전례 승계). Ctrl+C로
중단된 부분 리포트는 폐기하고 해당 배치를 처음부터 재수행한다. 이 규칙은
§4-2의 "미달 시 재측정"과는 **별개 트리거**다 — 미달 재측정은 정상 종료한
배치의 통과 수가 기준에 못 미칠 때, 배치 사망 재수행은 하네스/프로세스/
Ctrl+C 실패일 때 발동한다. 두 트리거가 같은 배치에서 겹칠 이유는 없다(사망한
배치는 통과 수 판정 자체가 성립하지 않는다).

## 6. GPU 시간 예산 (상한치)

앵커 ~40분 + 회귀 게이트 ~40분 ≈ **총 ~1.3~1.5시간**(서버 기동·스모크·
재측정 여유 포함). 32K 없음 — 이 두 배치는 8K 스포트(36런)만 다룬다.
재측정이 발동하면(§4-2) 추가로 최대 ~40분.

## 7. 지표 추출 (기록 방법)

`parse_fail_first`·관측 항목을 뽑는다. 판정 자체(통과 수 차이, 결함 하한,

report.json 직접 대조" 메모리 참조).
