## 컨텍스트 관리 전략 (2026-07-16, 사용자 아이디어 메모)

### 1. 서브에이전트 적극 이용 (컨텍스트 임대)
탐색·항해처럼 컨텍스트를 많이 태우는 작업을 일회용 서브런(신선한 컨텍스트)에
위임하고, 증류된 결과(파일 경로 목록, API 요약 등)만 본 루프로 회수한다.

- 레퍼런스 대비: aider = "컨텍스트를 미리 증류"(repo-map) vs Claude Code/codex =
  "컨텍스트를 딴 데서 빌려옴"(서브에이전트). loco(8K 소형 모델)에 어느 철학이
  맞는지가 M9의 핵심 설계 질문 후보.
- 리스크: ① 소형 모델은 위임 능력(과제 정식화·결과 통합)이 약해 오케스트레이터
  역할 자체가 병목일 수 있음 ② 로컬 GPU 1장에서는 서브런이 병렬이 아니라
  직렬 + 회당 프리필 추가 비용 — 클라우드와 비용 모델이 다름.
- 채택 근거가 될 M8 신호: "항해 중 컨텍스트 소진"(오버플로 중단·pack 축약 후
  맥락 상실)이 지배적 실패 유형으로 나오는가.

### 2. 도구 목록 온디맨드 로딩 (deferred tools)
skill/subagent/MCP 등 도구 목록 전체를 시스템 프롬프트에 상주시키지 않고,
한 줄 인덱스만 두고 필요 판단 시 검색(grep류 조회)으로 스키마를 로드한다.

- 현재 loco는 6도구라 절약분 미미 — **도구 표면이 커지는 시점(MCP/스킬 도입)의
  설계 원칙**으로 기록. 그 도입 자체는 아직 로드맵에 없음.
- 부수 효과: 소형 모델의 도구 선택 마비 완화(선택지 축소 장치를 겸함).
