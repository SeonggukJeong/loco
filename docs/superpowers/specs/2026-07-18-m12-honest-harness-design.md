## 1. 목표·성공 기준·비범위

**목표.** 하네스가 모델에게 제시하는 신호의 정직성 결함을 수선하고, 이미
유효성이 입증된 개입 메커니즘의 도달률 사각을 해소한다:

- 거짓 초록불 2건: 0-테스트 cargo 실행(exit 0)을 FINISH_NUDGE가 "성공
  검증"으로 무장하고 상태선이 "exited 0"으로 접지 (082449Z uv-8에서 둘 다
  거짓 finish에 직접 기여 — 정독 노트 §4)
- S/R 개입 사각: 2연속 트리거 도달 시 회복 2/2인데 12건 중 8건이
  비연속·래치 소진으로 미도달 (정독 노트 §3 B층)
- 오형(BadArgs) 스트릭 디코딩층 사각: SR_PERTURB가 S/R 전용이라
  missing-field 5연속 복사 루프에 개입 없음 (정독 노트 §5)

**성공 기준.** ① 신규 장치의 유닛·통합 테스트 전건 + 기존 게이트(tests·
clippy·verify 12/12·3/3) ② 회귀 게이트 통과(§5 — 스포트 ≥33/36) ③ 행동
지표는 관찰 기록만: "공허 검증을 근거로 한 FINISH_NUDGE 무장 0건"은 코드
계약으로 기계 확인(테스트), 배치 관찰은 판정 조건이 아님. **uv 통과율
개선은 목표가 아니다.**

**비범위.** A층 착수 실패 대응(뮤테이션 0 finish 게이트·상태선 조기
발화·과제 앵커 — M13 후보), 문법 제약 디코딩(백로그), grep 계수 헤더·
부분문자열 힌트(백로그), ④ grep-first 맵(기각 유지, 정독 노트 §6), 32K
측정, 신규 크레이트(금지 유지), tasks/·tasks-large/ 픽스처 변경(없음 —
판정기 불변).

## 2. 기함: 검증 실질 접지

원리(M11 연장): 하네스가 **이미 아는** 정보 — cargo test가 몇 개를 돌려
몇 개가 실패했는지는 stdout에 있다 — 를 모델과 하네스 장치 자신에게
접지한다. 지시가 아니라 사실 제시(소형모델 네이티브 원칙).

### 2-1. libtest 요약 파서 (순수 모듈)

신설 순수 함수 모듈(위치는 플랜에서 확정 — run_command·status_note·
finish_nudge 배선이 공유하는 lib 공용 지점). 입력: run_command 결과
body(절단 가능). 출력: `Option<TestSummary>`:

- `ran`(합산), `passed`, `failed`, `failed_names`(수집 순서, 상한 내),
  `filtered_out`(합산)
- 근거 줄: `running N tests`, `test <name> ... FAILED`,
  `test result: ok|FAILED. P passed; F failed; …; K filtered out; …`
  — 워크스페이스는 테스트 바이너리별 섹션이 반복되므로 **전 섹션 합산**
- libtest 요약 줄이 하나도 없으면 `None` — **모든 소비처는 None이면 기존
  동작 그대로**(보수 폴백: cargo가 아니거나, nextest거나, 문구가 바뀌었거나,
  8000바이트 중간 절단으로 요약이 잘렸으면 그냥 조용히 물러난다)
- 커맨드 문자열이 아니라 **출력 형상으로 감지**한다(`cargo test -p …`,
  래퍼 스크립트 등에 강건; run_command에 국한된 후처리라 다른 도구 무영향)

### 2-2. run_command 0-테스트 무효화 노트

Done ∧ exit 0 ∧ `summary.ran == 0` ∧ `summary.filtered_out > 0`이면
파이프 노트와 같은 자리(본문 뒤 append, 영문 1줄):

```
note: 0 tests ran (N filtered out) - cargo test filters by test NAME,
not file name; this exit 0 did not verify anything
```

- `filtered_out > 0` 조건이 "테스트가 원래 0개인 크레이트"(정상)와
  구분한다. exit 0 한정(실패 exit는 이미 자체 신호가 있음).
- 파이프 노트와 공존 가능(두 줄 — uv-8의 실제 명령은 파이프 없음이지만
  152633Z run-3의 `| tail` 위장과 결합될 수 있음).
- 타임아웃·취소 경로에는 붙지 않음(파이프 노트와 동일 규율).

### 2-3. 상태선 verification 필드 실질화

`StatusNote::record_command_exit(body)`를 확장(exit 접두 파싱에 더해
TestSummary 파싱을 함께 저장). 렌더 우선순위:

1. `mutated_since_verify` → `verification: none since your last edit` (불변)
2. TestSummary 有 ∧ `failed > 0` →
   `verification: last cargo test: K failed (name1, name2 and M more)`
   (실패명 2개 + "and M more", 이름은 마지막 세그먼트만이 아니라 libtest가
   출력한 전체 경로명 그대로 — 모델이 파일을 특정할 근거)
3. TestSummary 有 ∧ `ran == 0` ∧ `filtered_out > 0` →
   `verification: last cargo test ran 0 tests (filter matched nothing)`
4. TestSummary 有 ∧ 전부 통과 → `verification: last cargo test: all P passed`
5. TestSummary 無 → 기존 문안(`last command exited N` / `gave no exit
   code`) 유지

- 마커 계약(`"[status] "` 접두·9칸 연속 들여쓰기) **불변** — 내용 행만
  풍부해진다. 계약 공유 3파일(status_note.rs·session.rs·exp_metrics.py)
  중 앞 둘은 무변경, exp_metrics는 마커 존재만 세므로 기존 컬럼 무영향
  (§6에서 신규 컬럼만 추가).
- 덮어쓰기 핀(무-exit 본문이 stale 값을 덮는 규칙)은 TestSummary에도 동일
  적용 — 이전 요약이 다음 명령 결과로 항상 갱신/무효화된다.
- fail-fast로 가려진 잔여 실패는 표시할 수 없다(하네스도 모름) — 표시하는
  것은 마지막 실행의 실질뿐(정직 한계, 정독 노트 §3 B층 각주).

### 2-4. FINISH_NUDGE·VERIFY_NUDGE 공허 검증 배제

- FINISH_NUDGE `VerifyOk` 판정(현행: run_command Ok ∧ 본문 첫 줄
  `exit code: 0`)에 **∧ ¬(summary.ran==0 ∧ filtered_out>0)** 추가 —
  공허 런으로는 무장하지 않는다. uv-8에서 FINISH_NUDGE가 0-테스트 exit
  0을 근거로 거짓 finish를 처방한 역효과의 직접 수선.
- VERIFY_NUDGE 해제 조건(현행: run_command Ok이면 exit 무관 해제)에 동일
  술어 추가 — 공허 런은 "검증 시도"로 인정하지 않는다(노트 §2-2가 같은
  결과 안에서 이유를 접지하므로 모델은 다음 행동 근거를 갖는다).
- 두 장치의 술어는 §2-1 파서 호출 1회를 공유(배선 지점에서 계산해 전달).

## 3. 오형(BadArgs) 개입 일반화 — 소형 2종

### 3-1. 동일 오류 스트릭 온도 섭동 확장

- 현행: `tracker.sr_streak() >= 2` → `SR_PERTURB_TEMPERATURE`(0.7)
  오버라이드, run() 지역 수명.
- 확장: 술어를 `sr_streak() >= 2 ∨ badargs_streak() >= 2`로.
  `badargs_streak()`은 기존 동일-오류(첫 문장 키) 연속 추적을 재사용하되
  **`Error: invalid arguments: missing field` 접두 키로 한정**한 연속
  카운터. 0-매치 오류 등 다른 오류류로는 확대하지 않는다(오발동 봉쇄 —
  §7 리스크).
- 텍스트 채널은 확대하지 않음: BadArgs 스트릭의 교정문은 기존
  REPEAT_CORRECTION·동일오류 3연속 교정이 담당(현행 유지). 이 개입은
  M10 실험 1이 입증한 "저온 복사 어트랙터는 텍스트가 아니라 온도로
  깬다"의 트리거 확대일 뿐, 새 메커니즘이 아니다.

### 3-2. salvage 역방향 규칙 (args 안 `tool` 키)

관측 오형(정독 노트 §5): `{"args": {"path": …, "tool": "write_file"},
"tool": "write_file"}`(동일 중복 — 지배적) 및 `read_file` 액션에
`args.tool = "list_files"`(상이 — 의도 표현, 082449Z uv-2 5연속 정지의
원인). 규칙(디스패치 지점 — 레지스트리 조회 가능):

1. args에 문자열 값의 `tool` 키 존재 ∧ 값 == 액션 tool → 키 제거
   (salvaged 마크·기존 교정 노트 채널)
2. 값 != 액션 tool ∧ 값이 등록 도구명 → **액션 tool을 args의 값으로
   교체**하고 키 제거(salvaged + 교정 노트 — 모델의 최신 의도가 args
   쪽에 표현된 관측 사례 기반)
3. 값이 미등록 이름 → 키만 제거(잔여는 해당 도구의 BadArgs 경로가 처리)

protocol.rs의 결정론(BTreeMap 순서) 관례 유지; 이 규칙은 파싱 단계가
아니라 디스패치 직전 정규화로 배선한다(레지스트리 의존 때문). **정규화는
승인 게이트·preview 판정보다 먼저 적용한다** — 규칙 2의 교체로 비뮤테이션
액션이 뮤테이션 도구(write_file 등)로 바뀔 수 있으므로, 게이트는 교체
결과 도구를 기준으로 판정해야 한다.

## 4. S/R 트리거 도달률 + 소품 2종

### 4-1. 파일별 누적 S/R 카운터

- 현행 사각: `sr_streak()`은 연속 전용(read/cargo가 끼면 리셋),
  SR_CORRECTION 래치는 런당 1회 → B층 12건 중 8건 미개입(회복 1/8).
- 신설: **파일별 누적** S/R 카운터(키 = edit_file `path` 인자 문자열
  그대로 — 어휘적 키, 상태선 파일 목록의 lexical dedup과 동일 관례;
  비연속 허용). 해당 파일에 성공 뮤테이션(edit_file/write_file 성공
  디스패치)이 나면 그 파일 카운터 리셋.
- SR_CORRECTION 발화: 파일 누적 2 도달 시, **래치 파일별 1회**(런당
  1회에서 완화 — M10 arm-block의 파일별 계수를 차단이 아니라 교정 발화에
  재사용하는 구도).
- 섭동 술어 확장: "마지막 도구 결과가 S/R 오류 ∧ 그 파일 누적 ≥2"를
  §3-1 술어에 OR로 추가. 연속 2회 케이스에서는 기존 동작과 동일(연속
  2회면 누적도 2)하고, 비연속 재발 시점부터 새로 커버한다. 오버라이드
  수명·원복 규칙(run() 지역, 스트릭/조건 해소 시 원복)은 현행 유지.
- 기존 `sr_streak()`은 제거하지 않는다(연속 신호 자체는 지표·기존 테스트
  계약에 남음).

### 4-2. 소품 2종 (edit_file)

1. **S/R 동일 오류문에 무변경 사실 명시**: 현행 처방 문장 뒤에
   `The file was NOT modified - it still contains your search text
   unchanged.` 추가. 첫 문장(오류 키 — 스트릭·지표가 의존)은 **불변**.
   (082449Z uv-1의 "이미 수정돼 있다" 오독 직격.)
2. **검사 순서 교체**: 3단 매치 사다리 실행 → 0매치면 closest 인용
   오류(현행 문안) → ≥2 매치 ∧ ¬replace_all이면 다중 매치 오류 →
   매치가 확정된 뒤에만 search==replace 동일성 검사.
   현행은 동일성 검사가 먼저라 **존재하지 않는 환각 코드**가 "S/R
   동일"로 위장된다(082449Z uv-3 t14 — calc_total 환각이 S/R 오류로
   보고돼 오신념이 교정 기회를 잃음). 교체 후: 환각은 0매치+closest
   인용으로 즉시 반증된다. 부수 효과: sr_error 지표에서 "가짜 S/R"이
   0매치로 이동(정화) — 배치 간 sr_error 비교 시 각주 필요(§6).

## 5. 측정: 회귀 게이트 (경량 사전등록)

실험이 아니라 회귀 게이트다 — 신규 장치가 기존 성능을 깨지 않음을
확인하는 것이 목적이고, 개선 입증은 목적이 아니다. 단 PROTOCOL의 "GPU
배치 전 사전등록" 규율은 유지한다(경량 양식: 배치 목록·게이트 값·재측정
규칙·중단 규칙만).

- 배치 2개(ornith-1.0-9b, seed 0, 조건은 대조 스탬프와 동일):
  1. tasks/ 스포트 36런 — v2 조건(ctx 8192·out 4096·timeout 60·temp
     0.1), 비교 대조 = 115152Z. **게이트: ≥33/36.**
  2. tasks-large uv+fm@8K 20런 — M11 조건(timeout 240), 비교 대조 =
     082449Z. **게이트 아님, 관찰**(참고 하한 없음 — 통과율 개선을
     목표로 하지 않는다는 §0-3 결정의 직접 반영).
- 게이트 미달 시: 실패 런 전수 법의학 → 신규 장치 귀속 실패가 없으면
  **1회 재측정 ≥33/36 사전 공약**(M11 개정 1의 규칙을 사후 개정이
  아니라 처음부터 내장), 귀속 실패가 있으면 해당 장치 수선 후 재게이트.
- 관찰 지표(승패 아님, 기록 의무): 0-테스트 노트 발동 수, verification
  실질 렌더 분포, sr recovered(대조 23/30), 오형 스트릭발 반복정지,
  missing-field 오형률(배치 센서스 — M11 상승 인과 미확정 건의 추적 지속).
- 수행 규율: 러너(loco-experiment-runner) 무인 — lms 교체 AskUserQuestion
  승인 선행, 통지 의존 금지·종료는 exit code+스탬프로 확인, setsid
  데몬화(하네스 백그라운드 60분 상한 — 8K 배치 실측 61분), 측정 중 빌드
  병행 금지, `ls ${TMPDIR}/.cargo` 사전 점검.
- 총 GPU 예산: ~1.5–2h (스포트 ~40분 + 8K ~60분). 32K 없음.
