# M6 — 판정·평가 신뢰성 개편 설계

승인일: 2026-07-12. 본 문서는 loco 설계 스펙(2026-07-02)의 M6 마일스톤을 구체화한다.
직전 상태: M5 종료(main `fbde03f` 계열 머지) — gemma-4-e4b 66.7%, qwen3-vl-4b 50.0%
(`docs/baselines.md` "M5 최종 결과").

## 1. 배경 — 왜 판정기인가

M5는 스캐폴딩을 개선했고 측정 도구는 그대로 두었다(M5 스펙 §2가 과제 변별력 개편을
M6으로 명시 연기). 그 결과 남은 것은 스캐폴딩 문제가 아니라 **측정 도구 자체의 신뢰성
문제**다:

- **관대한 채점** (기준선 한계 1항): outcome 무관 check 종료 코드만 보므로
  MaxTurns·RepetitionStop으로 끝난 실행도 작업만 되어 있으면 통과. 과제 수행 능력과
  종료 규율이 한 지표에 섞여 있고, 분리하려면 outcome 분포를 수동 교차 해석해야 한다.
- **판정기 협소** (기준선 한계 7항): find-definition의 판정은 trim·`\`→`/`·선두 `./`
  제거까지는 정규화하지만, 따옴표로 감싼 답·후행 슬래시 같은 흔한 변형은 실패 판정.
  fix-off-by-one의 zero 테스트는 버그 상태에서도 통과하는 비변별 케이스.
- **거짓 성공 finish 계측 불가**: gemma M5 최종에서 6건(find-definition 3·count-usages 2·
  implement-from-doc 1, report.json 20260712T110845Z) — answer 계열 5건은 "자신 있는
  오답"(모델 실패)인지 "맞는 답의 형식 불인정"(판정기 실패)인지 현재 판정기로 구분할
  수 없다. answer 계열이 아닌 implement-from-doc 1건은 정규화로 해소되지 않는 진짜
  모델 신호로 남는다.
- **변별력이 검증되지 않는 속성**: check가 "버그 상태에서 실패하고 올바른 해결에서
  통과"하는지 보증하는 장치가 없다 — 과제를 수정·추가할 때마다 수동 확인에 의존한다.

이후 마일스톤(멀티라인 편집 심층, 새 과제 등)의 효과 측정이 전부 이 도구 위에서
이루어지므로, 스캐폴딩을 더 파기 전에 자를 먼저 신뢰할 수 있게 만든다.

## 2. 목표·비목표·성공 기준

**목표**:
1. 판정기 변별력을 기계 검증 가능한 속성으로 승격 — 과제별 레퍼런스 솔루션 +
   메타테스트(`eval --verify`)
2. 정답 형식 협소로 인한 오판 제거 — answer 계열 정규화 사다리
3. 과제 수행 능력과 종료 규율의 분리 측정 — 이중 리포트(`passed_strict_count`·`false_finish_count`)
4. 양 모델 풀 재측정으로 v2 기준선 수립

**비목표 (동결·이월)**:
- **에이전트 코드 동결**: `src/agent`·프롬프트·도구(`src/tools`) 일절 무변경 — v2 수치
  변화가 순수하게 판정기 변경분임을 보장한다. 구현 완료 시 diff로 확인
- 신규 과제 추가 없음 (기존 12개 수선만). 메타테스트 인프라 확보 후 후속 마일스톤에서
  확장하는 편이 안전하다
- 하드닝 부채 이월 유지: SALVAGE_NOTE 소실 경로, `$CARGO_HOME`/홈 `.cargo` 벡터,
  트립와이어 정리, grep 폴백+절단 결합 테스트, RepetitionTracker 윈도 경계 테스트
- 수정의 최소성·품질 판정(기준선 한계 2항)은 다루지 않음 — 행동 기반 판정 유지

**성공 기준** (M6 완료 판정):
1. `cargo run -- eval tasks/ --verify`가 12과제 전부 양방향 ✓ (변별성 + 해결가능성),
   **그리고** answer 계열 판정 테스트가 정규화 사다리 자체의 단위 테스트를 포함해
   메타테스트 2단계에서 함께 실행·통과 — 메타테스트는 정규형 솔루션만 보므로,
   사다리의 변형 허용·거부는 이 단위 테스트가 담보한다
2. 이중 리포트가 report.json·stdout에 반영되고 기존 필드는 이름·의미 하위 호환
3. v2 기준선 양 모델 측정 완료(각 `--repeats 3`) + baselines.md "v2 기준선" 절 문서화
4. 에이전트 코드 diff 0 · `cargo test`·`cargo clippy --all-targets -- -D warnings` 통과

통과율 수치 자체는 성공 기준이 아니다 — 판정기가 바뀌므로 수치 이동은 결과이지
목표가 아니다.

## 3. 판정기 수선 정책

- **answer 계열 정규화 사다리** (find-definition, count-usages — 판정 테스트가
  answer.txt를 읽는 과제): ① trim → ② 감싼 따옴표쌍(`"` `'` `` ` ``) 제거 → ③ 경로
  답변이면 `\`→`/` 치환, 선두 `./`·후행 `/`·후행 `.` 제거 후 정확 일치; 숫자 답변이면
  정수 파싱 후 수치 비교. **여러 줄·산문 답변은 계속 실패** — "한 줄로 저장" 지시
  불이행은 모델 실패이지 판정기 협소가 아니다. 사다리는 하네스가 아니라 **각 과제의
  판정 테스트 코드 내부**에 구현하고, 같은 파일에 고정 문자열 케이스로 사다리 자체의
  단위 테스트를 둔다(§7) — check 실행 시 항상 함께 돌므로 eval·verify 양쪽에서
  검증된다.
- **개별 테스트 변별성 감사**: 메타테스트가 보증하는 것은 check **전체 수준**의
  변별성(버그 상태에서 테스트 하나라도 실패하면 성립)이다. fix-off-by-one의 zero처럼
  비변별인 개별 테스트는 판정을 약화시키지 않으나, 12개 과제 전수 감사에서 발견되면
  변별 케이스로 교체한다(저비용 강화). 감사 결과는 과제별 수선 내역으로 baselines.md
  v2 절에 기록한다.
- **관대→정확의 방향**: 수선의 목적은 "맞는 답을 맞다고 판정"이지 통과율 부양이
  아니다. 정규화 확대는 위 사다리로 한정하고, 각 항목의 근거를 해당 판정 테스트 안에
  주석으로 남긴다.
- 픽스처 수정 시 edit-crlf-file의 `.gitattributes` 바이트 고정(CRLF)을 깨지 않는다.

## 4. 메타테스트 — `eval --verify`

- **호출**: `cargo run -- eval <tasks-dir> --verify`. 기존 eval 서브커맨드의 플래그로,
  LLM·서버 불필요(순수 로컬 실행) — 현행 main은 eval 분기 **앞**에서 `resolve_model`이
  `/v1/models`를 호출하므로(src/main.rs), verify 경로는 클라이언트 생성·모델 해석을
  건너뛰도록 재배선한다. `--repeats`/`--seed`/`--timeout-scale` 중
  `--timeout-scale`만 유효(check 실행 시간에 관여), 나머지와는 상호 배타로 거부한다.
- **과제 레이아웃 확장**: 각 과제 디렉터리에 `solution/` 추가 —
  `task.toml` + `fixture/` + `solution/`. `solution/`은 fixture와 같은 상대 경로 구조의
  **오버레이**(변경·추가되는 파일만 담고, 적용 시 덮어쓴다). 파일 삭제는 표현할 수
  없으나 현 12개 과제 모두 불필요 — 필요해지는 시점에 확장한다. `solution/`이 없는
  과제는 검증 실패로 처리해 전 과제가 레퍼런스 솔루션을 갖도록 강제한다. 솔루션은
  의도된 해결의 문서 역할을 겸한다. edit-crlf-file의 `solution/`처럼 EOL이 판정에
  관여하는 파일은 `tasks/.gitattributes`에 fixture와 동일하게 바이트 고정을 추가한다.
- **검증 시맨틱** (과제당, 기존 샌드박스·`tools/exec.rs` 재사용):
  1. fixture → 임시 샌드박스 복사 → check 실행 → **exit≠0 기대** (변별성: 버그/미해결
     상태에서 실패해야 한다)
  2. `solution/` 오버레이 복사 → check 재실행 → **exit==0 기대** (해결가능성)
  1↔2단계는 같은 샌드박스를 재사용한다(오버레이 방식과 정합, 웜 빌드로 소요 절감).
  check 타임아웃은 eval의 check와 동일하게 `check_timeout_secs`(기본 120s)에
  `scaled_timeout`(saturating, cap 3600s) 적용. `solution/`이 protected 경로(+암묵
  `.cargo`)를 덮는 파일을 포함하면 해당 과제는 검증 실패로 처리한다 — 판정기를 바꾼
  상태의 해결가능성 증명을 차단.
- **출력**: 과제 × (변별성 / 해결가능성) ✓/✗ 표를 stdout에. 전 과제 양방향 통과 시
  exit 0, 하나라도 실패 시 exit 1. report.json은 쓰지 않는다(측정이 아니라 게이트).
- **미적용은 protected 재동기화만**: 에이전트가 없으므로 재동기화는 불필요. 반면
  `.cargo` 트립와이어는 에이전트 벡터뿐 아니라 잔류 환경 벡터(`${TMPDIR}/.cargo` 등
  상위 경로의 가짜 러너)도 잡는 장치이므로 **verify에서도 실행한다** — 신뢰의 근원인
  게이트 자체가 오염된 환경에서 조용히 왜곡되는 것을 방지.

## 5. 이중 리포트

- **report.json**: per-run 데이터(outcome·passed)는 현행 유지. 과제별(TaskReport)과
  전체(Report 최상위)에 **정수 카운트 집계 필드 추가** — per-run bool `passed`·과제별
  `pass_rate`와의 키 충돌을 피해 `_count` 접미사로 명명:
  - `passed_count` — check 통과 수(주 지표, per-run passed의 합)
  - `passed_strict_count` — outcome==Finished **이고** passed (종료 규율 지표)
  - `false_finish_count` — outcome==Finished인데 !passed (자신 있는 오답 지표)
  기존 필드는 이름·의미 불변(하위 호환). `effective_config` 스냅샷 유지.
- **stdout 표**: 과제별 행에 엄격 통과 열 병기, 요약 줄에 두 통과율과 거짓 성공 수를
  함께 표기 (예: `통과 24/36 (66.7%) · 엄격 22/36 (61.1%) · 거짓 성공 finish 2`).
- 효과: 거짓 성공 finish가 grep 집계가 아닌 일급 지표가 되고, §3 수선 이후 남는
  건수는 순수하게 모델의 "자신 있는 오답" 신호가 된다. 한계 1항의 "Finished만 인정"
  환산도 report.json에서 즉시 읽힌다.

## 6. v2 기준선 재측정·문서화

- **측정 조건**: 기존 프로토콜 그대로 — LM Studio 모델 단독 로드 ctx 8192, 로컬 config
  `max_output_tokens = 4096`, seed 0, `--repeats 3`, 측정 중 병행 빌드 금지, 트립와이어
  사전 점검(`ls ${TMPDIR}/.cargo`). gemma-4-e4b·qwen3-vl-4b 각 1회(모델당 ~45-75분,
  모델 교체는 사용자 협조).
- **순서 보장**: 재측정은 판정기 수선 + 메타테스트 12/12 통과 **이후에만** 실행한다
  (측정 후 재수선으로 v2가 다시 무효화되는 것을 방지).
- **문서화**: `docs/baselines.md`에 "v2 기준선" 절 신설 — 과제별 수선 내역(감사 결과
  포함), 양 모델 v2 수치(통과/엄격/거짓 성공), **v1 계열 수치와 직접 비교 불가** 명시
  (판정기가 다르다). M7+ 마일스톤은 v2를 비교 기준으로 사용한다. CLAUDE.md에
  `--verify` 커맨드, `solution/` 구조, 이중 리포트 필드를 반영한다.

## 7. 테스트 전략

- **verify 모드 로직**: 가짜 과제(임시 디렉터리, `sh -c` check)로 기존 eval 통합 테스트
  패턴을 따라 `#[cfg(unix)]` 테스트 4경로 — 변별성 실패(픽스처에서 check 통과),
  해결가능성 실패(솔루션에서 check 실패), `solution/` 부재, 양방향 통과.
- **집계 필드**: `passed_strict_count`/`false_finish_count` 계산·직렬화 단위 테스트 (outcome 조합별).
- **판정 테스트 자체**: 정규화 사다리는 각 픽스처의 판정 테스트 코드 안에 구현하고,
  **같은 파일에 사다리 단위 테스트**(고정 문자열 케이스 — 따옴표쌍·후행 슬래시 허용,
  여러 줄·산문 거부 등)를 둔다. 메타테스트는 정규형 솔루션만 통과시키므로 사다리의
  변형 허용·거부는 이 단위 테스트만이 담보한다. 하네스 쪽에는 중복 테스트를 만들지
  않는다.
- 플래그 상호 배타(`--verify` + `--repeats` 등) 거부 경로 테스트.

## 8. 구현 순서 (플랜 힌트)

1. verify 모드 + `solution/` 12개 작성 → 현행 판정기로 실행. 정규형 솔루션은 현행
   판정기로도 통과하므로 12/12가 기대치다 — 이 단계의 감사가 찾는 것은 **솔루션 저작
   오류와 변별성 회귀**이지 협소성이 아니다. 협소성 수선 목록은 기준선 한계 7항 +
   M5 거짓 성공 분석에서 이미 확보되어 있다(find-definition·count-usages 정규화 사다리,
   fix-off-by-one zero 교체)
2. 판정기 수선(정규화 사다리 + 사다리 단위 테스트, 비변별 테스트 교체) → 메타테스트
   12/12 ✓ 유지 확인
3. 이중 리포트(집계 + stdout 표)
4. 문서화(CLAUDE.md·baselines.md 골격) → v2 재측정(양 모델) → baselines.md 수치 확정

1→2 순서의 의미: 메타테스트를 먼저 세워 두면 수선이 회귀 게이트 위에서 이루어진다
(수선이 변별성·해결가능성을 깨면 즉시 드러난다).
