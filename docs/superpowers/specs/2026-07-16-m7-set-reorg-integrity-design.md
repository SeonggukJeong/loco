# M7 — 모델 세트 재편·속도 지표 마감·판정 무결성 부채 설계

승인일: 2026-07-16. 본 문서는 `docs/m7-candidates.md`(스코핑 입력)를 브레인스토밍으로
확정한 결과다. 직전 상태: M6 완료(main `ddd6489` 계열), v2 기준선 확정 —
gemma-4-e4b 72.2%(엄격 69.4%), qwen3-vl-4b 50.0%(엄격 33.3%), 탐색 ornith-1.0-9b
94.4%(엄격 94.4%) (`docs/baselines.md` "v2 기준선").

## 1. 배경 — 무게중심 결정 경위

m7-candidates의 세 트랙 중 **트랙 A 경량(A1+A3) + 트랙 C(C1·C2·C4)**를 선택했다.

- **A 경량인 이유**: 9B 실측(94.4%·4B급 속도 67.3s/런)이 방향 전환("저사양 하드웨어에서
  대형 모델")에 유효 신호를 줬고, 모델 세트 재편은 그 후속으로 자명하다. 반면
  A2(큰-컨텍스트 과제군)는 기준선 무효화를 수반하는 큰 설계라 M8 후보로 미룬다.
- **A3 재해석**: 후보 문서의 "추론 속도 1급 지표화"에서 *속도를 높이는 일*은 서빙
  런타임(LM Studio 등)의 몫으로 loco 밖이다 — M7의 A3는 **측정 지표 마감**으로
  한정한다. 조사 결과 과제별 `avg_duration_secs`와 표의 "평균 시간" 열은 초기
  하네스부터 이미 구현돼 있고, 잔여 갭은 최상위(전체) 평균/런 하나다. tokens/sec
  계측은 하지 않는다.
- **트랙 B 제외**: B1(multiline)·B4(count-usages)는 스캐폴딩 변경이라 "판정기 변경과
  스캐폴딩 변경을 한 마일스톤에 섞지 않는다"(M6 원칙)에 따라 별도 마일스톤으로.
  C3(기존 백로그)도 에이전트 코드 변경이라 같은 이유로 제외.

**마일스톤 성격**: 판정기(tasks/)·스캐폴딩(src/agent·src/tools 프로덕션 경로) **모두
불변**. v2 기준선 비교 가능성이 그대로 유지되고, 재측정이 하나도 필요 없다.

## 2. 목표·비목표·성공 기준

**목표**:
1. 모델 세트 재편 — qwen3-vl-4b 은퇴, ornith-1.0-9b를 기준선으로 승격 (문서 작업만)
2. 속도 지표 마감 — report.json·요약 라인에 최상위 평균/런 추가 (순수 추가)
3. 판정 무결성 구멍 마감 — `$CARGO_HOME`/홈 `.cargo` 변조를 스냅샷으로 감지
4. 테스트 보강 2건 + 문서 스테일 정리

**비목표 (명시)**:
- A2 큰-컨텍스트 과제군(신규 과제 세트 설계) — M8 후보
- tokens/sec 계측 — 필요해지면 그때 (usage 청크 조사부터)
- CARGO_HOME **격리**(check 실행 시 임시 디렉토리로 교체하는 예방책) — 백로그로 기록,
  스냅샷 감지가 뚫리는 정황이 나오면 승격
- 에이전트·도구·프롬프트·tasks/ 프로덕션 변경 일절 없음 (C2는 `#[cfg(test)]` 모듈만)
- 모든 재측정 — Ornith 승격은 기존 측정의 재지정이다 (§3)

**성공 기준**:
1. `cargo test` 전부 통과 + `cargo clippy --all-targets -- -D warnings` 클린
2. `cargo run -- eval tasks --verify` 12/12 유지 (tasks/ 불변 확인)
3. report.json 변경은 신규 키 추가만 — 기존 키 보존을 스키마 테스트로 보장 (M6 §5 전례)
4. `git diff`에 tasks/ 변경 없음, src/agent·src/tools 변경은 `#[cfg(test)]` 내부뿐
5. 문서 3종(docs/baselines.md, README.md, CLAUDE.md) 갱신 완료

## 3. A1 — 모델 세트 재편 (문서만, 코드 0)

재편 후 기준선 세트: **google/gemma-4-e4b(4B 대표) + ornith-1.0-9b(Qwen3 계열 대표)**.

- **qwen3-vl-4b 은퇴**: v2 수치는 baselines.md에 역사 기록으로 유지하고 "은퇴(M7)"를
  표기한다. 이후 마일스톤의 측정 대상에서 제외. 은퇴 사유를 수치로 명기: 같은 계열
  9B가 4B급 속도로 대폭 상회(50.0%→94.4%), 종료 규율 약점(엄격 격차 16.7pp)도 9B에서
  해소(0pp).
- **Ornith 승격**: 기존 탐색 측정(report `20260716T062853Z`, 하네스 커밋 `4cb7325`)을
  기준선으로 **재지정**한다. 재측정하지 않는 근거: v2 기준선과 완전히 동일한
  프로토콜(단독 로드 ctx 8192, `max_output_tokens` 4096, seed 0, `--repeats 3`)과 동일
  하네스 커밋으로 이미 측정됐고, 이후 main에는 문서 커밋만 쌓였다.
- **"대체가 아닌 재편" 명기**: 계열은 같아도(Qwen3) 크기(4B→9B)·변종(vl vs Ornith
  파인튜닝)이 달라 동일 모델 대체가 아니다 — baselines.md에 의도적 세트 재편으로
  기록하고, 두 모델 수치를 시계열로 잇는 해석을 금지하는 주의를 남긴다.
- **문서 반영**: baselines.md에 "모델 세트 재편(M7)" 절 신설 + 기존 "탐색: 9B" 절을
  기준선으로 재지정(탐색 단서 제거·재편 절 참조로 대체), README 상태·요지 표 갱신
  (속도 열 포함 — 모델 비교에 평균 s/런을 1급 정보로 노출, A3와 짝).

## 4. A3 — 속도 지표 마감 (eval 코드, 순수 추가)

현황: 과제별 `avg_duration_secs`(report.rs)와 stdout 표 "평균 시간" 열은 이미 있다.
갭은 최상위 하나:

- **report.json**: `Report`에 최상위 `avg_duration_secs` 추가 = 전체 런의
  `duration_secs` 평균(**런 가중** — 과제별 평균의 평균이 아님; `total_pass_rate`와
  같은 정의 철학으로, 중단으로 반복 수가 다른 과제가 있어도 왜곡되지 않는다).
  기존 키는 전부 불변 — 스키마 테스트(`report.rs`의 키 목록 테스트)에 신규 키를 추가.
- **요약 라인**: `render_table()` 마지막 줄에 `· 평균 {:.1}s/런` 추가.
- 기존 report.json과의 호환: 추가 키만이므로 과거 리포트를 읽는 grep 레시피
  (docs/baselines.md)는 영향 없음.

## 5. C1 — `$CARGO_HOME`/홈 `.cargo` 변조 스냅샷 감지

**위협 모델**: cargo의 config 탐색은 cwd 상향(기존 트립와이어가 차단) 외에
`$CARGO_HOME/config.toml`(미설정 시 `~/.cargo/config.toml`)도 읽는다. 에이전트가
run_command로 거기에 가짜 러너 등을 심으면 이후 `check`의 cargo 동작을 오염시킬 수
있다 — 현행 트립와이어의 문서화된 잔여 한계.

**설계 — 감지(변조), 예방 아님**:
- **스냅샷 시점**: 하네스 시작 시 1회. 해석된 CARGO_HOME(env `CARGO_HOME` → 없으면
  `std::env::home_dir()` 밑 `.cargo`; Rust 1.86+에서 deprecated 해제, 현행 1.97)의
  `config.toml`·`config`(레거시명) **내용 바이트**를 기록한다. 파일 부재도 상태로
  기록(`Option<Vec<u8>>`) — 없던 파일이 생기는 것도 변조다. 파일이 작으므로 해시 없이
  바이트 비교(M6의 mtime 불신 교훈 승계 — mtime은 쓰지 않는다).
- **비교 시점**: 매 런의 `judge()`에서 기존 `cargo_tripwire(&sb.root)` 직후, `check`
  실행 **전**. 불일치 시 기존 트립와이어와 동일하게 하네스 중단(anyhow 에러 → exit 1,
  한국어 메시지에 변조된 파일 경로 명시). 정리는 하지 않는다(기존 "감지만, 정리 안 함"
  철학 유지).
- **오탐 수용**: 측정 중 사용자가 config를 직접 편집하면 중단된다 — 드물고, 중단이
  안전한 방향이므로 수용(측정 중 병행 작업 금지 프로토콜과도 일관). 문서에 명기.
- **verify 경로 제외**: `--verify`는 에이전트(LLM)가 없어 런 중 변조 주체가 없다 —
  스냅샷 비교가 공허하므로 넣지 않는다. 사용자의 기존 정당 config는 위협 모델 밖
  (현행 eval도 동일하게 읽는 조건에서 측정해 왔다).
- **테스트**: CARGO_HOME 해석과 스냅샷 비교를 경로 주입식으로 분리(env 전역 오염 회피,
  기존 `cargo_tripwire_from(root, base)` 패턴과 동일) — 무변조 통과·내용 변경 감지·
  생성 감지·삭제 감지 케이스.
- **문서·주석 정합**: mod.rs의 "잔여 한계" 주석, baselines.md 한계 절, CLAUDE.md의
  "unblocked" 문구를 갱신(감지로 승격, 격리는 백로그).

## 6. C2 — 테스트 보강 (프로덕션 코드 불변)

1. **grep 폴백+절단 결합**: 잘못된 정규식 → 리터럴 폴백 헤더가 붙은 상태에서 출력
   절단이 겹칠 때의 동작을 고정하는 테스트 (두 기능은 각각 테스트돼 있으나 결합
   경로가 비어 있음).
2. **RepetitionTracker 윈도 경계 재푸시**: 8턴 윈도에서 밀려난 해시가 다시 들어올 때
   카운트가 리셋되는지(오래된 히트가 3회 판정에 합산되지 않는지) 경계 케이스 고정.

둘 다 `#[cfg(test)]` 모듈 추가만 — 프로덕션 경로 diff 0.

## 9. 구현 순서 (플랜 힌트)

1. A3 최상위 평균 (report.rs — 작고 독립)
2. C1 스냅샷 감지 (eval/mod.rs + 테스트)
3. C2 테스트 2건
4. A1 문서 재편 + C4 스테일 정리 (코드 작업과 독립, 마지막에 몰아서)
