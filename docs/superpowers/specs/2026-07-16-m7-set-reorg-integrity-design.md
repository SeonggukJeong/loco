# M7 — 모델 세트 재편·속도 지표 마감·판정 무결성 부채 설계

승인일: 2026-07-16. 본 문서는 `docs/m7-candidates.md`(스코핑 입력)를 브레인스토밍으로
확정한 결과다. 직전 상태: M6 완료(main `ddd6489` 계열), v2 기준선 확정 —
gemma-4-e4b 72.2%(엄격 69.4%), qwen3-vl-4b 50.0%(엄격 33.3%), 탐색 ornith-1.0-9b
94.4%(엄격 94.4%) (`docs/baselines.md` "v2 기준선").

## 1. 배경 — 무게중심 결정 경위

m7-candidates의 세 트랙 중 **트랙 A 경량(A1+A3) + 트랙 C(C1·C2·C4)**를 선택했다.

- **A 경량인 이유**: 9B 실측(94.4%·4B급 속도 67.3s/런)이 방향 전환("저사양 하드웨어에서
  대형 모델")에 유효 신호를 줬고, 모델 세트 재편은 그 후속으로 자명하다. 반면
  A2(큰-컨텍스트 과제군)는 기준선 무효화를 수반하는 큰 설계라 M8 후보로 미룬다.
- **A3 재해석**: 후보 문서의 "추론 속도 1급 지표화"에서 *속도를 높이는 일*은 서빙
  런타임(LM Studio 등)의 몫으로 loco 밖이다 — M7의 A3는 **측정 지표 마감**으로
  한정한다. 조사 결과 과제별 `avg_duration_secs`와 표의 "평균 시간" 열은 초기
  하네스부터 이미 구현돼 있고, 잔여 갭은 최상위(전체) 평균/런 하나다. tokens/sec
  계측은 하지 않는다.
- **트랙 B 제외**: B1(multiline)·B4(count-usages)는 스캐폴딩 변경이라 "판정기 변경과
  스캐폴딩 변경을 한 마일스톤에 섞지 않는다"(M6 원칙)에 따라 별도 마일스톤으로.
  C3(기존 백로그)도 에이전트 코드 변경이라 같은 이유로 제외.

**마일스톤 성격**: 판정기(tasks/)·스캐폴딩(src/agent·src/tools 프로덕션 경로) **모두
불변**. v2 기준선 비교 가능성이 그대로 유지되고, 재측정이 하나도 필요 없다.

## 4. A3 — 속도 지표 마감 (eval 코드, 순수 추가)

현황: 과제별 `avg_duration_secs`(report.rs)와 stdout 표 "평균 시간" 열은 이미 있다.
갭은 최상위 하나:

- **report.json**: `Report`에 최상위 `avg_duration_secs` 추가 = 전체 런의
  `duration_secs` 평균(**런 가중** — 과제별 평균의 평균이 아님; `total_pass_rate`와
  같은 정의 철학으로, 중단으로 반복 수가 다른 과제가 있어도 왜곡되지 않는다).
  기존 키는 전부 불변 — 스키마 테스트(`report.rs`의 키 목록 테스트)에 신규 키를 추가.
- **두 duration의 의미 구분**: per-run `duration_secs`는 **에이전트 실행 시간만**
  계측한다(`run_once`가 `agent.run`을 감싸고 `judge`의 check는 그 뒤 — check·샌드박스
  오버헤드 제외). 신규 최상위 평균도 이 정의를 승계하므로 기존 최상위
  `duration_secs`(벽시계 총합, check 포함)를 런 수로 나눈 값과 **일치하지 않는다**
  (ornith 실측: 벽시계 68.1s/런 vs 에이전트 평균 67.3s/런). 두 필드에 의미 구분
  주석을 달아 소비자 혼동을 막는다.
- **요약 라인**: `render_table()` 마지막 줄에 `· 평균 {:.1}s/런` 추가.
- 기존 report.json과의 호환: 추가 키만이므로 과거 리포트를 읽는 grep 레시피
  (docs/baselines.md)는 영향 없음.

## 5. C1 — `$CARGO_HOME`/홈 `.cargo` 변조 스냅샷 감지

**위협 모델**: cargo의 config 탐색은 cwd에서 **파일시스템 루트까지 모든 조상**의
`.cargo/config.toml`을 읽고, 추가로 `$CARGO_HOME/config.toml`(미설정 시
`~/.cargo/config.toml`)을 읽는다. 기존 트립와이어의 실효 검사는 **`temp_dir/.cargo`
하나뿐**이다 — 샌드박스 루트가 `temp_dir` 바로 아래라 `parent()`가 곧 `temp_dir`이고
루프가 `base`(=temp_dir)에서 중단하므로(`mod.rs cargo_tripwire_from`), temp_dir
**상위** 조상(예: macOS `/var/folders/<...>` — 사용자 쓰기 가능)은 스캔되지 않는다.
즉 열린 벡터는 두 갈래다: (1) `$CARGO_HOME`/홈 config, (2) temp_dir 상위 조상의
`.cargo/config.toml`. 에이전트가 run_command로 어느 쪽에든 가짜 러너 등을 심으면
이후 `check`의 cargo 동작을 오염시킬 수 있다. (기존 CLAUDE.md의 "a `.cargo` anywhere
above the sandbox aborts the harness" 서술은 이 실효 범위를 과장한 것 — C4에서 정정.)

**설계 — 감지(변조), 예방 아님**:
- **감시 대상**: ① 해석된 CARGO_HOME(env `CARGO_HOME` → 없으면 `std::env::home_dir()`
  밑 `.cargo`; 현행 1.97에서 비-deprecated 실측)의 `config.toml`·`config`(레거시명),
  ② **temp_dir의 모든 조상(파일시스템 루트까지) 각각의 `.cargo/config.toml`·
  `.cargo/config`** — 위협 모델의 두 갈래를 같은 메커니즘으로 덮는다. 존재-중단이
  아니라 **변화-감지**이므로, TMPDIR가 홈 아래인 환경에서 정당한 `~/.cargo`가 조상에
  걸려도 오탐하지 않는다(기존 트립와이어를 루트까지 올리는 대안을 배제한 이유).
- **스냅샷 시점**: 하네스 시작 시 1회. 각 대상 파일의 상태를 3종으로 기록 —
  **부재 | 읽기불가(권한 등) | 내용 바이트**. 파일이 작으므로 해시 없이 바이트
  비교(M6의 mtime 불신 교훈 승계 — mtime은 쓰지 않는다). 없던 파일이 생기는 것,
  읽히던 파일이 읽기불가로 바뀌는 것 등 **상태 전이 일체가 변조**다. 시작 시점의
  읽기불가는 상태로 기록만 하고 진행(깨끗한 측정을 사전 조건으로 막지 않는다).
  CARGO_HOME 해석 자체가 불가(홈 없음 + env 미설정)면 ①만 생략하고 stderr로 알린다.
- **비교 시점**: 매 런의 `judge()`에서 기존 `cargo_tripwire(&sb.root)` 직후, `check`
  실행 **전**. 불일치 시 기존 트립와이어와 동일하게 하네스 중단(anyhow 에러 → exit 1,
  한국어 메시지에 변조된 파일 경로 명시). 정리는 하지 않는다(기존 "감지만, 정리 안 함"
  철학 유지).
- **오탐 수용**: 측정 중 사용자가 config를 직접 편집하면 중단된다 — 드물고, 중단이
  안전한 방향이므로 수용(측정 중 병행 작업 금지 프로토콜과도 일관). 문서에 명기.
- **verify 경로 제외**: `--verify`는 에이전트(LLM)가 없어 런 중 변조 주체가 없다 —
  스냅샷 비교가 공허하므로 넣지 않는다.
- **위협 모델 밖(정직한 잔여)**: 하네스 시작 **전부터** 존재하던 오염 config는 변화-
  감지로 잡을 수 없다(사용자의 정당 config와 구별 불가 — 현행 eval도 동일 조건에서
  측정해 왔다). 이 잔여까지 닫는 것은 백로그의 **CARGO_HOME 격리**(§2 비목표) 몫이다.
- **테스트**: CARGO_HOME 해석과 스냅샷 비교를 경로 주입식으로 분리(env 전역 오염 회피,
  기존 `cargo_tripwire_from(root, base)` 패턴과 동일) — 무변조 통과·내용 변경 감지·
  생성 감지·삭제 감지 케이스(+조상 대상 경로 열거 케이스).
- **문서·주석 정합**: mod.rs의 "잔여 한계" 주석, baselines.md 한계 절, CLAUDE.md의
  "unblocked" 문구와 **"anywhere above the sandbox" 과장 서술**을 갱신(감지로 승격,
  격리는 백로그; CLAUDE.md는 영문 유지).

## 6. C2 — 테스트 보강 (프로덕션 코드 불변)

1. **grep 폴백+절단 결합**: 잘못된 정규식 → 리터럴 폴백 헤더가 붙은 상태에서 출력
   절단이 겹칠 때의 동작을 고정하는 테스트 (두 기능은 각각 테스트돼 있으나 결합
   경로가 비어 있음).
2. **RepetitionTracker 부분 축출 경계**: 완전 축출-재푸시(밀려난 해시의 재등장 →
   `Ok`)는 기존 `window_caps_at_eight_entries`가 이미 커버한다 — 중복 금지. 비어 있는
   케이스는 **부분 축출 오프바이원**: 새 푸시 자체가 가장 오래된 동일 해시를 밀어내
   윈도 내 카운트가 3에 못 미치는 경계(예: 히트 2 + 패딩 6 = 윈도 만석에서 3번째
   히트 푸시 → 최고령 히트 축출 → 카운트 2 → `Ok`)와, 패딩 하나 적어 3회가 성립하는
   직전 케이스(`InjectCorrection`)의 쌍. 정확한 산술은 플랜에서 확정.

둘 다 `#[cfg(test)]` 모듈 추가만 — 프로덕션 경로 diff 0.
