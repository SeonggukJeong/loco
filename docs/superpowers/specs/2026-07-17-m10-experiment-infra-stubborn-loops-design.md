## 1. 배경 — 두 개의 문제를 함께 푼다

**문제 A (개입)**: M9 잔존 실패 2건은 텍스트 교정으로 못 끊는다는 실측이 있다.

- **완고 S/R 루프** — E fm0 정독(18이벤트): 파일을 한 번 읽은 뒤
  `LineKind::Sale => acc - line.amount_krw,`를 search와 replace에 동일하게 넣는
  호출을 6연속. 도구 처방(1회차)→SR_CORRECTION(2회차)→REPEAT_CORRECTION(3회째)
  3층이 전부 주입됐는데도 5번째 시도에서 블록을 몇 줄 변주했을 뿐 여전히
  search==replace. 사고(thought)는 올바른 의도를 서술 — "수정된 텍스트 생성"
  단계의 붕괴다. 정확-인자 반복만 막는 개입(윈도 기반)은 변주 케이스를 놓친다.
- **순수 재검증 루프** — B/E uv1: `cargo test` exit 0 동일 반복 5회 →
  반복정지가 FINISH_NUDGE보다 선점(M9 §4-2의 의도된 우선순위). 과제는 이미
  완료·검증된 상태였으므로 정지(exit 2)는 엄격 통과를 버리는 결정이다.

**문제 B (방법론)**: M9는 풀 배치 7회(~6시간+)를 쓰고도 완고 루프 표본이
배치당 1~2런이라 판정이 "방향성"에 머물렀다. lms CLI로 모델 교체까지 자동화
가능함이 확인됐으므로(M9 측정 실적), 표적 과제 고반복 + 무인 수행으로
정보/시간 비율을 끌어올리는 실험 체계 자체가 M10의 산출물이다.

기본 temperature 0.1(config 기본값)은 "저온 샘플링의 복사 어트랙터" 가설과
정합적 — 프롬프트에 무엇을 넣어도 디코딩이 복사를 이기지 못한다면 텍스트 교정
3층의 전면 실패가 설명된다. 이 가설은 실험 암 ③으로 직접 검증한다.

## 4. 개입 1 — S/R 강제 전환 (실험 암 ②)

텍스트가 아니라 **행동 공간을 좁히는** 4번째 단. 기존 3층(도구 처방 →
SR_CORRECTION → REPEAT_CORRECTION)은 불변, 그 위에 얹힌다.

- **상태**: 에이전트 루프에 파일별 S/R 오류 누적 카운트(`HashMap<경로, usize>`,
  키는 `edit_file` 인자의 `path` 문자열). 판정은 기존 SR_KEY(오류 첫 문장)
  일치. **누적이며 연속 불요** — 현행 스트릭(전역·연속)과 별도 상태인 이유:
  E fm0형 연속 루프뿐 아니라 "다른 시도가 끼어드는" 완고 케이스(F uv0형)도
  잡는다.
- **동작**: 같은 파일에 S/R 오류 **3회 누적** 시 그 파일을 차단 목록에 등록.
  이후 그 파일 대상 `edit_file`은 **디스패치 없이** 오류를 반환한다 —

  > `Error: edit_file is disabled for this file after repeated identical
  > search/replace failures. Rewrite the complete file with write_file,
  > applying your fix.`

  런당 파일별 지속(해제 없음 — 단순성 우선; write_file 성공 후에도 유지되며,
  그 파일의 후속 수정도 write_file로 하게 된다). 임계 3회의 근거: 1회차 도구
  처방·2회차 SR_CORRECTION이 이미 소진된 시점 — M9 §7이 예고한 "3회째 강제
  전환"과 일치. 0단계 ⓐ·ⓑ 결과로 최종 확정.
- **상호작용**: 차단 오류도 tool_result이므로 반복 윈도에 계수된다 — 모델이
  차단에도 계속 부딪히면 기존 반복정지(5회)가 백스톱. 차단 오류의 첫 문장은
  자체 스트릭 키가 되어 3연속 시 일반 교정(EDIT_STRATEGY_CORRECTION —
  "write_file로 재작성하라")이 얹히는데, 처방 방향이 차단 오류문과 일치하므로
  모순 없음. finish_nudge 상태기계에는 "edit_file 실패 시도"로 취급되어
  armed 해제(M9 §4-2 표의 기존 행 그대로 — 신규 분기 없음).

## 5. 개입 2 — 디코딩 섭동 (실험 암 ③)

프롬프트 층이 아니라 **디코딩 층**을 겨냥 — 반복 호출이 문자 단위 복사라면
원인은 저온 샘플링의 복사 어트랙터라는 가설(H2)의 직접 검증.

- **트리거**: S/R 스트릭 2연속(SR_CORRECTION과 같은 시점, 같은 판정 —
  `tool == "edit_file"` ∧ SR_KEY 일치 ∧ 2연속) → 에이전트가 temperature
  오버라이드를 활성화(`Option<f32>`, 요청 조립 시 치환). 값은 **0.7**(기본
  0.1 대비 — 0단계 결과로 최종 확정).
- **원복**: S/R 스트릭이 끊기는 즉시(비-S/R 결과가 오면 — 성공이든 다른
  오류든) 오버라이드 해제. 래치 없음 — 스트릭이 다시 2연속에 도달하면 재활성.
- **관측성**: 섭동 발동·해제는 트랜스크립트의 S/R 오류 연쇄에서 결정론적으로
  유도 가능 — 별도 계측 코드·트랜스크립트 스키마 변경 불요. 지표 스크립트가
  유도 규칙을 구현한다.
- **부수 효과 감시**: 고온 구간에서 다른 품질 저하(환각·형식 붕괴) 가능 —
  실험 1의 보조 지표(엄격 통과율·거짓 finish·salvage 빈도)로 감시한다.

## 6. 개입 3 — 재검증 루프 선점 교체 (암 ②·③ 공통 탑재)

M9 §4-2의 "반복정지 우선" 결정을 **좁은 케이스에서만** 뒤집는다.

- **조건**: 반복 윈도 5회째 동일 히트(현행 Stop 판정)가 동시에 ① 디스패치된
  도구가 `run_command` ② 결과 본문 첫 줄 `exit code: 0` ③ finish_nudge
  상태기계가 `armed` ④ FINISH_NUDGE 미래치 — 전부 충족이면 정지 대신
  FINISH_NUDGE를 주입(래치 소진)하고 루프를 계속한다.
- **이후**: 니지에도 동일 반복이 이어지면 6회째 히트가 Stop(래치가 이미
  소진됐으므로 조건 ④ 불충족 — 기존 반복정지 경로). 즉 "정지 1회 유예"다.
- **근거**: 관측된 uv1 패턴은 과제 완료·검증 후의 루프 — 정지(exit 2)는 엄격
  통과를 버린다. 조건 ①~③이 "성공 검증의 문자 그대로 반복"만 좁게 잡으므로
  미완성 조기 finish 압박은 제한적이고, 거짓 finish 카운트로 감시한다(§8 보조
  지표). S/R 암 비교와 교락되지 않도록 판정은 uv1 패턴 발생 런 전수 나열로만
  한다.

## 7. 실험 인프라

**7-1. loco 최소 기능** (`src/main.rs`, `src/eval/`): `eval --filter <과제명>`
— 반복 지정 가능한 clap 플래그, 과제 디렉토리명 정확 일치, 매치 0이면 하네스
오류(exit 1), `--verify`와 조합 가능(표적 검증). report.json은 필터된 과제만
담고 스키마 불변. 제품 표면 추가는 이 플래그 하나다.

**7-2. 암 = 브랜치**: 개입에 config 토글을 만들지 않는다. 실험 암은 피처
브랜치 빌드로 구분한다 —

| 암 | 브랜치 | 내용 |
|---|---|---|
| ① 기준선 | `m10/base` | main + §7-1 --filter + 인프라 파일 — 에이전트 행동 불변 |
| ② 강제 전환 | `m10/arm-block` | base + §4 + §6 |
| ③ 디코딩 섭동 | `m10/arm-perturb` | base + §5 + §6 |

승자 암 브랜치가 main으로 병합되고(§6은 양 암 공통이므로 승자와 무관하게
포함), 패자 암은 브랜치로 보존. 각 배치 시작 전 러너가 `git rev-parse HEAD`와
빌드 로그를 리포트에 기록해 "잘못된 빌드로 측정" 사고를 방지한다.

**7-3. 지표 스크립트** (`scripts/exp_metrics.py`): M9 추출 레시피
(baselines.md)의 일반화·체크인. 입력 = eval 스탬프 디렉토리(들), 출력 = 런별·
암별 표: S/R 오류 수, 오류당 2시도 내 회복, 교정/차단/니지/섭동(유도) 발동,
finish 인자누락 연속 길이, 정지 원인 분류(S/R발/finish발/재검증발/기타),
report.json 집계(통과/엄격/거짓 finish) 병합. 마커 문자열은 §4 차단 오류문
포함해 갱신. `--selftest`가 고정 샘플 트랜스크립트(스크립트에 내장)로 자기
검증 — cargo 게이트 밖의 파이썬이므로 자체 게이트를 갖는다. 의존성은 표준
라이브러리만(폐쇄망 원칙의 개발 도구 연장).

**7-4. 실험 규약** (`docs/experiments/`):

## 8. 실험 1 — 사전등록 요강

본문은 `docs/experiments/2026-07-17-sr-loop-arms/pre-registration.md`로 별도
작성·승인한다(이 절은 요강 — 스펙 승인이 곧 사전등록 승인은 아니다).

- **가설**: H1 완고 S/R 루프는 행동 공간 차단(암②)으로 종결된다. H2 루프의
  원인은 저온 복사 어트랙터이며 디코딩 섭동(암③)으로 회복된다. H3 재검증
  루프는 선점 니지(§6)로 finish 전환된다.
- **표본**: ornith-1.0-9b@8K 고정(로드 12288, M8 공통 조건), 표적 2과제
  (fix-monthly-total·update-vat-rate) × 10반복(`--repeats 10 --seed 0`) ×
  암 ①②③ = 60런 ≈ 2.1h. **gemma 제외** — 27런 S/R 0회, 개입 발동 자체가
  없어 정보 0. 재검증 루프(H3)는 같은 배치의 uv 런에서 관측.
- **승자 검증**: 승자 암 @32K(로드 49152) 2과제 × 10반복 ≈ 1.2h + tasks/
  스포트 36런(v2 조건: timeout 60s·로드 8192) ≈ 0.9h, 게이트 ≥ 33/36.
  총 ≈ 4.5h + 모델 교체·게이트 — 밤샘 예산 내.
- **판정 규칙(사전 확정)**: 주 지표 = ① 암별 S/R발 반복정지 수 ② 완고 루프
  (동일 파일 S/R 3회+) 발생 런의 종결 전환율 ③ 오류당 2시도 내 회복률 ④ uv1
  패턴 런의 finish 도달. 보조 = 엄격 통과율 비악화·거짓 finish 비증가·평균
  턴/시간. 승자 규칙: 주 지표 우세 암을 병합, 동률이면 단순한 쪽(암② — 섭동은
  요청 파라미터를 건드리는 만큼 부수 효과 면이 넓다). 두 암 모두 기준선보다
  나쁘면 병합 없이 §2 비목표의 예비 후보(실패 턴 제거)를 M11 입력으로.
- **중단 규칙**: 하네스 오류(LLM 에러·부분 리포트) 시 해당 암 1회 재수행,
  재실패면 실험 중단·원인 조사. Ctrl+C 부분 리포트는 폐기하고 해당 암 재수행.
