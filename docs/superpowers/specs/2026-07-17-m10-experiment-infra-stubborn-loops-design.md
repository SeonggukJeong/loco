## 1. 배경 — 두 개의 문제를 함께 푼다

**문제 A (개입)**: M9 잔존 실패의 지배 축인 완고 S/R 루프는 텍스트 교정으로 못
끊는다는 실측이 있다. E fm0 정독(18이벤트): 파일을 한 번 읽은 뒤
`LineKind::Sale => acc - line.amount_krw,`를 search와 replace에 동일하게 넣는
호출을 6연속. 도구 처방(1회차)→SR_CORRECTION(2회차)→REPEAT_CORRECTION(3회째)
3층이 전부 주입됐는데도 4번째 시도에서 블록을 몇 줄 변주했을 뿐(여전히
search==replace) 5·6번째는 원래의 한 줄 인자로 **회귀**했다 — 변주조차
유지하지 못하는 회귀는 복사 어트랙터 가설을 강화하는 데이터다. 사고(thought)는
올바른 의도를 서술 — "수정된 텍스트 생성" 단계의 붕괴다. 정확-인자 반복만 막는
개입(윈도 기반)은 변주 케이스를 놓친다.

**재검증 루프는 이번 개입 대상이 아니다 (리뷰 1R에서 실측 반증)**: baselines.md
"M9 2단"이 "B/E uv1 = `cargo test` exit 0 반복 5회"로 기록했던 패턴의 실체는
**뮤테이션 0회**(edit_file/write_file 시도 자체가 0회)의
`cat inv-report/tests/check_vat_report.rs`(exit 0) 동일 반복이었다 — 두 런 모두
passed=false인 **미완성 탐색 루프**이며, finish_nudge는 뮤테이션 없이는 무장하지
않으므로 "정지 대신 finish 유도" 개입은 동기 사례에서 발동 자체가 불가능하고,
발동시킨다 해도 아무 작업도 안 한 런에 날조 finish를 유도한다. M9 데이터에서
반복정지가 엄격 통과를 실제로 버린 유일한 런은 F fd1(passed=True·
repetition_stop)인데 반복 히트가 write_file이라 이 역시 다른 패턴이다. 처분은
§6, 문서 정정은 §2 목표 4.

**문제 B (방법론)**: M9는 풀 배치 7회(~6시간+)를 쓰고도 완고 루프 표본이
배치당 1~2런이라 판정이 "방향성"에 머물렀다. lms CLI로 모델 교체까지 자동화
가능함이 확인됐으므로(M9 측정 실적), 표적 과제 고반복 + 무인 수행으로
정보/시간 비율을 끌어올리는 실험 체계 자체가 M10의 산출물이다.

기본 temperature 0.1(config 기본값)은 "저온 샘플링의 복사 어트랙터" 가설과
정합적 — 프롬프트에 무엇을 넣어도 디코딩이 복사를 이기지 못한다면 텍스트 교정
3층의 전면 실패가 설명된다. 이 가설은 실험 암 ③으로 직접 검증한다.

## 4. 개입 1 — S/R 강제 전환 (실험 암 ②)

텍스트가 아니라 **행동 공간을 좁히는** 4번째 단. 기존 3층(도구 처방 →
SR_CORRECTION → REPEAT_CORRECTION)은 불변, 그 위에 얹힌다.

- **상태**: 에이전트 루프에 파일별 S/R 오류 누적 카운트(`HashMap<경로, usize>`).
  키는 `edit_file` 인자의 `path`를 **렉시컬 정규화**한 값(`Path::components`
  정리 수준 — `./src/x.rs`와 `src/x.rs`의 분산·차단 우회 방지; 파일시스템
  조회는 하지 않는다). `path` 부재·비문자열(BadArgs)인 호출은 카운트·차단 판정
  모두 건너뛴다. 판정은 기존 SR_KEY(오류 첫 문장) 일치. **누적이며 연속 불요**
  — 현행 스트릭(전역·연속)과 별도 상태인 이유: E fm0형 연속 루프뿐 아니라
  "다른 시도가 끼어드는" 완고 케이스(F uv0형)도 잡는다.
- **동작**: 같은 파일에 S/R 오류 **3회 누적** 시 그 파일을 차단 목록에 등록.
  이후 그 파일 대상 `edit_file`은 **디스패치 없이** 오류를 반환한다 — 차단
  검사는 확인 게이트·preview보다 **앞**이다(preview의 파일 읽기·사다리 실행도
  건너뛰고, 대화형 모드에서 승인 후 차단되는 UX 모순을 없앤다). 모델에게
  가는 본문(영문, `Error:` 접두 — 오류 스트릭 게이트 통과용):

  > `Error: edit_file is disabled for this file after repeated identical
  > search/replace failures. Rewrite the complete file with write_file,
  > applying your fix.`

  런당 파일별 지속(해제 없음 — 단순성 우선; write_file 성공 후에도 유지되며,
  그 파일의 후속 수정도 write_file로 하게 된다). 임계 3회의 근거: 1회차 도구
  처방·2회차 SR_CORRECTION이 이미 소진된 시점 — M9 §7이 예고한 "3회째 강제
  전환"과 일치. 0단계 ⓐ·ⓑ 결과로 최종 확정. 발동 시 REPL에는 한국어 Notice
  한 줄(M5/M9 관례).
- **상호작용**: 차단 오류도 tool_result로서 반복 윈도에 계수된다(게이트 거부
  결과가 계수되는 기존 경로와 동일) — 단, 반복정지(5회) 백스톱은 모델이 **같은
  인자로** 차단에 부딪힐 때만 성립하고, search를 변주하면 윈도 키가 달라져
  실질 상한은 max_turns다. 차단 오류의 첫 문장은 자체 스트릭 키가 되어 3연속
  시 일반 교정(EDIT_STRATEGY_CORRECTION — "write_file로 재작성하라")이 얹힐 수
  있는데(`error_corrected` 래치가 미소진일 때만), 처방 방향이 차단 오류문과
  일치하므로 모순 없음. finish_nudge 상태기계에는 "edit_file 실패 시도"로
  취급되어 armed 해제(M9 §4-2 표의 기존 행 그대로 — 신규 분기 없음).

## 5. 개입 2 — 디코딩 섭동 (실험 암 ③)

프롬프트 층이 아니라 **디코딩 층**을 겨냥 — 반복 호출이 문자 단위 복사라면
원인은 저온 샘플링의 복사 어트랙터라는 가설(H2)의 직접 검증.

- **트리거**: S/R 스트릭 2연속(SR_CORRECTION과 같은 시점, 같은 판정 —
  `tool == "edit_file"` ∧ SR_KEY 일치 ∧ 2연속) → 에이전트가 temperature
  오버라이드를 활성화(`Option<f32>`, 요청 조립 시 치환). 값은 **0.7**(기본
  0.1 대비 — 0단계 결과로 최종 확정). 발동 시 REPL에는 한국어 Notice 한 줄.
- **원복 의미론(핀)**: S/R 스트릭 상태는 기존 `error_correction` 경로가 갱신
  한다 — 즉 **디스패치되거나 게이트에서 거부된 턴에서만** 움직인다. 따라서:
  비-S/R 결과(성공·다른 오류)가 오면 해제; 게이트 거부된 edit_file(`Denied:`
  본문)도 스트릭 리셋이므로 해제; 무액션 턴(파싱 재시도·length-cut)과 finish
  시도 턴은 스트릭 불변이므로 오버라이드 **유지**. 래치 없음 — 스트릭이 다시
  2연속에 도달하면 재활성.
- **관측성**: 섭동 발동·해제는 트랜스크립트의 S/R 오류 연쇄에서 결정론적으로
  유도 가능 — 별도 계측 코드·트랜스크립트 스키마 변경 불요. 지표 스크립트
  (§7-3)의 유도 규칙은 위 원복 핀과 **동일 규칙**으로 구현하고 selftest로
  고정한다.
- **부수 효과 감시**: 고온 구간에서 다른 품질 저하(환각·형식 붕괴) 가능 —
  실험 1의 보조 지표(엄격 통과율·거짓 finish·salvage 빈도)로 감시한다.

## 6. 강등 기록 — 반복정지 선점 교체 (v1 §6, 이번 마일스톤에서 제외)

v1 스펙은 "5회째 동일 히트가 armed 상태의 exit-0 run_command면 정지 대신
FINISH_NUDGE 1회 유예"를 개입 3으로 실었다. 리뷰 1R의 트랜스크립트 실측이
동기 사례를 반증했다(§1): B/E uv1은 비무장·미완성 런이라 조건이 성립하지 않고,
유일한 엄격 손실 정지(F fd1)는 write_file 반복이라 역시 불충족. 잘못된
문서 문장(baselines.md)이 설계로 전이된 케이스로, 스펙 §3이 약속한 "전제가
뒤집히면 개정·재리뷰"를 리뷰가 선행 수행한 셈이다.

**처분**: 개입에서 제외하고 백로그로 —
- 미완성 탐색 루프(B/E uv1형, `cat` 동일 반복)는 현행 반복정지가 **정당한
  종결**이다(finish 유도는 날조 위험). 별도 개입 없음.
- 완료 후 반복정지로 엄격 손실(F fd1형, write_file 반복) — 1런 관측, 재현
  데이터 축적 대기. 0단계 ⓓ가 법의학 노트에 근거를 남긴다.

## 7. 실험 인프라

**7-1. loco 최소 기능** (`src/main.rs`, `src/eval/`): `eval --filter <과제명>`
— 반복 지정 가능한 clap 플래그, 과제 디렉토리명 정확 일치, 매치 0이면 하네스
오류(exit 1), `--verify`와 조합 가능(표적 검증). 필터는 **전체 과제 로드·검증
후** 적용한다(비매치 과제의 task.toml 정의 오류도 계속 시끄럽게 — load_tasks
단일 지점이라 eval·verify 공용). report.json은 필터된 과제만 담고 스키마 불변.
제품 표면 추가는 이 플래그 하나다.

**7-2. 암 = 브랜치**: 개입에 config 토글을 만들지 않는다. 실험 암은 피처
브랜치 빌드로 구분한다 —

| 암 | 브랜치 | 내용 |
|---|---|---|
| ① 기준선 | `m10/base` | main + §7-1 --filter + 인프라 파일 — 에이전트 행동 불변 |
| ② 강제 전환 | `m10/arm-block` | base + §4 |
| ③ 디코딩 섭동 | `m10/arm-perturb` | base + §5 |

각 암은 단일 개입만 실어 지표 귀속이 깨끗하다. 승자 암 브랜치가 main으로
병합되고 패자 암은 브랜치로 보존. report.json은 암을 자증하지 못하므로
(`loco_version`이 전 브랜치 동일 — M7 절의 기지 한계), 러너가 **배치마다
eval 스탬프 ↔ `git rev-parse HEAD` 쌍을 report.md에 명시 기재**한다(§7-5).

**7-3. 지표 스크립트** (`scripts/exp_metrics.py`): M9 추출 레시피
(baselines.md)의 일반화·체크인. 입력 = eval 스탬프 디렉토리(들), 출력 = 런별·
암별 표: S/R 오류 수, 오류당 2시도 내 회복, 교정/차단/니지/섭동(유도) 발동,
finish 인자누락 연속 길이, 정지 원인 분류(S/R발/finish발/기타),
report.json 집계(통과/엄격/거짓 finish) 병합. 마커 문자열은 §4 차단 오류문
포함해 갱신. `--selftest`가 고정 샘플 트랜스크립트(스크립트에 내장)로 자기
검증 — cargo 게이트 밖의 파이썬이므로 자체 게이트를 갖는다. 의존성은 표준
라이브러리만(폐쇄망 원칙의 개발 도구 연장).

**7-4. 실험 규약** (`docs/experiments/`):
- `PROTOCOL.md` — ① 사전등록 없이 GPU 시간을 쓰지 않는다(사전등록 = 가설·암·
  표본·지표·판정 규칙·중단 규칙·예산이 승인된 문서) ② 측정 중 빌드/테스트 병행
  금지(CPU 경합 — CLAUDE.md 기존 원칙 승계) ③ 소표본 판정 규칙(M9 §2 승계) ④
  배치 전 게이트: 두 tasks 트리 `--verify` + 모델 로드 상태·컨텍스트 길이 검증
  ⑤ 리포트는 report.json 스탬프·git hash·lms 상태를 기록해 재현 가능하게
- `TEMPLATE.md` — 사전등록 양식
- 실험별 디렉토리 `YYYY-MM-DD-<slug>/` — `pre-registration.md`(승인본)와
  `report.md`(러너 초안 → 사용자 판정)

**7-5. 무인 수행 에이전트** (`.claude/agents/loco-experiment-runner.md`):
입력 = 승인된 사전등록 문서 경로. 동작 = 게이트(§7-4 ④) → 암별 체크아웃·빌드
(스탬프↔git hash 쌍 기록) → `lms unload --all`/`lms load <모델>
--context-length <N>` + `curl localhost:1234/api/v0/models` 검증 → 배치 순차
수행(`cargo run -- eval ... --filter ...`) → `scripts/exp_metrics.py` →
`report.md` 초안 작성. 금지 = 제품 코드 수정, 측정 중 빌드 병행(암 전환 시의
체크아웃·빌드는 배치 사이에만), 사전등록에 없는 배치 수행, 픽스처 변경.
LLM 에러로 부분 리포트가 나오면 해당 암 1회 재수행, 재실패면 실험 중단 후
보고(사전등록의 중단 규칙).

## 8. 실험 1 — 사전등록 요강

본문은 `docs/experiments/2026-07-17-sr-loop-arms/pre-registration.md`로 별도
작성·승인한다(이 절은 요강 — 스펙 승인이 곧 사전등록 승인은 아니다).

- **가설**: H1 완고 S/R 루프는 행동 공간 차단(암②)으로 종결된다. H2 루프의
  원인은 저온 복사 어트랙터이며 디코딩 섭동(암③)으로 회복된다.
- **표본**: ornith-1.0-9b@8K 고정(로드 12288, M8 공통 조건), 표적 2과제
  (fix-monthly-total·update-vat-rate) × 10반복(`--repeats 10 --seed 0`) ×
  암 ①②③ = 60런. **gemma 제외** — 근거는 §2 비목표(발동 임계 미도달, 정보 0).
- **승자 검증**: 승자 암 @32K(로드 49152) 2과제 × 10반복 + tasks/
  스포트 36런(v2 조건: timeout 60s·로드 8192), 게이트 ≥ 33/36.
- **시간 예산(상한치 — M9 실측 과제별 평균 기준)**: 8K 3암 ≈ 2.2~2.4h, 32K
  승자 배치 ≤ 1.5h(F 배치 uv 평균 458s/런 기준), 스포트 ≈ 0.9h — 총
  ≈ 4.6~4.8h + 모델 교체·게이트. 밤샘 예산 내. 사전등록에는 상한치로 기재
  (중단 규칙의 판단 기준).
- **판정 규칙(사전 확정)**: 주 지표 = ① 암별 S/R발 반복정지 수 ② 완고 루프
  (동일 파일 S/R 3회+) 발생 런의 종결 전환율 ③ 오류당 2시도 내 회복률. 보조 =
  엄격 통과율 비악화·거짓 finish 비증가·평균 턴/시간·salvage 빈도. 승자 규칙:
  주 지표 우세 암을 병합, 동률이면 단순한 쪽(암② — 섭동은 요청 파라미터를
  건드리는 만큼 부수 효과 면이 넓다). 두 암 모두 기준선보다 나쁘면 병합 없이
  §2 비목표의 예비 후보(실패 턴 제거)를 M11 입력으로.
- **중단 규칙**: 하네스 오류(LLM 에러·부분 리포트) 시 해당 암 1회 재수행,
  재실패면 실험 중단·원인 조사. Ctrl+C 부분 리포트는 폐기하고 해당 암 재수행.
