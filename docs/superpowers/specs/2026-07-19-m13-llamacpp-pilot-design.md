## 1. 배경과 이 마일스톤이 다른 점

M1~M12는 합성 평가(`tasks/`·`tasks-large/`)의 통과율로 진척을 판정해 왔다.
M12 스코핑에서 사용자와 합의한 전환점은 다음과 같다: **북극성 판정을 합성
평가에서 실사용 신호로 옮긴다.** 추가 스캐폴딩 마일스톤은 더 이상 기본값이
아니다.

여기에 M13 착수 시점에 사용자가 제기한 배포 제약이 겹친다: **사내 폐쇄망
머신에 LM Studio를 설치할 수 없다.** 따라서 실사용 파일럿을 하려면 스택을
llama.cpp로 옮겨야 한다. 이 둘은 별개 과제가 아니라 하나의 사슬이다 —
파일럿은 배포 스택 위에서 돌아야 의미가 있다.

M13의 성격은 이전 마일스톤과 다르다. **효과를 입증하는 실험이 아니라, 측정
대상을 바꾸는 이행 작업이다.** 통과율을 올리는 것이 목표가 아니다.

## 3. llama.cpp 전환과 앵커 배치

### 3-1. 실측으로 확인된 것

스펙 작성 전(스모크)과 리뷰 1R(검증 실행)에서 확인했다. llama-server b9960,
ornith-1.0-9b-Q4_K_M.

| 항목 | 결과 |
|---|---|
| `response_format` json_schema 강제 | **동작.** loco의 실제 스키마(properties 없는 `{"type":"object"}` args)로 args가 정상 채워짐. 불가능한 enum 주입 시 모델 의도를 무시하고 끌려감 → 문법 실강제 확인 |
| 컨텍스트 산식 | `-c 4096 -np 2` → `n_ctx_slot = 2048` (**분할 확인**). `-np` 생략(기본 `-1`=auto) 시 `n_slots = 4`이나 `kv_unified = 'true'`로 `n_ctx_slot = 8192` — **분할되지 않음** |
| 컨텍스트 초과 응답 | **`400`** + 본문 `"request (8010 tokens) exceeds the available context size (2048 tokens)…"` → `"context"` 포함. `looks_like_context_overflow`가 정상 발동, `src/agent/mod.rs:611-616` **변경 불필요** |
| `/v1/models` | 모델 id가 **GGUF 전체 경로**로 반환. `--alias ornith` 지정 시 `ornith`로 바뀜 |
| `--reasoning-format none` | **json_schema와 병용 시 전 요청 400** — C1, §3-3 참조 |

⚠️ 초판은 `-c 12288 -np 1 → 12288`을 근거로 분할을 주장했으나, **1로 나누는
것은 분할을 입증하지 못한다.** 위 표는 리뷰가 `-np 2`로 다시 확인한 값이다.
진짜 함정은 초판이 상상한 방향의 반대다 — 기본값은 분할하지 않으며, "안전하게"
`-np 4`를 주는 쪽이 컨텍스트를 1/4로 만든다.

### 3-2. 사고 토큰이 출력 예산을 잠식한다

llama-server는 Ornith의 사고 토큰을 **`reasoning_content`로 분리**한다.
loco는 이를 보지 못하지만 **토큰 예산은 소비된다.**

기전 정정: 초판은 "SSE 파서가 `delta.content`만 읽어서"라고 적었으나
**틀렸다.** 에이전트 경로는 `stream: false`이며(`src/agent/mod.rs:164`),
비스트리밍 응답에서 `ResponseMessage`(`src/llm/types.rs:38-43`)가
`role`/`content`만 역직렬화하므로 serde 단계에서 버려진다. 결론은 같고
기전이 다르다 — 그리고 **측정 경로는 스트리밍이 아니다.**

실패 양식은 초판이 적은 것보다 나쁘다. 리뷰 실측:

```
max_tokens=40   finish_reason='length'  content: ''   reasoning len: 192
max_tokens=120  finish_reason='length'  content: ''   reasoning len: 551
```

예산이 사고로 소진되면 **content가 잘린 접두가 아니라 완전히 빈 문자열**로
온다. loco는 견딘다(`mod.rs:224-227`이 `"(empty)"` 자리표시자를 밀어넣고
턴이 소비되며 `max_turns`가 경계를 준다) — 그러나 **salvage 여지가 0**이라
LM Studio의 잘린 응답보다 엄격히 나쁘다.

관측된 사고 길이는 39~551 토큰이고 `max_output_tokens = 4096`이므로 **eval
조건에서 위험은 실재하되 완만하다.** 긴 출력에서 문다.

⚠️ `--reasoning-format`은 이 문제를 **해결하지 못한다.** 그 플래그는 토큰을
어디에 *보고*할지만 정하고 생성을 막지 않는다(`--help`: "`none`: leaves
thoughts unparsed in `message.content`"). `--reasoning-budget 0`도 이 모델에서
사고를 억제하지 못함을 리뷰가 확인했다. **실효 레버는 `.loco/config.toml`의
`max_output_tokens` 상향 하나뿐이다.**

관측 방법(§9): 앵커 배치에서 `finish_reason == "length"` 턴 수를 센다.

### 3-3. 설계 — 핀 4개와 `scripts/serve.sh`

loco가 요청에 싣는 것은 `model`/`messages`/`temperature`/`max_tokens`/
`stream`/`response_format`/`seed` 뿐이고(`src/llm/types.rs:22-35`), 나머지
샘플러는 **서버 기본값에 좌우된다.**

`scripts/serve.sh` 하나가 다음을 고정하고, 그 스크립트가 곧 배포 산출물이자
실험 조건 기록이 된다:

1. **`-np 1`** — `n_ctx_slot == -c`를 결정론적으로 만든다. (기본 auto는
   `kv_unified=true`로 분할하지 않지만, 그 동작에 의존하지 않고 못 박는다.
   `--kv-unified` 상호작용을 스크립트 주석에 기록)
2. **샘플러 4종 명시** — `--top-k` / `--top-p` / `--min-p` /
   `--repeat-penalty`. 값은 구현 태스크에서 두 스택의 실효 기본값을 조사해
   결정하고, 근거를 스크립트 주석에 남긴다
3. **reasoning 처리 = 기본값(`auto`, 분리) 고정.** 이것은 선택지가 아니라
   **확정 사항**이다 — `--reasoning-format none`은 `response_format:
   json_schema`와 병용 시 b9960에서 **모든 요청을 400**으로 만든다
   (`Failed to initialize samplers: std::exception`). 이 문자열을 사유로
   스크립트 주석에 박는다. §3-2의 예산 문제는 이 핀이 다루지 않는다
   (`max_output_tokens`가 레버다)
4. **`--alias`** — 모델 id를 짧은 이름으로. `report.json` 최상위 `model`
   필드에 GGUF 전체 경로가 기록되는 것을 막는다 (§3-5)

핀의 목적은 LM Studio를 흉내내는 것이 **아니다.** 배포 타깃이 llama.cpp인
이상 LM Studio 기본값은 성역이 아니다. 목적은 (a) 우리가 무엇을 돌리는지 아는
것, (b) 배치가 차이를 재 주는 것이다.

#### 3-3-1. C1이 왜 Critical이었나 (설계 교훈, 보존)

`--reasoning-format none`의 실패는 조용하다. 400 본문에 `"context"`가 없어
오버플로 감지는 비켜가고, `src/agent/mod.rs:594`의 **json_schema 폴백 사다리가
대신 발동해 `Agent` 수명 내내 스키마 없이 돈다.** 그러면 매 턴 `parse_turn`이
실패하지만 배치는 정상 종료한다. `AgentEvent::Notice`는 eval에 영속되지
않으므로(`src/eval/`·`src/session.rs` 확인) **`report.json`에도 트랜스크립트에도
흔적이 없다.** 40분 배치가 그럴듯한 낮은 숫자를 뱉고 §3-6이 그것을 정당한
앵커로 처리했을 것이다.

이 조합 — 조용한 오설정 + 하한 없는 판정 규칙 — 이 §3-6의 결함 하한(C2)을
낳은 직접 근거다.

### 3-4. 배치 전 필수 스모크 (사전 점검)

앵커 배치를 포함한 **모든 배치 전에** 다음을 확인한다. LM Studio 전제인
`PROTOCOL.md` 항목 4의 ②③(`lms unload/load`, `curl /api/v0/models`)은 llama.cpp
에서 **전부 죽은 명령**이므로, 태스크 1에서 `PROTOCOL.md`를 갱신하는 것이
마일스톤 산출물에 포함된다.

1. json_schema 요청 1건이 **HTTP 200**을 반환 (C1 재발 방지)
2. 서버 기동 로그의 `n_ctx_slot == context_tokens`
3. `curl /v1/models`의 `data[0].id == <alias>`, `data[0].meta.n_ctx == context_tokens`
4. `.loco/config.toml`이 이번 배치 조건인지 — **현재 M12 배치 2의
   `command_timeout_secs = 240`으로 남아 있다**
5. `ls ${TMPDIR}/.cargo` 트립와이어 (존재 시 수동 제거)
6. 측정 중 빌드 병행 금지
7. 데몬화: **macOS에 `setsid`가 없다.** M12가 확립한
   `python3 -c "os.setsid(); os.execvp(...)"` 래퍼를 쓴다
   (하네스 백그라운드 60분 수명 상한)

### 3-5. `report.json`의 모델 기록 (초판 오류 정정)

초판은 "`model = \"\"` 자동선택이 `effective_config`에 박힌다"고 적었다.
**위치가 틀렸다.** `EffectiveConfig`(`src/eval/report.rs:20-28`)에는 model
필드가 없다. 그러나 `report.json` **최상위에 `model` 키가 있고 값도 채워진다** —
전 배치에서 `'ornith-1.0-9b'`로 확인.

따라서 pin 4(`--alias`)의 정당화는 유효하다: alias 없이 돌리면 그 최상위
필드에 GGUF 전체 경로가 들어간다. **`EffectiveConfig`에 model 필드를 추가할
필요는 없다** — 이미 기록되고 있다.

⚠️ 리뷰 1R의 I1은 "14개 배치 전부 `model=None`"이라며 구조체 필드 추가를
제안했으나, 이는 `effective_config` 내부만 조회한 결과다. 컨트롤러가
`report.json` 최상위를 직접 대조해 반증했다.

#### 3-6-2. 동등성 판정 규칙 (사전등록 대상)

하한을 통과한 배치에 대해:

- **판정은 (b) 하나로 내린다** — 총합이 대조(33/36) 대비 **±4런 이내**면 동등
  성립. (대조가 33이므로 상한 37은 도달 불가 — 실효 구간은 29~36이다)
- **(a) 안정 집합은 판정 항이 아니라 분류(triage) 방아쇠다** — 아래 6개 과제
  중 하나라도 3/3 아래로 떨어지면, **어떤 판정도 기록하기 전에 해당 런의
  트랜스크립트를 정독한다.** 읽고 실패가 스택 차이로 귀속될 때에만 판정을
  불성립으로 뒤집는다. 귀속되지 않으면 (b)의 판정을 그대로 기록하되 그
  정독 결과를 배치 기록에 남긴다

안정 집합 = **ornith 36런 배치 6개 전부**(`20260716T062853Z`·`20260717T041725Z`·
`20260717T215729Z`·`20260718T092740Z`·`20260718T115152Z`·`20260718T222824Z`)에서
3/3인 과제:

```
add-function, chain-edits, count-usages, create-module,
find-definition, fix-off-by-one
```

분산은 나머지 6개에 집중된다: `edit-crlf-file`·`fix-compile-error`·
`fix-failing-test`·`implement-from-doc`·`rename-function`, 그리고
`multiline-string-edit`(3,2,2,1,1,1 — 변동이 아니라 기지의 지속 약점이며
최근 배치에서 더 나빠졌다).

**동등 불성립 시**: llama.cpp 앵커를 새로 잡고 이후 비교에 각주를 단다. 원인
분석(§3-2의 사고 토큰 가설 우선)은 수행하되 **수선은 M13 범위 밖.**

##### 규칙의 근거 — 왜 (a)를 판정 항에서 뺐는가

초판의 ±3은 너무 좁다. 이항 근사로 P(|X−Y| > 3 | 두 스택 동일) ≈ 0.13이며,
런이 12과제×3반복으로 상관되어 있어 실제 오경보율은 그보다 높다. ±4는 같은
근사에서 ≈0.05다.

안정 집합 자체는 두 번의 반증을 거쳤다. 리뷰 1R의 원안은 **2개 배치**에서
3/3인 9과제였는데, 컨트롤러가 제3의 배치로 검증하니 그중 2개가 이미 2/3로
흔들렸다. 컨트롤러가 제시한 개정안은 **3개 배치** 교집합 7과제였는데,
재리뷰가 **6개 배치 전수**로 검증하니 `implement-from-doc`이
`20260716T062853Z`에서 2/3였다(컨트롤러 재확인 완료). 즉 집합을 넓힐 때마다
멤버가 떨어져 나갔다 — **표본을 늘릴수록 "안정"의 정의가 좁아지는 중이고,
6개가 마지막이라는 보장은 없다.**

더 근본적으로, (a)를 **연언 항으로 쓰면 18개 베르누이 시행이 전부 성공할 것을
요구**하게 되어 오경보율이 추정 불가능한 per-run 실패 확률에 과민해진다.
108/108이라는 증거는 "진짜 결정론적"(오경보 ≈1%)과 "p_fail = 0.01"(오경보
≈17%)을 **둘 다 허용한다.** 후자라면 내가 비판했던 ±3의 13%보다 나쁘다.
집합을 좁히는 것으로는 이 문제가 해결되지 않는다 — 시행 수는 줄어도 남은
과제마다 여전히 완벽한 3/3을 요구하기 때문이다.

그래서 (a)의 진짜 가치인 **진단력**만 남기고 판정 부담은 (b)로 옮겼다.
안정 집합이 흔들리는 것은 여전히 "스택이 뭔가 깼다"의 가장 좋은 신호이지만,
그 신호는 **사람이 읽어야 할 이유**이지 자동 판정이 아니다.

## 4. 실사용 파일럿

### 4-1. 무대와 신호

- **무대**: 사용자의 맥, **loco와 별개인 개인/OSS 레포**. loco 자신을 대상으로
  하면 마일스톤 중인 레포를 그 마일스톤의 도구로 고치는 오염이 생긴다
- **모델**: ornith-1.0-9b @ llama-server (§3의 핀 적용)
- **규모**: **20세션**, 탐색형. 임계값 없음
- **신호**: 실패 유형 분류(주) + 채택률·세션 판정(보조) — §4-4

### 4-2. `scripts/pilot.sh`

세션을 감싸는 얇은 래퍼다. loco 프로덕션 코드는 **0 변경** — M6부터 지켜온
"에이전트 코드 동결 = 비교가능성" 관례를 유지하며, 파일럿 종료 후 잔존물은
스크립트뿐이다.

```
[세션 전] 과제 유형 1단어 + 난이도 추정(상/중/하) 입력  ← 사후 합리화 차단
시작 rev 기록 → loco 실행 → 종료 직후 diff 캡처
→ 판정 한 줄 물음 → 원장(JSONL)에 한 행 append
```

원장 한 행: 세션 id, 시작/종료 rev, **과제 유형·사전 난이도 추정**, 과제 한 줄,
loco 세션 트랜스크립트 경로, 캡처된 diff, 소요 시간, 사용자 판정
(`성공` / `수정해서 씀` / `버림` + 사유).

**난이도를 세션 *전에* 받는 이유**: 사후에 매기면 결과를 아는 상태의 추정이
되어 분모로 못 쓴다. 20개 세션의 난이도 분포가 없으면 채택률 x%는 해석
불가능하다 — 쉬운 과제 20개와 어려운 과제 20개가 같은 숫자를 내고 뜻이
정반대다.

JSONL을 고른 이유: 실측 확인 결과 `.loco/sessions/*.jsonl`과 eval 트랜스크립트가
**같은 `kind`/`content`/`ts` 스키마**이므로 과정 지표는 `scripts/exp_metrics.py`
확장으로 재사용된다.

대안 기각: **loco 내장**(REPL이 직접 판정을 물음)은 누락 불가라는 장점이 있으나
파일럿 전용 기능이 프로덕션 코드에 영구 잔존한다. **사후 분석만** 하는 방식은
잔존물 0이나 세션 경계와 커밋 경계가 어긋나 채택 판정이 부정확하다.

### 4-3. 채택률 — 대리 지표임을 명시하고 보조로 강등

세션 종료 시점의 diff는 "제안"이지 "채택"이 아니다. 2단으로 나눈다: 세션
시점에는 diff를 보존만 하고, 파일럿 종료 시점에 각 세션이 추가한 줄이 최종
HEAD에 살아남았는지 대조한다.

⚠️ **줄 생존율은 대리 지표이며, 알려진 왜곡이 최소 5종이다**:

1. 사용자가 손으로 고쳐 쓰면 부분 생존
2. 무관한 리팩터링에 휩쓸리면 채택됐는데 미채택으로 계산
3. **가치가 삭제였던 세션**, 또는 **올바른 진단만 내놓고 구현은 사람이 한
   세션**은 생존율 0인데 성공이다
4. **rebase/squash가 앵커를 파괴한다** — 대상 레포에서 squash 머지가 일어나면
   줄 귀속이 끊긴다. 파일럿 기간 중 대상 레포에서 히스토리 재작성을 하지 않기로
   사전 약속한다
5. 같은 파일을 건드린 두 세션의 간섭은 캡처 시점이 아니라 **생존 대조 시점에**
   되돌아온다

세션 판정 한 줄과 **반드시 교차**해서 본다. 둘이 어긋나는 세션이 가장 읽을
가치가 있다. 다만 20세션×3판정이면 셀당 ≈6, 불일치 축은 n≈2이므로 **불일치는
통계가 아니라 사례로 읽는다.**

### 4-4. 주 산출물은 분류표다 (사전 선언)

채택률 하나로는 M14가 소비할 것이 없다. **주 산출물은 사전 선언된 실패 범주별
건수**이며, 채택률은 기술 통계로 강등한다.

범주는 **세션 1 이전에** 확정한다(사후에 그리면 일어난 일에 맞춰 그리게 된다).
M8/M9/M12가 확립한 어휘를 승계:

```
실패 없음 · S/R 루프 · 뮤테이션 0회 거짓 finish · 뮤테이션 없는 탐색 루프 ·
컨텍스트 오버플로 · 엉뚱한 파일 편집 · length 루프 · 인자 누락(BadArgs) ·
(기타 — 신규 범주는 발견 시 추가하되 추가 시점을 원장에 기록)
```

**분류 절차** (없으면 건수가 서로 비교 불가능해진다):

- **증거 출처를 분리해 기록한다.** `exp_metrics.py`가 트랜스크립트에서
  기계적으로 판정할 수 있는 범주는 그렇게 판정하고, 나머지는 사용자의 사유
  한 줄에서 분류한다. 각 건에 어느 쪽이었는지를 남긴다 — 둘이 어긋나는 것이
  §4-3이 말한 가장 값진 축이므로 **섞으면 그 축이 사라진다**
- **다중 라벨을 허용한다.** 한 세션이 S/R 루프와 뮤테이션 0회 거짓 finish를
  동시에 보일 수 있다. 따라서 **범주별 건수의 합은 세션 수와 같지 않다** —
  보고 시 이를 명시한다
- **`실패 없음`을 명시적 범주로 둔다.** 모든 범주가 실패 양식이면 성공한
  세션이 분류표 밖으로 떨어져 건수가 분모 없이 뜬다. 이 범주를 포함하면
  최소한 "세션 수 = 실패없음 + 실패를 하나 이상 가진 세션 수"가 성립한다

### 4-5. 이 설계의 가장 큰 미통제 요인

**n=1, 비맹검, 도구 제작자 본인의 자기 평가.** 사용자가 언제 loco를 부를지
고르며 그 기준이 20세션에 걸쳐 재조정되므로, 채택률은 도구의 능력만큼이나
사용자의 캘리브레이션을 잰다. 이것은 M13 범위에서 통제할 수 없다 —
**통제하는 대신 명시한다.** 채택률을 주 지표에서 내린 이유이기도 하다.

## 5. 저비용 수선

### 5-1. 전환 부수물

코드 실측 결과 LM Studio 결합은 **사용자 대면 문구 2곳**뿐이다
(`grep -rn "LM Studio" src/`가 이 둘 + `mod.rs` 주석만 반환):

- `src/llm/client.rs:13` — 연결 실패 안내
- `src/llm/client.rs:186` — 로드된 모델 없음 안내

둘 다 llama.cpp를 포함하도록 일반화하고, `src/llm/client.rs:297`의 단언
테스트를 함께 갱신한다. README의 서버 기동 절은 llama-server 경로를 1급으로
올린다.

`src/agent/mod.rs:611-616`의 오버플로 휴리스틱은 **변경 불필요**하다 — §3-1이
llama-server의 실제 응답이 `400` + 본문 `"context"`임을 확인했다.

`base_url` 기본값(`http://localhost:1234/v1`)은 **바꾸지 않는다.** 기본값 변경은
기존 사용자에게 파괴적이고 `serve.sh`가 포트를 고정하므로 문서화로 족하다.

태스크 1의 산출물에 **`PROTOCOL.md` 항목 4 갱신**(§3-4)과 **json_schema 폴백
발동의 `report.json` 기록**(§3-6-1)을 포함한다.

### 5-2. 상태선 무뮤테이션 접지 — 2건으로 확장

**당초 가정**: 케이던스 `[5,10,15,20]`(`src/agent/status_note.rs:15`)이
성기어서 거짓 finish를 놓쳤다.

**실측 재검토 결과 부정확하다.** M12 법의학(`report.md` §6-3,
`run-fix-failing-test-1`)의 실제 양식:

1. turn1 `run_command`(exit 101) → turn2·3 `read_file` → turn4 `edit_file`
   **실패**(성공 뮤테이션 0건) → turn5 `finish`(거짓 성공)
2. 케이던스 지점 turn5에 **도달은 했으나** 그 턴을 `finish`가 소비
3. `finish`는 루프가 처리하므로 **붙일 tool_result가 없다**

원인은 "성김"이 아니라 **구조**다. 상태선은 주입 채널이 tool_result인 한 finish
턴에 원리적으로 도달할 수 없다.

**증거 강도 (초판보다 상향)**: 초판은 n=1로 적었으나 **너무 낮췄다.** 리뷰가
두 번째 사례를 독립 확인했다 — `run-update-vat-rate-0`:
`grep → read_file → read_file → list_files → finish`, 5턴, 뮤테이션 0,
상태선 미렌더. 동일 형태이며 **케이던스에 3이 있었다면 둘 다 turn3에서
렌더됐을 것이다.** 다만 `run-update-vat-rate-5`는 turn2에서 finish하므로
**어떤 케이던스로도 도달 불가** — 이것이 이 수선의 상한을 준다.

#### 5-2-1. 수선 A — 케이던스 조밀화

`ZERO_MUT_CADENCE`를 `[3, 5, 7, 10, 15, 20]`로. 초기를 촘촘히 하고 후반은
유지한다.

비용 정정: 초판은 "노트 1건당 3행"으로 계산했으나 **틀렸다.** 무뮤테이션
분기는 조기 반환으로 **한 줄**이다(`status_note.rs:97-99`):

```
[status] files edited: none yet | turns: 5 of 25 used
```

즉 컨텍스트 비용을 3배 과대계상했다. 이 정정은 변경에 **유리한** 방향이다.

#### 5-2-2. 수선 B — 무뮤테이션 분기에 검증 줄 렌더 (리뷰 I7, 사용자 승인)

수선 A만으로는 "너는 아무것도 안 고쳤고 지금 3턴째다"를 더 일찍 말할 뿐이다.
turn3이면 모델이 파일 두 개를 막 읽은 시점이라 **한계 정보가 0에 가깝다.**

정작 값진 접지는 조기 반환이 막고 있다. 검증 줄은 뮤테이션이 있을 때만
만들어진다. `fix-failing-test-1`의 turn1은 `cargo test`(exit 101)였고
`parse_test_summary`가 그 본문에서 passed 4 / failed 1 /
`failed_names ["max_of_list"]`를 얻는다. 무뮤테이션 분기에서 이를 렌더하면
turn3에 이것이 모델 앞에 놓인다:

```
[status] files edited: none yet | verification: last cargo test: 1 failed (max_of_list) | turns: 3 of 25 used
```

**모델이 곧 환각으로 고치고 finish해 버린 바로 그 테스트의 이름이다.** 기존
규칙 2~5를 그대로 재사용하므로 변경 크기는 상수 조정과 비슷하다.

#### 5-2-3. 무엇을 주장하지 않는가

이 수선이 **뮤테이션 0회 거짓 finish 갭을 닫는다고 주장하지 않는다.**
`update-vat-rate-5`형(turn2 finish)은 도달 불가이고, 구조적 해법(finish 게이트)은
M14로 재이월한다. 주장하는 것은 "도달 가능한 케이스에서 접지의 정보량을
0에서 유의미로 올린다"뿐이며, 효과 입증은 M13의 목표가 아니다.

  6개는 **판정 항이 아니라 분류 방아쇠**다 — 흔들리면 재측정을 걸기 전에
  트랜스크립트를 먼저 읽고, §5-2 변경으로 귀속될 때에만 미달로 취급한다
- 미달 시: 재측정 **1회** 사전 공약. 재측정도 미달이면 §5-2 변경을 **되돌리고**
  (파일럿은 태스크 1 코드로 진행) 사용자에게 보고. 추가 재측정 없음

여기서 (a)를 연언 항으로 두지 않는 것이 §3-6-2보다 더 중요하다. 이 게이트의
미달 결과는 **~40분 GPU 재측정과 §5-2 작업 되돌림**이므로, 우연한 흔들림
하나가 그 비용을 자동으로 발동시켜서는 안 된다.

M12의 절대값 게이트(≥33/36) 대신 **앵커 상대값**인 이유는 태스크 2 이전에
llama.cpp에서의 절대 기준이 존재하지 않기 때문이다.

## 7. 성공 기준

M13은 통과율 개선 마일스톤이 아니므로 기준도 그에 맞춘다.

1. **전환 완료**: llama-server 위에서 `eval tasks --repeats 3`이 완주하고
   `serve.sh`가 조건을 재현 가능하게 고정한다. §3-4 스모크 7항이 배치 전 통과
2. **앵커 확립**: §3-6-1 하한과 기계 검사를 통과한 배치가 `docs/baselines.md`에
   스탬프와 함께 기록되고, §3-6-2 판정 결과가 명시된다
3. **회귀 없음**: §6-3 게이트 통과 (또는 되돌림 후 보고)
4. **파일럿 산출**: **수행된 세션 수(목표 20)**의 원장이 채워지고, **§4-4의
   사전 선언 범주별 건수**가 §4-4의 분류 절차대로 산출된다. 채택률·세션
   판정은 함께 보고하되 기술 통계다. (세션 수 미달을 허용하는 §9의 완화책과
   정합하도록 "20세션"을 고정 요구에서 목표치로 바꾼 것이다)
5. **M14 입력**: 범주별 건수에서 도출된 후보가 우선순위와 함께 기록된다

기준 4는 **수치의 크기를 요구하지 않는다.** 채택률이 낮게 나오는 것도
결과이며, 그 경우 "합성 평가 통과율과 실사용 채택률의 괴리"가 M14의 가장
중요한 입력이 된다.

## 8. 조사 결과 (기각 근거 보존)

### 8-1. ik_llama.cpp — 기각

- **가장 강한 근거**: ik는 **Vulkan 지원을 명시적으로 거부**한다. 배포 타깃이
  내장 그래픽을 가지므로 Vulkan 오프로드가 가장 큰 미개척 레버로 보이는데,
  ik를 택하면 그 카드를 버리고 시작한다
- **이득이 불확실**: 유리한 CPU 벤치마크는 2024-12 자료이며 그 사이 업스트림이
  CPU 재패킹을 상당 부분 따라잡았다. 2025~26년 제3자 보고 2건은 메인라인 계열
  Q4_K에서 ik가 **열세**였다(한 건은 PP ~9배). 큰 이득은 IQ*_K 재양자화를
  전제하는데 그 포맷은 업스트림이 읽지 못하므로 **되돌릴 수 없는 고착**이다
  (역방향은 열려 있다 — ik는 표준 GGUF를 읽는다)
- **loco에 직접 걸리는 것**: ik는 컨텍스트 초과에 **HTTP 500**을 반환한다.
  업스트림이 전용 400(`exceed_context_size_error`)을 주는 것과 대조되며(§3-1
  실측), loco의 감지는 *400 + 본문 "context"* 이므로 `pack()` 후 재시도가
  **아예 발동하지 않는다.** 조용한 회귀다
- **폐쇄망 운영 부담**: 사전 빌드 바이너리 0개, CI 워크플로 1개(CUDA 컨테이너
  전용), 2025-07 GitHub 계정 삭제·복구 전력. 업스트림은 Vulkan 포함 사전 빌드
  바이너리를 배포하므로 오프라인 설치 요건이 그대로 해결된다

프로젝트 자체는 건강하다(주간 16~40커밋, 기여자 복수, 최신 아키텍처 추종).
기각 사유는 품질이 아니라 **우리 시나리오와의 적합성**이다.

⚠️ 조사 한계: CPU 전용·최신·독립 재현 벤치마크는 확보하지 못했고
Ornith-1.0-9B가 ik에서 로드되는지도 미검증이다. 재검토 조건은 "대표 머신
1대에서 실제 Ornith Q4_K_M으로 8K/32K 벤치, PP 이득 30% 초과".

### 8-2. Vulkan iGPU 오프로드 — M14 이월

인용 수치는 프롬프트 처리 약 2배(~34→~76 t/s), 토큰 생성 거의 평평. loco의
부하는 매 턴 컨텍스트를 다시 밀어넣는 **프롬프트 처리 중심**이므로 사실이면
유리한 축이다.

⚠️ **출처가 블로그급이고 실리콘이 특정되지 않아 방향만 취하고 수치는 근거로
쓰지 않는다.** 개발 머신이 macOS/Metal이라 M13에서는 검증 수단이 없다.
