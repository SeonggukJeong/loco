# M8 — 대형 저장소 트랙 설계 (측정 + 레퍼런스)

승인일: 2026-07-16. 브레인스토밍으로 확정. 직전 상태: M7 완료(main `b9a4f3e`),
기준선 세트 gemma-4-e4b 72.2%(엄격 69.4%) + ornith-1.0-9b 94.4%(엄격 94.4%),
평균 s/런 gemma 52.3 · ornith 67.3 (`docs/baselines.md`).

## 1. 배경 — 방향 결정 경위

"쓸만하겠는데"의 판정 기준을 **폐쇄망 동료 배포**로 확정했고, 그 전제는 UX가 아니라
**모델 출력 품질**이다. 동료들의 주 작업은 **수만 라인 기존 사내 코드베이스** 수정이며
소규모 유틸은 부차적이다.

현 `tasks/` 12과제는 전부 소형 단일 크레이트라 이 장면을 전혀 측정하지 못한다.
ornith 94.4%는 "작은 코드에서의 추론 안정성" 점수이지 "큰 저장소에서 올바른 파일을
찾아 고치는 능력" 점수가 아니다. 8K 컨텍스트 소형 모델에게 대형 저장소는 항해
(navigation) 문제이고, 어디서 어떻게 무너지는지는 예측이 잘 맞지 않는 영역이다 —
따라서 M8은 **측정 먼저**다. M5~M7이 모두 eval이 방향을 정해줬기에 성공한 전례를
따른다.

레퍼런스 후보 조사 결과(2026-07-16 확인):

- **anthropics/claude-code**: 소스 비공개(이슈 트래커+문서+플러그인 저장소, CLI는
  번들 npm 배포). 코드 탐구 불가 — 행동 관찰 대상으로만 유효.
- **openai/codex**: codex-rs가 Rust 96.7%·Apache-2.0·소스 공개. 같은 언어, 같은
  OpenAI-호환 프로토콜, 로컬 모델(`--oss`) 지원 — **유일하게 유효한 동종 코드 탐구
  대상**.
- **xai-org/grok-build**: Rust 99.6%·Apache-2.0·소스 공개이나 풀스크린 TUI가
  정체성이라 loco의 REPL 노선과 결이 다름. UX 단계(후속) 참고감.
- **aider**(후보 목록 외 추가): repo map(tree-sitter 태그 추출 + 심볼 랭킹 + 토큰
  예산 맞춤)이 "작은 컨텍스트로 큰 저장소 항해"를 정면으로 푼 대표 설계 — 이번 문제에
  가장 수확이 큰 레퍼런스.

결정: **A(측정 트랙 신설) + B(레퍼런스 탐구) 병행**, 구현(C: repo-map 도구·컨텍스트
관리 개선)은 실패 데이터 확보 후 M9로. B는 컨텍스트 처리에 한정하지 않고, 실패
데이터가 가리키는 주제는 깊이 제한 없이 파는 상시 레퍼런스로 운용한다.

## 3. 픽스처 — 가상 사내 시스템 `inv-workspace`

일관된 도메인(재고/물류 관리)의 멀티 크레이트 워크스페이스를 손으로 작성해 커밋한다.

```
fixture/
  Cargo.toml          (workspace)
  .gitignore          (/target — 필수, 기존 규칙)
  inv-core/           도메인 타입·비즈니스 규칙
  inv-parse/          설정·데이터 파일 파서
  inv-store/          저장 계층 (인메모리·파일)
  inv-report/         집계·보고서 생성
  inv-cli/            바이너리 (얇은 CLI)
```

- 규모: 합계 10~20K LOC, 50~150 파일. 외부 의존성 0(워크스페이스 내부 path 의존만) —
  폐쇄망 `cargo test` 성립
- 각 크레이트에도 `/target` `.gitignore` 필요 여부를 제작 시 확인(워크스페이스
  루트 target 단일이면 루트 1개로 충분)
- 현실성 장치: 일부 한국어 주석, 모듈당 파일 수·길이 편차, 일부 400줄+ 파일
- CRLF 특수 케이스 없음 — `tasks/.gitattributes` 같은 바이트 고정 불필요, 일반 LF

### 함정 카탈로그 (11종)

함정은 측정 계기다 — 어디에 뭘 심었는지는 **함정 대장**(`tasks-large/README.md`,
`fixture/` 밖이므로 샌드박스에 복사되지 않아 모델에게 새지 않음)에 번호로 기록하고,
트랜스크립트 분석 때 발동 여부를 센다. 밀도는 절제한다 — 사람이라면 풀 수 있는
현실적 지저분함 수준.

| # | 함정 | 계열 | 노리는 실패 |
|---|---|---|---|
| 1 | 주석화된 옛 코드 덩어리(수십~수백 줄) | grep 오염 | 주석 안 심볼을 수정하거나 죽은 로직을 베낌 |
| 2 | v1/v2 공존(`calc_total`/`calc_total_v2`, 호출부 혼재) | grep 오염 | 버그는 v2에, grep은 v1을 먼저 보여줌 |
| 3 | 테스트/목에 실제 구현과 같은 이름 | grep 오염 | 목(mock)을 수정 |
| 4 | 비슷한 파일명(`report.rs`/`reporting.rs`/`report_v2.rs`, 크레이트마다 `util.rs`) | grep 오염 | 엉뚱한 파일 진입 |
| 5 | 딴 이름 같은 값 변수(`MAX_RETRY` vs `RETRY_LIMIT`, 크레이트 분산) | 부분 수정 | 한쪽만 고쳐 증상 반만 해소 |
| 6 | 복붙 함수(거의 동일 본문 2곳, 한쪽만 실호출) | 부분 수정 | 호출되지 않는 쪽 수정 |
| 7 | 매직 넘버 산포(같은 리터럴 4곳 하드코딩) | 부분 수정 | 일부만 갱신 |
| 8 | 거짓말하는 주석("10% 적용" 아래 15% 코드) | 오신뢰 | 주석을 믿고 오진 |
| 9 | 거짓 FIXME(멀쩡한 함수에 "버그 있음" 미끼) | 오신뢰 | 미끼에 시간 소모·오수정 |
| 10 | 갓파일(700줄+ `mod.rs`, 목표 함수가 하단) | 컨텍스트 압박 | read_file 페이징·중간 소실 |
| 11 | re-export 사슬(`pub use` 체인으로 정의 위치≠임포트 경로) | 컨텍스트 압박 | 정의 찾기 실패 (과제 3 연동) |

## 4. 과제 3개 — 항해 깊이 경사

| 과제 | 유형 | 측정 대상 | 판정 |
|---|---|---|---|
| 1 | 증상 기반 버그 수정 — 설명에 **파일 경로 없음** ("보고서 합계가 음수로 나온다"류) | 1-hop 항해: 증상 문자열 grep 한 번으로 도달 가능 | `cargo test` |
| 2 | 크레이트 경계를 넘는 수정 — parse→core→report로 이어지는 변경 | 2~3-hop 항해 + 다중 파일 편집 | `cargo test` (protected tests) |
| 3 | 정의 찾기/사용처 세기 (대규모판) | 편집을 배제한 순수 항해력 | 기존 answer-format 판정 사다리 재사용 (M6 §3) |

- 난이도 조준: 과제 1은 현 모델이 가끔 성공할 수준, 과제 3은 어려운 수준 — 전부 0%면
  M9 개선을 추적할 눈금이 없다 (전부 0%여도 실패 트랜스크립트 자체는 수확)
- 세 과제는 같은 베이스 픽스처를 공유하되 각자 버그/과제가 심긴 사본을 가진다.
  워킹 트리는 3벌(수 MB 텍스트)이지만 git은 내용 해시로 blob을 저장하므로 동일
  파일은 1회만 저장 — 저장소 비용은 1벌 + 차이분
- `task.toml`: `timeout_secs` 120~180(콜드 빌드가 매 런 발생), `protected`는 판정
  테스트 경로(크레이트별 tests)와 워크스페이스 매니페스트 — 구체 목록은 플랜에서
- 모든 과제에 `solution/` 오버레이 필수(--verify 해결가능성 절반), 함정 5·6·7이
  걸린 과제의 solution은 **모든 산포 지점**을 고친 완전판
- `max_turns` 25 등 하니스 설정은 기본값 그대로 첫 측정 — max_turns 도달 자체가
  실패 신호

## 6. 256K 확장 기각 근거 (계산)

ornith-1.0-9b의 명목 최대 컨텍스트는 256K이나 배포 축으로는 기각한다.

KV 캐시 = 2(K,V) × 레이어 × KV헤드 × 헤드차원 × 컨텍스트 길이 × 원소 크기.
9B급 대표 아키텍처(토큰당 56KB~336KB, fp16) 기준:

| 컨텍스트 | KV 캐시 (fp16) | Q4 가중치(~5.5GB) 포함 |
|---|---|---|
| 8K | 0.5~2.7 GB | ~6~8 GB |
| 32K | 1.8~11 GB | ~7~17 GB |
| 128K | 7~43 GB | ~13~49 GB |
| 256K | 14~86 GB | ~20~92 GB |

- 256K는 폐쇄망 동료 일반 장비(8~16GB) 밖. KV 양자화(Q8=½·Q4=¼)로도 수 GB~20GB대
- llama.cpp/LM Studio는 로드 시 설정 컨텍스트만큼 KV를 선할당 — 채우지 않아도 지불
- 프리필 O(n²): 256K 프리필은 턴당 수 분~수십 분. 특히 loco `pack()`은 히스토리를
  제자리 변형하므로 축약 발생 시 서버 프리픽스 캐시가 무효화되어 전체 재프리필 —
  컨텍스트가 클수록 이 비용이 폭발
- 유효 컨텍스트 ≪ 명목(RULER류 반복 확인): 9B급에서 256K를 채우면 중간 소실 상태로
  비용만 지불하기 쉬움

결론: 검색·항해가 뼈대, 컨텍스트 확장은 §5의 32K 민감도 실험이 준 눈금 위에서
얹는 보너스.

## 7. 레퍼런스 탐구 (B)

- 산출물: `docs/research/` 노트, 각 노트는 **"loco에의 시사점"** 섹션으로 종결
- 1차 대상:
  - **aider repo-map** — tree-sitter 태그 추출·심볼 랭킹(그래프 기반)·토큰 예산
    맞춤이 어떻게 도는지, loco(Rust·8K) 이식 시 형태 스케치
  - **codex-rs** — 컨텍스트 압축, 도구 표면(apply_patch·검색 설계), 소형/로컬 모델
    (`--oss`) 대응 흔적
- 운용: 타임박스로 시작하되, M8 실패 데이터가 특정 주제를 가리키면 그 주제는 깊이
  제한 없이 확장(grok-build 등 추가 대상 포함) — 상시 레퍼런스
- 픽스처 제작과 독립이므로 병행 가능

## 8. 리스크와 백로그

**리스크**:
- 픽스처 제작 품질 — 10~20K LOC가 컴파일되고 verify를 통과해야 함. 제작 중간마다
  `cargo test`(픽스처 로컬)와 `--verify`로 조기 검증
- 난이도 조준 실패(3과제 전부 0% 또는 전부 100%) — 경사 설계로 완화, 빗나가면 과제
  설명의 힌트 수준을 조정(판정기 자체는 불변 유지)
- eval 소요 시간 증가 — 대형 과제는 런당 시간이 길다. 3과제×3반복×2모델 + 32K 1회로
  한정, 기존 12과제 재측정 없음
