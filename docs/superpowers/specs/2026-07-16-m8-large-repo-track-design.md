## 1. 배경 — 방향 결정 경위

"쓸만하겠는데"의 판정 기준을 **폐쇄망 동료 배포**로 확정했고, 그 전제는 UX가 아니라
**모델 출력 품질**이다. 동료들의 주 작업은 **수만 라인 기존 사내 코드베이스** 수정이며
소규모 유틸은 부차적이다.

현 `tasks/` 12과제는 전부 소형 단일 크레이트라 이 장면을 전혀 측정하지 못한다.
ornith 94.4%는 "작은 코드에서의 추론 안정성" 점수이지 "큰 저장소에서 올바른 파일을
찾아 고치는 능력" 점수가 아니다. 8K 컨텍스트 소형 모델에게 대형 저장소는 항해
(navigation) 문제이고, 어디서 어떻게 무너지는지는 예측이 잘 맞지 않는 영역이다 —
따라서 M8은 **측정 먼저**다. M5~M7이 모두 eval이 방향을 정해줬기에 성공한 전례를
따른다.

레퍼런스 후보 조사 결과(2026-07-16 확인):

- **anthropics/claude-code**: 소스 비공개(이슈 트래커+문서+플러그인 저장소, CLI는
  번들 npm 배포). 코드 탐구 불가 — 행동 관찰 대상으로만 유효.
- **openai/codex**: codex-rs가 Rust 96.7%·Apache-2.0·소스 공개. 같은 언어, 같은
  OpenAI-호환 프로토콜, 로컬 모델(`--oss`) 지원 — **유일하게 유효한 동종 코드 탐구
  대상**.
- **xai-org/grok-build**: Rust 99.6%·Apache-2.0·소스 공개이나 풀스크린 TUI가
  정체성이라 loco의 REPL 노선과 결이 다름. UX 단계(후속) 참고감.
- **aider**(후보 목록 외 추가): repo map(tree-sitter 태그 추출 + 심볼 랭킹 + 토큰
  예산 맞춤)이 "작은 컨텍스트로 큰 저장소 항해"를 정면으로 푼 대표 설계 — 이번 문제에
  가장 수확이 큰 레퍼런스.

결정: **A(측정 트랙 신설) + B(레퍼런스 탐구) 병행**, 구현(C: repo-map 도구·컨텍스트
관리 개선)은 실패 데이터 확보 후 M9로. B는 컨텍스트 처리에 한정하지 않고, 실패
데이터가 가리키는 주제는 깊이 제한 없이 파는 상시 레퍼런스로 운용한다.

## 3. 픽스처 — 가상 사내 시스템 `inv-workspace`

일관된 도메인(재고/물류 관리)의 멀티 크레이트 워크스페이스를 손으로 작성해 커밋한다.

```
fixture/
  Cargo.toml          (workspace)
  .gitignore          (/target — 필수, 기존 규칙)
  inv-core/           도메인 타입·비즈니스 규칙
  inv-parse/          설정·데이터 파일 파서
  inv-store/          저장 계층 (인메모리·파일)
  inv-report/         집계·보고서 생성
  inv-cli/            바이너리 (얇은 CLI)
```

- 규모: 합계 10~20K LOC, 50~150 파일. 외부 의존성 0(워크스페이스 내부 path 의존만) —
  폐쇄망 `cargo test` 성립
- `.gitignore`는 루트 1개로 충분 — 워크스페이스 멤버는 루트 단일 `target/`을
  공유하고, 워커는 `require_git(false)`+parents 적용이라 하위 디렉토리 나열에도
  루트 .gitignore가 적용됨(리뷰 실측)
- 현실성 장치: 일부 한국어 주석, 모듈당 파일 수·길이 편차, 일부 400줄+ 파일
- **알파벳 편향 통제**: 시스템 프롬프트 트리(100항목·깊이 3)와 list_files(200항목)는
  파일명 정렬 절단이라 알파벳 뒤쪽 크레이트(inv-report·inv-store)가 체계적으로
  가려질 수 있다 — 과제 목표 파일의 노출/비노출을 우연이 아닌 **의도 변수**로
  설계하고 함정 대장에 기록
- **3벌 사본 드리프트 방지**: 베이스 픽스처 수정 시 3벌 사본+solution 재반영 절차를
  `tasks-large/README.md`에 명문화 — 사본 간 불일치는 `--verify`가 잡지 못한다
- CRLF 특수 케이스 없음 — `tasks/.gitattributes` 같은 바이트 고정 불필요, 일반 LF

### 함정 카탈로그 (11종)

함정은 측정 계기다 — 어디에 뭘 심었는지는 **함정 대장**(`tasks-large/README.md`,
`fixture/` 밖이므로 샌드박스에 복사되지 않아 모델에게 새지 않음)에 번호로 기록하고,
트랜스크립트 분석 때 발동 여부를 센다. 밀도는 절제한다 — 사람이라면 풀 수 있는
현실적 지저분함 수준.

| # | 함정 | 계열 | 노리는 실패 |
|---|---|---|---|
| 1 | 주석화된 옛 코드 덩어리(수십~수백 줄) | grep 오염 | 주석 안 심볼을 수정하거나 죽은 로직을 베낌 |
| 2 | v1/v2 공존(`calc_total`/`calc_total_v2`, 호출부 혼재) | grep 오염 | 버그는 v2에, grep은 v1을 먼저 보여줌 |
| 3 | 테스트/목에 실제 구현과 같은 이름 | grep 오염 | 목(mock)을 수정 |
| 4 | 비슷한 파일명(`report.rs`/`reporting.rs`/`report_v2.rs`, 크레이트마다 `util.rs`) | grep 오염 | 엉뚱한 파일 진입 |
| 5 | 딴 이름 같은 값 변수(`MAX_RETRY` vs `RETRY_LIMIT`, 크레이트 분산) | 부분 수정 | 한쪽만 고쳐 증상 반만 해소 |
| 6 | 복붙 함수(거의 동일 본문 2곳, 한쪽만 실호출) | 부분 수정 | 호출되지 않는 쪽 수정 |
| 7 | 매직 넘버 산포(같은 리터럴 4곳 하드코딩) | 부분 수정 | 일부만 갱신 |
| 8 | 거짓말하는 주석("10% 적용" 아래 15% 코드) | 오신뢰 | 주석을 믿고 오진 |
| 9 | 거짓 FIXME(멀쩡한 함수에 "버그 있음" 미끼) | 오신뢰 | 미끼에 시간 소모·오수정 |
| 10 | 갓파일(700줄+ `mod.rs`, 목표 함수가 하단) | 컨텍스트 압박 | read_file 페이징·중간 소실 |
| 11 | re-export 사슬(`pub use` 체인으로 정의 위치≠임포트 경로) | 컨텍스트 압박 | 정의 찾기 실패 (과제 3 연동) |

## 4. 과제 3개 — 항해 깊이 경사

| 과제 | 유형 | 측정 대상 | 판정 |
|---|---|---|---|
| 1 | 증상 기반 버그 수정 — 설명에 **파일 경로 없음** ("보고서 합계가 음수로 나온다"류) | 1-hop 항해: 증상 문자열 grep 한 번으로 도달 가능 | `cargo test` |
| 2 | 크레이트 경계를 넘는 수정 — parse→core→report로 이어지는 변경 | 2~3-hop 항해 + 다중 파일 편집 | `cargo test` (protected tests) |
| 3 | 정의 찾기/사용처 세기 (대규모판) | 편집을 배제한 순수 항해력 | 기존 answer-format 판정 사다리 재사용 (M6 §3) |

- 난이도 조준: 과제 1은 현 모델이 가끔 성공할 수준, 과제 3은 어려운 수준 — 전부 0%면
  M9 개선을 추적할 눈금이 없다 (전부 0%여도 실패 트랜스크립트 자체는 수확)
- 세 과제는 같은 베이스 픽스처를 공유하되 각자 버그/과제가 심긴 사본을 가진다.
  워킹 트리는 3벌(수 MB 텍스트)이지만 git은 내용 해시로 blob을 저장하므로 동일
  파일은 1회만 저장 — 저장소 비용은 1벌 + 차이분
- `task.toml` 타임아웃 — **필드 구분 주의(리뷰 실측)**: `timeout_secs`는 에이전트
  실행 전체(LLM 호출 포함)의 상한(기본 300)이므로 대형 항해+샌드박스 내 빌드를
  감안해 **600**으로 상향. check의 콜드 빌드는 별도 필드 `check_timeout_secs`
  (기본 120)가 담당 — **240~300**으로 명시. `--timeout-scale`은 두 값 모두에 적용
- `protected` **원칙**: 판정 테스트를 가진 모든 멤버 크레이트의 `Cargo.toml` +
  워크스페이스 매니페스트 + **존재하는** tests 디렉토리 전부 — 멤버 매니페스트의
  `autotests = false` 등으로 판정 테스트를 조용히 스킵시키는 우회를 차단한다.
  로더가 protected 경로의 fixture 내 존재를 강제하므로 tests/ 없는 크레이트는
  목록에서 제외하거나 전 크레이트에 tests/를 두는 제작 규칙을 채택(플랜에서 확정).
  중첩 경로(`inv-core/tests`)는 sync_protected·verify 겹침 검사가 지원함을 리뷰에서
  실측 확인. 구체 목록은 플랜에서
- 모든 과제에 `solution/` 오버레이 필수(--verify 해결가능성 절반), 함정 5·6·7이
  걸린 과제의 solution은 **모든 산포 지점**을 고친 완전판
- **산포 함정의 판별력 요구**: 함정 5·6·7이 걸린 과제의 protected 테스트는 산포
  지점 **각각을 독립적으로 실패**시켜야 한다(부분 수정 ⇒ check 실패). `--verify`는
  원본 FAIL/완전 솔루션 PASS 두 점만 보므로 이 판별력을 검증하지 못한다 — 제작 시
  지점별 부분 오버레이로 수동 확인하고 결과를 함정 대장에 기록
- 과제 3의 판정 사다리 재사용에는 **경로 조정 1건** 필요: 기존 판정 테스트는
  `CARGO_MANIFEST_DIR/answer.txt`를 읽는데, 워크스페이스에서는 판정 테스트가 멤버
  크레이트에 살므로 답 파일 경로를 워크스페이스 루트 기준으로 조정하거나 프롬프트에
  저장 위치를 명시(사다리 함수·단위 테스트 자체는 자기완결이라 그대로 이식)
- `max_turns` 25 등 나머지 하니스 설정은 기본값 그대로 첫 측정 — max_turns 도달
  자체가 실패 신호. 예외 2건(`max_output_tokens`·`command_timeout_secs`)은 §5의
  측정 설정 고정 참조

## 5. 측정 프로토콜과 실패 분류

- 기존 프로토콜 준수(docs/baselines.md): 측정 중 빌드 금지, 모델 교체 시 언로드,
  LM Studio 로드 컨텍스트 ≥ context_tokens
- **측정 설정 고정(두 배치 공통, 리뷰 반영)**: 로컬 `./.loco/config.toml`에
  `max_output_tokens = 4096`(v2 프로토콜 유지 — 코드 기본값 2048 아님) +
  **`command_timeout_secs = 240` 상향**. 근거: 기본 60초는 샌드박스 내 콜드 빌드와
  충돌해 모델의 `cargo test` 검증이 전부 타임아웃으로 오염되고, 타임아웃 결과도
  run_command Ok라 VERIFY_NUDGE가 해제되어 엄격/거짓-finish 해석까지 오염된다.
  `--timeout-scale`은 이 값에 적용되지 않으므로 config로만 조정 가능. 두 값 모두
  report.json `effective_config`가 자증 — baselines.md M8 측정 조건에 명기
- **8K 베이스라인**: gemma + ornith, `--repeats 3`, 시드 고정, `context_tokens`
  8192. **LM Studio 로드 컨텍스트는 12288**(여유분): 토큰 추정이 bytes/4라 밀집
  Rust 코드·한국어 주석(3바이트/자)에서 과소추정되어 서버측 400 위험이 커지는데,
  로드 ctx 여유가 이를 흡수한다("로드 ≥ context_tokens" 프로토콜 준수. 로드 ctx는
  effective_config에 안 남으므로 baselines.md에 수기 기록. tasks-large는 신규
  기준선이라 v2와의 로드값 일치 의무 없음 — 배치 내 일관성만 유지)
- **32K 민감도**: ornith만, 8K와 동일하게 `--repeats 3` 한 배치(비교 가능성 유지) —
  `context_tokens` 32768 + **로드 컨텍스트 최소 40960, 권장 49152(컨텍스트 비례 ×1.5)**.
  여유분은 절대치가 아니라 예산에 비례해야 한다(리뷰 산술: 로드 36864면 과소추정
  한계비가 8K의 2.2~3.3에서 1.27~1.43으로 떨어져, 밀집 Rust r≈1.2·한국어 혼입
  1.3~1.4와 겹침 — 특히 파싱 재시도 호출은 축약-재시도 미포장이라 마진 소진이 곧
  배치 재실행). **순서: 아래 ornith 실측 사양표를 먼저 측정해 KV 비용을 확인한 뒤
  32K 로드값을 확정**한다. 통과율·함정 발동·s/런 변화로 "검색 강화 vs 컨텍스트
  확장"의 교환비를 데이터화
- **ornith 실측 사양표**: 8K/16K/32K 각 로드 시 메모리 점유(`api/v0/models` + 시스템
  모니터)와 프리필 속도 기록 → 배포 요구사양 표의 근거 + ornith의 실제 토큰당 KV
  비용 확정. 프리필 속도 측정 방법: **고정 길이 프롬프트(6K 토큰급)의 첫 토큰
  지연을 tok/s로 환산**, 컨텍스트 설정별 동일 프롬프트 반복. 배포 요구사양 표로
  문서화할 때 "실운용 로드는 여유분 포함(예: 32K 운용 = 로드 40960~49152)" 각주를
  달아 요구 메모리 과소 기재를 방지
- **실패 분류 스키마** (런별): outcome은 report.json의 5종 그대로(finished/
  max_turns/반복정지/parse_failed/타임아웃) + **검증 타임아웃 열**(트랜스크립트에서
  "command timed out" 계수 — `command_timeout_secs` 상향 후 잔존 여부 확인) +
  발동 함정 번호(트랜스크립트 판독) + 도구 사용 패턴(첫 정답 파일 도달 턴 수 —
  과제별 "정답 파일" 집합은 함정 대장에 사전 정의, grep/list_files/read_file 비율)
- **컨텍스트 오버플로는 outcome이 아니라 하네스 중단 이벤트(리뷰 실측)**: 오버플로
  400은 실행당 2회 축약-재시도 후 LlmError로 전파되어 **하네스 전체 중단 +
  report.json 미생성**이 된다(부분 리포트는 Ctrl+C 경로뿐). 발생 시 stderr와
  `run-*.jsonl` 트랜스크립트로 판독해 실패 분석에 별도 행으로 기록하고, **재시작
  프로토콜 = 해당 배치 전체 재실행**(배치 내 설정 일관성 유지). 위의 로드 ctx
  여유분이 1차 완화책
- **관측 한계(리뷰 실측)**: 회복된 오버플로(축약-재시도 성공)와 pack 축약 빈도는
  현 하니스에서 수집 불가 — eval은 Notice 이벤트를 버리고, pack의 제자리 변형은
  트랜스크립트에 흔적을 남기지 않는다. 따라서 "컨텍스트 압박" 신호를 32K 민감도의
  해석 축으로 쓰지 않는다. 계측 추가는 M9 요구사항 후보(§8 오버플로 내성 ④)
- 기록 위치: 수치는 `docs/baselines.md` M8 섹션, 실패 분류·M9 요구사항 도출은
  `docs/research/`의 M8 실패 분석 노트(파일명에 작성일 부여). baselines.md의 grep
  레시피는 outcome 직렬화 실명(`finished`/`max_turns`/`repetition_stop`/
  `parse_failed`/`timeout`)을 사용

## 6. 256K 확장 기각 근거 (계산)

ornith-1.0-9b의 명목 최대 컨텍스트는 256K이나 배포 축으로는 기각한다.

KV 캐시 = 2(K,V) × 레이어 × KV헤드 × 헤드차원 × 컨텍스트 길이 × 원소 크기.
9B급 대표 아키텍처(토큰당 56KB~336KB, fp16) 기준:

| 컨텍스트 | KV 캐시 (fp16) | Q4 가중치(~5.5GB) 포함 |
|---|---|---|
| 8K | 0.5~2.7 GB | ~6~8 GB |
| 32K | 1.8~11 GB | ~7~17 GB |
| 128K | 7~42 GB | ~13~48 GB |
| 256K | 14~84 GB | ~20~90 GB |

- 256K는 폐쇄망 동료 일반 장비(8~16GB) 밖. KV 양자화(Q8=½·Q4=¼)로도 수 GB~20GB대
- llama.cpp/LM Studio는 로드 시 설정 컨텍스트만큼 KV를 선할당 — 채우지 않아도 지불
- 프리필 O(n²): 256K 프리필은 턴당 수 분~수십 분. 특히 loco `pack()`은 히스토리를
  제자리 변형하므로 축약 발생 시 서버 프리픽스 캐시가 무효화되어 전체 재프리필 —
  컨텍스트가 클수록 이 비용이 폭발
- 유효 컨텍스트 ≪ 명목(RULER류 반복 확인): 9B급에서 256K를 채우면 중간 소실 상태로
  비용만 지불하기 쉬움

결론: 검색·항해가 뼈대, 컨텍스트 확장은 §5의 32K 민감도 실험이 준 눈금 위에서
얹는 보너스.

## 8. 리스크와 백로그

**리스크**:
- 픽스처 제작 품질 — 10~20K LOC가 컴파일되고 verify를 통과해야 함. 제작 중간마다
  `cargo test`(픽스처 로컬)와 `--verify`로 조기 검증
- 난이도 조준 실패(3과제 전부 0% 또는 전부 100%) — 경사 설계로 완화, 빗나가면 과제
  설명의 힌트 수준을 조정(판정기 자체는 불변 유지)
- eval 소요 시간 증가 — 대형 과제는 런당 시간이 길다. 3과제×3반복×2모델 + 32K
  배치 1회(ornith, repeats 3)로 한정, 기존 12과제 재측정 없음

**백로그 (M8에서 하지 않음)**:
- repo-map 도구·검색 강화·컨텍스트 관리 개선 — M9, 실패 데이터가 정한다
- **컨텍스트 오버플로 내성(M9 후보, 리뷰 실측으로 명명)**: ① 토큰 추정 보정 —
  bytes/4는 밀집 코드·한국어에서 체계적 과소추정(보수 계수 또는 실측 캘리브레이션)
  ② 파싱 재시도 경로의 축약-재시도 포장(M3 알려진 갭 — 현재 이 경로의 오버플로는
  곧장 하네스 중단) ③ eval LLM 에러 시 부분 리포트 저장(현재 부분 리포트는 Ctrl+C
  경로뿐 — 배치 결과 증발 방지) ④ 회복된 오버플로·pack 축약 빈도 계측(§5 관측
  한계 해소). M8 측정에서 오버플로 중단이 실제 발생하면 그 데이터가 우선순위 승격
  근거

- CARGO_HOME 격리·cargo 해석 고정 — M7 백로그 그대로 유지
- UX·온보딩(동료 배포의 2단계) — 품질 입증 후
