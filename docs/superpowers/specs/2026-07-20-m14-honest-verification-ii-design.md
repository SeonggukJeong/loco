## 1. 배경 — 왜 이 축인가

M13은 북극성을 합성 평가에서 실사용으로 옮겼고, 그 첫 관측이 헤드라인을
뒤집었다: **합성 통과율이 실사용을 예측하지 못한다.** 동시에 파일럿은 loco
하네스 자신의 결함을 여러 형태로 드러냈다.

M12는 "테스트가 통과했는가"를 exit code 대신 파싱된 libtest 결과에 접지했다.
파일럿이 보여준 것은 그 위층이 비어 있다는 사실이다 — **"맞는 것을 테스트했는가"는
아무것도 접지하지 않는다.** `docs/m14-candidates.md` §1-3이 다섯 형태를 센다.

| 형태 | 파일럿 관측 | 기전 |
|---|---|---|
| 파이프가 exit 0 날조 | 5건 | `has_unquoted_pipe`가 노트 채널에만 연결 |
| 컴파일 실패인데 검증으로 접수 | 3건 | **해제 술어가 종료 코드를 안 본다** — §3-3-2 |
| `tail`이 실패 섹션 절단 | 1건 | 통과 섹션만 남아 규칙 2·4를 모두 통과 |
| `cargo test --lib`을 bin 전용 크레이트에 | 4건 | 실행 0건 |
| 크레이트 스코프 오지정 | 2건 | `ran==0 ∧ filtered_out==0` — M12 공허 런 가드가 침묵 |

**계수 주의(1R I4)**: 위 숫자는 **다중 라벨 기준 건수(합 15)이지 세션 수가
아니다.** `docs/baselines.md` M13이 같은 이유로 *"다중 라벨이라 합(42) ≠ 세션
수(20)"*를 못박고 있고, 이 스펙 자신의 §3-3도 #2가 #1과 겹친다는 것을 인정한다
(파이프 기인분). **"15세션에서 관측됐다"고 쓰면 안 된다.**

초판은 #2의 기전을 *"`test result:` 줄 부재 → `parse_test_summary`가 `None`"*으로
적었는데 **틀렸다**(1R I5). 컴파일이 실패한 `cargo test`는 exit 101이지 0이
아니다. 실제 기전은 §3-3-2에 적는다.

이 축을 고르는 근거는 세 가지다. ① 검증 날조는 봉투 안/밖과 무관하게 발생한다 —
과제문이 파일 경로를 줘도 모델은 exit 0을 날조한다. ② M12 스코핑에서 사용자가
정한 원칙("잘 못하는 걸 잘하게 만들려고 시간 쓰지 말고, 하네스 자신의 거짓
초록불을 고쳐라")의 직접적 연속이다. ③ 파일럿의 성공 판정 4건이 실행·대조하면
뒤집혔다는 관측(`docs/m14-candidates.md` §4)과 같은 뿌리다.

### 1-1. 이 마일스톤이 무엇을 주장하지 않는가

**효과는 `tasks/`로 실증되지 않는다.** 검증 날조는 파일럿 실레포에서 관측됐지만
합성 세트에서는 거의 발동하지 않는다. M12가 같은 상황을 명시적으로 받아들여
회귀 게이트로 축소했고, M14도 같다. 측정은 **무회귀 확인 + 장치 발동 여부
관측**이며, **효과 확인은 다음 파일럿의 몫이다.**

**게이트의 검출력도 낮다**(1R M2, §8-1). 36런 이항으로 통과율 97.2% → 87%의
회귀는 재측정을 포함해도 **약 절반이 그냥 통과한다.** 따라서 §7의 "무회귀"는
*"통과율 80% 미만의 붕괴는 배제한다"* 이상을 뜻하지 않는다. 게이트 통과로
"검증 날조가 줄었다"를 주장하면 안 되는 것은 물론이고, **"회귀가 없다"조차
그 해상도 안에서만 참이다.**

**그리고 이 마일스톤의 두 Critical(2R)은 게이트가 아니라 단위 테스트에서
잡혔다** — §3-3-3의 풍선효과는 배치를 돌리기 전에 이미 결정론적으로 성립했고,
§4-2-1의 잔여물은 통과율에 나타나지 않는다. **§7 수용 기준이 게이트보다
중요하다.**

### 3-2. 왜 "긍정 증거 요구"로 전면 뒤집지 않는가

가장 단순해 보이는 설계는 검증 성립 술어를 뒤집는 것이다 — `parse_test_summary`가
`ran > 0 ∧ failed == 0`을 실제로 확인했을 때만 검증으로 친다. 이러면 다섯 형태
중 넷이 한 번에 덮인다.

**이 안을 기각한다. `parse_test_summary`는 libtest 전용이기 때문이다.**

loco는 Rust 전용 도구가 아니다. `run_command`는 `make`·`npm test`·`pytest`
무엇이든 실행하고, 그 출력에는 `test result:` 줄이 없다. 긍정 증거를 요구하면
비-Rust 레포에서 **검증이 영영 성립하지 않고** VERIFY_NUDGE가 매 런 발동한다.
M12가 `None`을 보수적 폴백으로 둔 이유가 정확히 이것이며, 그 판단은 옳았다.

따라서 거부 술어에 넣을 수 있는 것은 **언어 무관하게 판별 가능한 사실**뿐이다.
파이프가 그 조건을 만족한다. 종료 코드도 언어 무관하지만, 그것을 거부 조건으로
쓰는 것은 "실패한 검증도 검증"이라는 기존 거래를 뒤집는 일이라 §3-3-2에서
명시적으로 범위 밖에 둔다.

### 3-3. A-1 · 파이프 가드를 **해제 술어에만**

`has_unquoted_pipe`를 `pub(crate)`로 승격하고, `agent/mod.rs`의
`cmd_exit`/`cmd_summary`/`empty_verify` 계산부에서 **하나에만** 가드를 건다.

> **가드 대상은 `mutated_since_verify = false`(VERIFY_NUDGE 해제) 하나다.**
> `finish_nudge::TurnEvent::VerifyOk` 매핑은 **바꾸지 않는다.**

즉 **파이프가 있는 `run_command` Ok는 "검증이 성립했다"로 인정하지 않되,
재검증 루프 감지에서는 그대로 센다.**

이 수선의 논거는 새 정책 도입이 아니라 **자기모순 해소**다. M11이 이미 파이프
명령의 Done 결과에 "이 exit code는 파이프라인 마지막 명령의 것"이라는 출처
안내를 붙이고 있다. 그런데 정작 하네스 자신은 같은 턴에 그 exit 0을 검증으로
세고 있었다. 모델에게 믿지 말라고 알린 신호를 하네스가 믿고 있었던 것이다.

**덮는 범위**: #1(5건) · #3(`tail` 절단, 1건) · #2 중 파이프 기인분.

**구현 제약(2R I4)**: **`cmd_exit`/`cmd_summary`를 `None`으로 만드는 방식으로
가드를 구현하지 말 것.** 가장 짧은 구현은 파이프면 `cmd_exit = None`으로 놓는
것이지만(그러면 `VerifyOk` 조건이 자동으로 깨진다), `cmd_exit`는 같은 지점에서
`status.record_command_result(...)`로도 넘어가고(`agent/mod.rs:472`) `None`을
받으면 `last_test_summary`까지 함께 `None`으로 덮인다(`status_note.rs:74~78`,
의도된 동작). **그러면 A-2가 렌더할 exit code 자체가 사라진다.** 별도 `is_piped`
값을 도입하고 각 소비 지점이 스스로 판단할 것.

#### 3-3-1. VERIFY_NUDGE 문구를 반드시 함께 고친다 (1R C3)

**이것은 A-1의 선택 사항이 아니라 구성 요소다.** 현행 문구는
(`agent/mod.rs:70`):

> `"You modified files but never ran a verification command. Run the project's tests (e.g. cargo test) with run_command, then finish."`

파이프 가드가 발동하면 이 문구는 **파이프로 `cargo test`를 방금 돌린 모델에게
"never ran a verification command"라고 말한다.** 거짓이고, 파이프를 지목하지도
않는다. "정직한 검증 신호"라는 이름의 마일스톤이 하네스에 새 거짓말을 심는
셈이다.

예상되는 모델 반응도 자명하다 — "테스트를 돌리라"는 지시를 받았으니 **같은
파이프 명령을 다시 돌리고**, `RepetitionTracker`의 동일 호출 창에 들어가 반복
교정/정지로 간다. **§9 R1이 걱정하는 풍선효과의 가장 그럴듯한 구체적 경로가
이것이며, 문구를 고치지 않으면 R1은 관측 대상이 아니라 예정된 결과다.**

따라서 파이프로 거부된 경우 **별도 문구**를 낸다. 대의는 *"검증 명령을 돌리긴
했지만 셸 파이프라인이라 그 종료 코드가 테스트 통과 여부를 말해주지 않는다.
파이프 없이 다시 돌린 뒤 finish하라"*이며, 정확한 영문은 §11 열린 질문 1.

**플래그의 정의를 정확히 한다(2R I2).** "마지막 `run_command`가 파이프였는가"로
정의하면 하네스가 **다시 틀린 이유를 말한다**:

```
edit_file                        → mutated_since_verify = true
run_command "cargo test | tail"  → 파이프 → 해제 안 됨
edit_file                        → mutated_since_verify = true (그대로)
finish                           → nudge 발동 — 그런데 진짜 이유는 "그 뒤에 또 고쳐서"다
```

**따라서 플래그는 "현재 미해제 상태를 만든 원인이 파이프인가"로 정의한다.**

개정 2는 이를 *"파이프 때문에 건너뛸 때**만** `true`"* + *"뮤테이션이 `false`로
리셋"*으로 적었는데, **`true`로 세우는 조건과 `false`로 되돌리는 조건이 비대칭이라
공허 런 경로에서 부패한다**(3R I1, 실측). 공허 런은 해제를 건너뛰지만 파이프도
뮤테이션도 아니라 **어느 규칙도 타지 않아 직전 파이프의 `true`가 그대로 남고**,
모델은 파이프가 원인이 아닌데 파이프 문구를 받는다.

개정 3은 이를 `else` 분기 안의 덮어쓰기로 적었는데, **해제에 성공하면 플래그를
건드리지 않아 낡은 `true`가 남는다**(4R C2, 계측으로 재현). VERIFY_NUDGE에는
무해하다 — 해제 후엔 `mutated_since_verify == false`라 nudge 자체가 안 뜬다.
**그러나 §3-3-3-1이 도입한 FINISH_NUDGE 파이프 변형은 `mutated_since_verify`와
아무 결합이 없어 낡은 값을 그대로 읽는다.** 그 결과 마지막 검증이 파이프가
아니었고 해제까지 됐는데 "네 마지막 검증이 셸 파이프라인이었다"고 말한다.

**정정된 정의 — 매 `run_command` 디스패치마다 무조건 대입한다:**

```rust
let released = !empty_verify && !is_piped;
if released { mutated_since_verify = false; }
unreleased_due_to_pipe = !released && is_piped;   // else 안이 아니라 밖
```

- 뮤테이션이 `mutated_since_verify = true`로 놓을 때 **`false`로 리셋**한다(불변)
- 갱신 지점은 해제 로직과 같은 **`if dispatch_ok` 블록 안**이다 — 실패한
  디스패치(`run_command` 툴 에러)가 플래그를 오염시키지 않게

**이 플래그는 소비자가 둘이고 결합 조건이 다르다.** VERIFY_NUDGE는
`mutated_since_verify`와 묶여 있어 낡은 값을 못 읽지만, FINISH_NUDGE는 묶여
있지 않다. **따라서 해제 성공 시에도 반드시 갱신되어야 한다** — 이것이 조건부
건너뛰기를 금지하는 이유다. 수명은 현행 `verify_nudged` 래치와 같은 런 스코프다.

**우선순위(4R M5)**: 파이프이면서 공허 런이면 플래그는 `true`(파이프 문구)가
된다. **두 원인이 모두 참**이므로 어느 쪽도 거짓이 아니고, 파이프가 먼저 조치
가능하다 — 파이프를 빼면 온전한 출력이 보이고 그제야 공허 런이 드러난다.
현 동작을 유지한다.

**실측된 경로** — 3R·4R 리뷰가 사양대로 구현해 확인했다.

| 순서 | 개정 2 | 개정 3 | 정정 후 |
|---|---|---|---|
| 파이프 → 정상 검증 → finish | 안전 (nudge 미발동) | 동일 | 동일 |
| 파이프 → 정상 검증 → **read ×4** → FINISH_NUDGE | — | **파이프 변형 (틀림)** | 기본 변형 ✓ |
| 파이프 → **비파이프** 타임아웃 → finish | 안전 (nudge 미발동) | 동일 | 동일 ✓ |
| 파이프 → **파이프** 타임아웃 → finish | 미해제 + 파이프 문구 | 동일 | 동일 — §3-3-2 참조 |
| 파이프 → **공허 런** → finish | 파이프 문구 (틀림) | 기존 문구 ✓ | 동일 ✓ |

#### 3-3-2. A-1이 바꾸지 않는 것 — 해제 술어는 종료 코드를 안 본다 (1R I5)

`agent/mod.rs:466~471`은 `run_command`가 Ok이기만 하면(공허 런 제외)
`mutated_since_verify = false`로 놓는다. **종료 코드를 전혀 보지 않으며, 이는
의도된 동작이다** — 코드 주석이 *"해제 조건이었던 'Ok이면 종료코드 무관'에서
공허 런만 제외"*라고 명시하고, `agent/mod.rs:1850`의 테스트가 타임아웃 경로에
대해 이를 고정한다.

따라서:

> **A-1은 여기에 파이프 조건만 더한다. 비-파이프 실패 검증(exit 101 등)이
> 여전히 VERIFY_NUDGE를 해제한다는 사실은 바꾸지 않는다.**

#2("컴파일 실패인데 검증으로 접수", 3건) 중 **파이프가 아닌 분량은 A-1로
덮이지 않는다.** A-2의 상태선이 그 사실을 알리지만 상태선은 알림 채널이라
finish를 막지 않는다. 이것을 바꾸는 것은 §3-2가 기각한 "긍정 증거 요구"에
해당하므로 §2-1에 범위 밖으로 명시했다. **M14 종료 후 "#2를 덮었다"고 쓰지 말 것.**

**A-1이 바꾸는 타임아웃 동작(4R I2)**: `is_piped`는 `dispatch_ok ∧ 명령에 파이프`이고
타임아웃은 `Ok(본문)`을 반환하므로 `dispatch_ok == true`다. 따라서 **타임아웃한
파이프 명령은 해제되지 않는다** — M14 이전에는 해제했다. 동작 자체는 방어
가능하지만(타임아웃한 파이프는 검증 여부를 알 수 없다) **기존 회귀 테스트
(`agent/mod.rs:1840`)가 파이프 없는 명령을 쓰므로 이 변경은 스위트에 안 걸린다.**
문구는 파이프 지목으로 두어도 무해하다(파이프를 빼면 출력이 온전히 보인다) —
세 번째 변형을 추가하지 않는다.

**기록해 두는 기존 잔여(3R I1 부기)**: §3-3-1의 플래그 정정 후 공허 런 경로는
기존 문구(*"You modified files but never ran a verification command"*)를 받는데,
모델은 `cargo test --lib`을 **실제로 돌렸으므로** 이것도 참이 아니다. **이는 M12의
공허 런 거래에 이미 있던 잔여이며 A-1이 만든 것이 아니다.** A-2의 상태선 규칙 3이
알림으로 덮는다. 이번 범위에 넣지 않되, **다음 라운드가 신규 결함으로 재발견하지
않도록** 여기 기록한다.

#### 3-3-3. `VerifyOk` 매핑은 바꾸지 않는다 — 2R이 실측으로 기각한 대안 (2R C1)

**개정 1은 가드를 해제와 무장 "양쪽"에 걸고 파이프 턴을 `TurnEvent::Other`로
떨어뜨리라고 지시했다. 그것이 FINISH_NUDGE를 파이프 루프에서 완전히
무력화한다.** 2R 리뷰가 스펙대로 구현해 에이전트 레벨에서 재현했다.

기전은 `finish_nudge.rs:52~62`다. `TurnEvent::Other => {}`는 **아무 상태도 바꾸지
않고**, `VerifyOk`는 `if armed { count(repeat) } else if mutated { arm }`이다.
즉 파이프를 `Other`로 보내면 **무장도 안 되고 idle 창 카운트도 안 올라간다** —
파이프만 쓰는 재검증 루프(`finish_nudge` 모듈 주석이 명시한 표적 패턴)가
**개입 사정권 밖으로 나간다.** `VerifyOther`(disarm)도 같은 문제를 갖는다.

실측(`write_file` → 파이프 명령 A/B 교대 5회 → `finish`; A/B 교대는
`RepetitionStop` 선점 회피용):

| 구성 | outcome | LLM 호출 | FINISH_NUDGE | VERIFY_NUDGE |
|---|---|---|---|---|
| 현행 (가드 없음) | `Finished("done")` | 7 | **true** | false |
| 개정 1 (양쪽 가드 + `Other`) | `Finished("done2")` | 8 | **false** | **true** |
| **개정 2 (해제만 가드, `VerifyOk` 유지)** | `Finished("done2")` | **8** | **true** | **true** |

**개정 2가 공짜가 아니다(3R M2)**: 턴 비용은 개정 1과 **동일하게 +1**이다
(VERIFY_NUDGE 1회 반려의 대가 — §3-3-4 논거 ①이 이미 수용한 것). 개정 2가 되살린
것은 **FINISH_NUDGE이지 턴 비용이 아니다.** 초판의 빈칸이 "개정 2가 현행으로
되돌린다"는 오독을 허용했으므로 채워 둔다.

**개정 1의 구성은 FINISH_NUDGE를 죽이고 VERIFY_NUDGE로 실패를 전이시켰다 —
§9 R1이 "관측하겠다"고 한 풍선효과가 배치 전에 이미 결정론적으로 성립했다.**

**진단**: 두 장치를 묶을 이유가 애초에 없었다. 목적이 다르기 때문이다.

| 장치 | 묻는 것 | 파이프에 대한 답 |
|---|---|---|
| `mutated_since_verify` 해제 | "검증이 성립했다고 인정할 것인가" | **인정하면 안 된다** → 가드 |
| `VerifyOk` 무장/카운트 | "재검증 루프를 감지할 것인가" | **감지해야 한다** → 불변 |

파이프는 "검증이 아니다"가 아니라 **"검증 여부를 알 수 없다"**이며(§3-3이 그렇게
쓴다), **알 수 없다는 것이 루프 감지를 끌 이유는 되지 않는다.**

**대안(무장 차단)은 루프 감지 자체를 잃는 것으로 실측에서 더 나빴다.** 그러나
`VerifyOk` 유지에는 대가가 있고, 그 대가는 상태가 아니라 **텍스트**에 있다.

#### 3-3-3-1. FINISH_NUDGE 문구도 함께 고쳐야 한다 (3R C1)

**개정 2는 이 긴장을 "궤적은 정합한다"로 닫았는데, 상태기계로만 검토한 결과였다.
모델이 소비하는 것은 상태가 아니라 텍스트이고, 텍스트는 정면으로 모순된다.**
3R 리뷰가 개정 2 사양대로 구현해 모델이 받는 문장을 순서대로 뽑았다:

```
>> FINISH_NUDGE: You already ran a successful verification. If the task is
   complete, call finish with a summary now; do not re-verify what you have
   already confirmed.
>> VERIFY_NUDGE(pipe): You ran a verification command, but it was a shell
   pipeline, so its exit code does not tell whether the tests passed.
   Re-run it without a pipe, then finish.
```

**하네스가 "검증에 성공했으니 재검증하지 마라"고 지시한 뒤, 그 지시를 따른
finish를 거부하며 "재검증하라"고 말한다 — 그것도 같은 `run_command` 이벤트에
대해서다.** 한쪽은 그 파이프 실행을 `successful verification`으로 부르고 다른
쪽은 "테스트 통과 여부를 말해주지 않는다"고 부른다.

**§3-3-1이 거짓말 하나를 고치는 마일스톤에서, 개정 2가 다른 채널에 새 거짓말을
만들었다.** §3-3-1이 자기모순으로 규정하고 고친 것("모델에게 믿지 말라고 알린
신호를 하네스가 믿고 있었다")과 같은 종류다 — 이번엔 **하네스의 두 입이 같은
신호를 반대로 부른다.**

이 조합은 **개정 2에서만 성립한다**: 개정 1은 파이프가 `Other`라 FINISH_NUDGE가
발동하지 않았고, 개정 전은 파이프가 해제해서 VERIFY_NUDGE가 발동하지 않았다.
(M14 이전에도 둘이 한 런에 발동하는 경로는 있으나 **서로 다른 이벤트**를 가리키므로
모순이 아니다. 새로 생긴 것은 *동일 이벤트에 대한 동시 상반 서술*이다.)

**수선 — 검토한 세 갈래 중 (c)를 채택한다.**

| 갈래 | 판정 |
|---|---|
| (a) FINISH_NUDGE에서 `successful`을 뺀다 | 싸지만 정상 경로(파이프 없는 재검증 루프)의 신호가 약해진다 |
| (b) 파이프면 FINISH_NUDGE를 억제한다 | **채택 불가** — 2R C1이 실측으로 기각한 방향(루프 감지 상실) |
| **(c) FINISH_NUDGE에 파이프 변형 문구를 둔다** | **채택.** 루프 감지와 정직성을 둘 다 유지하고, 모델은 상반된 두 지시 대신 **하나의 실행 가능한 지시**를 받는다 |

대의는 *"여러 번 재검증했다. 다만 마지막 검증이 셸 파이프라인이라 테스트 통과를
입증하지 못했으니, 파이프 없이 한 번 돌린 뒤 finish하라"*이며 정확한 영문은
§11 Q1. **이로써 모델 대면 파이프 텍스트가 3종에서 4종이 된다** — Q1의 상호
비중복 대상에 FINISH_NUDGE를 포함시켰다.

#### 3-3-4. 정당한 파이프 검증의 처리

`cargo test 2>&1 | tail -50`은 실제 검증 의도이지만 이 가드에 걸린다. 설계상
수용하며 근거는 셋이다.

1. **`VERIFY_NUDGE`는 런당 1회 래치**라 finish를 영구 차단하지 않는다.
   1R 리뷰가 코드로 확인했다 — `agent/mod.rs:200` `verify_nudged = false` →
   `:304` `if mutated_since_verify && !verify_nudged`, 리셋 경로 없음
2. **거부 시 파이프를 지목하는 이유가 전달된다** — 단 §3-3-1을 구현한 경우에만.
   초판은 현행 문구가 이유를 전달한다고 썼는데 **틀렸다**(1R C3)
3. **파이프로 절단된 출력은 실제로 하네스가 검증 여부를 알 수 없는 출력이다** —
   가드가 참을 말하고 있다

**그럼에도 풍선효과는 실재하는 위험이다.** §8-3 관측 항목과 §9 R1에 고정한다.

### 3-4. A-2 · 알림 전면

`StatusNote::record_command_result`가 명령 문자열을 받도록 시그니처를 넓히고,
상태선의 verification 줄이 다섯 형태를 **정직하게** 렌더한다.

원칙: **언어 무관한 사실만 적는다.** libtest 파싱이 `None`이어도 파이프 여부와
"요약이 없다"는 사실은 알 수 있다.

#### 3-4-1. 줄 수 제약 — 반드시 한 줄 안에 (1R C1)

**초판이 "사실 한 줄을 더한다"고 쓴 것은 M11 상태선 블록 계약을 깨는 지시였다.**
1R 리뷰가 실행으로 재현했고 컨트롤러가 코드로 독립 확인했다.

`verification_line()`은 **한 줄을 반환하는 계약**이고 두 렌더 분기가 그 전제
위에 서 있다(`status_note.rs:99~121`):

- 무뮤테이션 분기 — 전체를 **단일 줄**로 조인: `"{MARKER}files edited: none yet | {verification} | {turns_line}"`
- 뮤테이션 분기 — `"...\n{CONT_INDENT}{verification}\n{CONT_INDENT}{turns_line}"`.
  `verification` 안에 `\n`이 들어가면 **그 둘째 줄에는 `CONT_INDENT`가 안 붙는다**

블록 계약("마커 줄 + 9칸 들여쓴 연속 줄")을 벗어난 줄은 `session.rs:156`
`remove_status_note()`가 걷어내지 못한다. 결과는 **영구 잔존 + 케이던스
(5/10/15/20)·pacing마다 누적**이다. 리뷰의 프로브 실행에서는 `turns:` 줄까지
고아 줄에 딸려 나가 제거 후에도 남았다.

이것이 왜 Critical인가: **누적되는 고아 줄이 컨텍스트를 조용히 먹는데, 그것이
바로 같은 마일스톤의 B-2가 고치려는 문제다. A-2가 B-2를 상쇄한다.** 게다가
게이트 배치는 이 회귀를 잡지 못한다 — 통과율에 즉시 안 나타나고,
`exp_metrics.py`의 status 마커 카운트는 마커 줄만 세므로 고아 줄은 계측되지 않는다.
그리고 **현행 스위트 370건 중 이 불변식을 지키는 테스트가 0개다**(리뷰가 A-2
형태의 둘째 줄을 넣고 전체 스위트를 돌려 `370 passed` 확인).

**따라서 설계 제약은 다음과 같다.**

- 추가 사실은 **규칙 5의 한 줄 안에** 넣는다. 예:
  `verification: last command exited 1 (no test summary in output)`
- 여러 줄이 불가피하면 `verification_line()`이 `Vec<String>`을 반환하도록 바꾸고,
  두 렌더 분기가 각각 ` | ` 조인 / `\n{CONT_INDENT}` 조인을 **책임진다**
- **§7에 불변식 테스트를 수용 기준으로 올린다**(수용 기준 1). 형태는 §11 Q6

#### 3-4-2. 무엇을 알릴 것인가

- **파이프 — 한정자 규칙을 규칙별로 분리한다**(4R C1). 개정 2·3은 규칙 4·5에
  같은 처방(짧은 한정자)을 걸었는데, **두 규칙의 렌더 근거가 다르다.**

  | 규칙 | 렌더 근거 | 파이프일 때 |
  |---|---|---|
  | **규칙 5** (`last command exited N`) | exit code | **짧은 한정자로 충분** — 사실이 정확히 "exit code 출처"이고 M11 노트와의 중복도 여기서만 실재한다. 예: `verification: last command exited 0 (via pipe)` |
  | **규칙 4** (`all N passed`) | **파싱된 요약** | **한정자로 불충분 — 규칙 5로 폴백한다** |

  **규칙 4가 왜 다른가**: 규칙 4는 `failed == 0 ∧ ran > 0 ∧ last_cmd_exit == "0"`로
  "전부 통과"를 주장한다. `tail`이 실패 섹션을 잘라낸 본문(#3)에서는 통과 섹션만
  파싱되므로 **그 주장 자체가 거짓**이고, `(via pipe)`는 exit code 출처에 대한
  한정자라 거짓을 건드리지 않는다. 4R 리뷰가 재현했다 — 절단 본문이
  `TestSummary { ran: 8, passed: 8, failed: 0 }`으로 파싱되어
  `verification: last cargo test: all 8 passed`가 렌더된다.

  **더 나쁜 점**: 규칙 4의 exit 교차검증은 M12가 *"절단된 exit-101 본문의 생존
  통과 섹션이 깨끗한 통과로 오독되는 것을 막으려고"* 넣은 장치인데,
  **파이프에서는 exit code가 `tail`의 0이라 그 교차검증이 구조적으로 무조건
  통과한다.** #3이 M14의 실패 목록에 오른 이유가 정확히 이것이다.

  **따라서 폴백은 새 정책이 아니라 M12 결정의 자연스러운 확장이다** —
  "교차검증 실패 시 규칙 5로 폴백"이라는 기존 규칙에, 파이프가 그 교차검증을
  무효화하는 조건이라는 사실을 더한 것이다. 결과적으로 상태선은
  `last command exited 0 (via pipe)`가 되어 §3-3-1의 파이프 VERIFY_NUDGE와
  정합하고, `(via pipe)` 한정자는 원래 의도대로 규칙 5에서만 쓰인다.
- **요약 부재** — `parse_test_summary`가 `None`이면 현행 규칙 5가 "last command
  exited N"만 렌더한다. 요약을 찾지 못했다는 사실을 같은 줄에 더한다.
  #2(컴파일 실패)와 #4(`--lib`을 bin 전용 크레이트에)가 여기 해당.

  **조건은 `last_test_summary.is_none()`이며 "규칙 5에 도달했다"가 아니다**(5R I2).
  규칙 5는 **요약이 있는 채로도 도달한다** — M12의 exit 교차검증 실패
  (`allpass` + `exit 101`)와 이번 폴백(파이프 + `allpass`)이 그렇다. 문구를 규칙 5의
  **렌더 지점**에 붙이는 가장 자연스러운 구현은 요약이 있는데도
  `(no test summary in output)`을 찍어 **거짓이 된다** — 그것도 이 마일스톤의
  기함 사례(#3) 렌더에서. §7 기준 1이 이 참/거짓을 단언한다
- **공허 런** — #5(`ran==0 ∧ filtered_out==0`)는 **거부하지 않는다.** M12가
  `zero_ran_zero_filtered_out_is_a_real_verification_not_a_vacuous_run` 테스트로
  못박은 거래를 반전시키지 않고, 상태선이 침묵하지 않게만 한다

**M12 거래를 왜 유지하는가**: 파일럿이 그 거래의 반증(2건)을 제공한 것은 사실이다.
그러나 반전하면 테스트가 0건인 프로젝트·`cargo build`만으로 검증되는 과제에서
검증이 성립하지 않는다. 거래를 바꾸는 대신 **정보를 준다** — 사용자 결정 2의 직접 적용.

### 3-5. A-3 · 편집 결과를 모델에게

**결함의 실체는 "신호 부재"가 아니라 "승인자에게만 가고 모델에게는 안 감"이다.**
`render_diff`는 이미 `edit_file.rs:361`·`write_file.rs:46·51`에서 호출되어
`preview()`를 통해 승인 게이트에 뜬다. 모델이 받는 것은 `edit_file` 성공 시
±3줄 컨텍스트뿐이다.

파일럿에서 관측된 세 형태 전부가 이 비가시성의 산물이다.

- **조용한 삭제** — `search`가 실존 텍스트와 맞고 `replace`가 더 짧으면 정확히
  요청대로 지운다. 공개 상수 3개가 소실됐고 finish 요약은 추가만 보고했다
- **조용한 중복** — 앵커 기반 삽입은 멱등이 아니다. 재실행 시 앵커가 살아 있어
  또 1회 매치 → 테스트 사본 2개(E0428)
- **무효 편집이 진짜 편집으로** — `replace == search + "\n"`이 동일성 검사를
  통과해 `Edited (matched exact)`가 되고 상태선이 `files edited: 1`로 없던
  진척을 접지했다. 그 개행은 커밋됐다

**A-3이 덮지 않는 것(5R M2)**: 세 번째 형태에 대해 A-3은 그 편집이 사소했음을
**가시화**할 뿐 `status.record_mutation`을 건드리지 않으므로 **상태선은 여전히
`files edited: 1`을 센다.** 인용한 결함의 후반부(상태선 접지)는 M14 이후에도
잔존한다 — §3-3-2가 #2에 대해 한 것과 같은 "덮지 않는 것 명시"다.

#### 3-5-1. 예산은 중립이 아니라 유계다 (1R I2)

**초판의 "기존 ±3줄을 대체하므로 예산 중립"은 두 방향으로 거짓이다.** 리뷰가
실측했다.

**거짓 A — `write_file`에는 대체할 ±3줄이 없다.** `write_file::run()`은
`"Wrote {} ({} lines)"` 한 줄만 반환한다(`write_file.rs:66`). `render_context`는
`edit_file.rs:381`에만 있다. 따라서 `write_file`에 diff를 붙이는 것은 **순수 추가**다.

**거짓 B — `edit_file`에서도 중립이 아니다.** `render_context`는 편집 크기와
무관하게 상수(한 구간)인 반면 `render_diff`는 헝크 수에 비례한다.

| 케이스 | `render_context` | `render_diff` |
|---|---|---|
| 단일 줄 편집 | 9줄 / 123B | 7줄 / 72B |
| 21줄 삭제 | 9줄 / 120B | **27줄 / 250B** |
| `replace_all` ×8 | 9줄 / 116B | **56줄 / 554B** |

하필 **조용한 삭제**(A-3의 주 표적)에서 3배, `replace_all`에서 6배다.

**정정된 서술**: 15줄 상한이면 편집 턴당 최대 ~15줄로 `render_context` ~9줄 대비
**약 1.7배**다 — **유계이되 중립이 아니다.** §9 R2가 이 위에 선다.

#### 3-5-2. 설계

성공한 `edit_file`/`write_file` 결과에 diff를 첨부한다.

- **모델 채널 전용 상한**을 새로 둔다. `MAX_DIFF_LINES = 120`은 승인 게이트용이며,
  그 값을 매 편집 턴에 붙이면 B-2가 고치려는 문제를 스스로 악화시킨다.
  15줄 안팎으로 하되 정확한 값은 §11 Q2
- **`edit_file`에서는 기존 ±3줄 컨텍스트를 대체한다**(추가가 아니다).
  `write_file`에는 대체할 것이 없으므로 순수 추가임을 인정하고 상한으로만 유계
- **`write_file`을 범위에 포함한다.** 2R 리뷰가 실측으로 이 결정을 지지했다
  (100줄 파일 기준):

  | 케이스 | diff 줄 수 | 15줄 상한에서 |
  |---|---|---|
  | 1줄만 변경 | 7줄 | 전량 |
  | **1줄 소실 (파일럿 사례형)** | **6줄** | **전량** |
  | 10곳 분산 변경 | 70줄 | 2헝크만 |
  | 전면 재작성 | 121줄 | 첫 ~7 변경줄만 |

  **파일럿의 동기가 된 사례(전체 재작성 중 한 줄 소실)는 6줄로 상한에 여유
  있게 들어간다.** 진짜 전면 재작성에서 상한이 무는 것은 사실이나, 그 경우는
  모델이 방금 자기가 쓴 내용이라 신호 가치가 낮다
- **신규 파일 판정은 `existing_text()`가 `None`일 때다**(2R M1). *"`old`가 빈
  문자열"이 아니다* — 기존 빈 파일은 `Some("")`이고, `None`은 새 파일과 **비-UTF-8
  파일 양쪽**에 온다. 두 경우 모두 diff 대신 현행 요약 줄을 유지하며, 비-UTF-8
  덮어쓰기가 같은 분기로 떨어지는 것은 의도다(diff를 낼 원문이 없다)
- **절단 시 삭제 줄을 우선 보존하고, 요약 헤더를 반드시 붙인다**(2R M4).
  **이 "필수"는 장식이 아니라 정합성의 전제다**(4R I1): `EDIT_STRATEGY_CORRECTION`과
  `SR_CORRECTION`이 막힌 모델을 **`write_file` 전면 재작성으로 유도**하는데,
  헤더가 없으면 15줄 상한에서 전면 재작성 diff(121줄)는 첫 몇 줄만 남아 무의미해진다.
  헤더가 있어야 그 유도와 A-3이 정합한다.
  21줄 삭제에 15줄 상한을 대입하면 삭제 줄만 남아 **무엇이 그 자리에 들어왔는지가
  사라진다.** `"-N lines, +M lines"` 형태의 헤더 한 줄은 **선택이 아니라 필수**다 —
  그것만 있으면 121줄 diff도 "10곳 바뀌었고 3줄 사라졌다"는 사실은 전달된다.
  정확한 형태는 §11 Q2

## 4. 축 B — 확정 결함 티켓

### 4-1. B-1 · `reasoning_content` 미파싱

`src/llm/types.rs:39`의 `ResponseMessage`는 `role`·`content` 두 필드뿐이다.
llama.cpp가 Ornith의 사고 토큰을 `reasoning_content`로 분리해 흘리면 serde가
그것을 버린다. 파일럿의 "빈 응답" 4건(각 111~122초 생성, 포착 0자)이 전부
이것이고, 그 자리에 문자열 `"(empty)"`가 모델 히스토리에 주입된다.

**수선 대상은 `Delta`가 아니라 `ResponseMessage`다.** 에이전트 루프는
`stream: false`이므로 `Delta`에 필드를 넣으면 `/chat` 경로만 바뀌고 파일럿
세션은 하나도 안 바뀐다. 1R 리뷰가 라이브 서버로 **비스트리밍
`choices[0].message`에도 `reasoning_content`가 온다**는 것을 확인했다(§3-1).

**기대 효과를 낮춰 잡는다 — 이것은 교정이 아니라 진단이다.** 이 수선은 4개 턴
중 어느 것도 되살리지 못한다. `finish_reason: length` + `content` 공백은 모델이
예산을 전부 추론에 쓰고 JSON을 한 글자도 못 냈다는 뜻이라, 잘린 산문을
`parse_turn`에 넣으면 length 실패가 파싱 실패로 바뀔 뿐이다.

**`usage` 노출의 정확한 서술**(1R M3 — 라이브 실측): 서버 응답은
`{"completion_tokens":40,"prompt_tokens":23,"total_tokens":63,...}` 형태로
**`reasoning_tokens` 같은 분해 필드가 없다.** 확인한 턴은 content 0자 /
`reasoning_content` 127자인데 `completion_tokens`는 40 — 추론 토큰이
`completion_tokens`에 **포함**되어 있다. 따라서 산출물은
*"`usage.completion_tokens`를 노출한다. content가 빈 턴에서는 그 값이 곧 추론
소비량이고, 그 외에는 합산값이라 분리되지 않는다"*이다. `ChatResponse`의
`usage` 파싱이 현재 없으므로 함께 추가한다.

**수용 기준**: §7 기준 2.

### 4-2. B-2 · `pack()` 데이터 손실

`src/session.rs`의 쌍 삭제 `drain(1..=2)`가 사용자 과제 메시지를 제거한다.
세션 1(Z1)은 예산 3686에 대해 3751(65 초과)로 축약 대상이 없어 쌍 삭제가
발동했고, **user 과제 + assistant blob이 함께** 사라졌다. 이후 과제 부재가
영구적이고 모델은 한국어 과제에 영어로 되물으며 끝났다.

**교란 요인을 먼저 적는다**: 심각도는 파일럿 자신의 `max_output_tokens=4096`
탓이다. 기본값 2048이면 예산 5529로 이 세션은 통과하고 `pack()`이 발동하지
않는다. **기본 설정에서는 재현되지 않는다.** 결함은 실재하지만 우선순위 근거를
"파일럿에서 N세션"으로 쓰면 안 된다.

**(c) — `max_output_tokens` 상향 시 `input_budget` 붕괴 방지.** 하한을 두거나
경고한다. **가장 싸고, (a)·(b)가 못 덮는 경우까지 덮는다.** 이 항목을 먼저 한다.

**(b) — `finish_reason: length` 절단 blob을 히스토리에 push하지 않는다.**
세션 1을 단독 해결한다. 문자 그대로 구현하면 부작용이 **3건** 발생하므로
설계에 포함한다.

1. **role 교대 위반** — assistant push를 생략하면 `[system, user 과제, user 회복]`으로
   user가 연속된다. 교정 담당 `merge_adjacent_same_role`는 `pack()`의 쌍 삭제 루프
   안에서만 호출되는데 (b)의 목적이 바로 pack 미발동이라 **영영 교정되지 않는다.**
   → `push`가 아니라 `push_user_request`(`session.rs:95`, 꼬리가 user면 병합)를 탈 것
2. **트랜스크립트 증거 소멸** — `Session::push`는 `transcript.record`와
   `messages.push`를 함께 한다. push를 생략하면 절단 blob이 **JSONL에서도**

   → `record_extra()`(`session.rs:104`)로 트랜스크립트는 보존할 것
3. **회복 문구 누적** — 아래 §4-2-1

#### 4-2-1. 부작용 3 · 회복 문구가 회수 불가능한 자리에 쌓인다 (2R C2)

**개정 1이 도입한 부작용이다.** `push_user_request`(`session.rs:95`)는 꼬리가
user면 **기존 메시지에 병합**한다. length 턴에서 assistant push를 생략하면 꼬리는
항상 user(직전 tool_result 또는 과제 메시지)이므로 회복 문구
(`agent/mod.rs:243`)가 **매번 그 메시지 뒤에 덧붙는다.** length 루프가 N번 돌면
같은 문장이 N벌 쌓인다.

그리고 그 자리는 `</tool_result>` **뒤 접미**다. `pack()`의 축약 단계는 이 접미를
**의도적으로 보존**하고(`session.rs:133~136`, 주석이 *"push_user_request의 후속
요청은 보존한다"*고 명시), 쌍 삭제 단계는 메시지를 통째로 지울 뿐 접미만 걷어낼
수단이 없다. **회수 경로가 존재하지 않는다.** M11이 상태선에 대해 정확히 이
문제를 겪고 `remove_status_note()`(keep-only-latest)로 해결했는데 대응물이 없다.

2R 리뷰가 `pack(10)`(가능한 가장 공격적인 예산)에서 재현했다:

```
[1] user:
<tool_result name="run_command">
[tool result elided]
</tool_result>

Your previous response was cut off by the output token limit. …
Your previous response was cut off by the output token limit. …
Your previous response was cut off by the output token limit. …
```

**과제 메시지는 삭제됐고 중복 보일러플레이트 3벌(≈360B)은 전부 살아남았다.**
`pack()`이 도구 결과 본문을 5단어로 줄이면서 자기가 못 건드리는 접미의 360B는
그대로 뒀다.

**심각도를 정확히 적는다.** (b)의 **순효과는 여전히 크게 양(+)이다** — 제거되는
절단 blob은 최대 `max_output_tokens`(파일럿 설정 4096 토큰)이고 새로 쌓이는
회복 문구는 회당 ~30 토큰이다. **"(b)가 pack() 문제를 악화시킨다"고 쓰면
틀린다.** 정확한 진술:

> (b)는 큰 손실을 없애는 대신 **작지만 구조적으로 회수 불가능한 잔여물**을
> 만든다. `max_turns=25` 전량 length 루프에서 ~750 토큰이 영구 고착되며, 이는
> `input_budget`(파일럿 형태에서 3686)의 **약 20%**다.

파일럿의 최대 실패 범주가 **"length 루프 9건"**이라는 점에서 이론적 우려가 아니다.

**수선**: `push_user_request` 직전에 **중복 제거** — 꼬리 메시지가 이미 회복
문구로 끝나면 덧붙이지 않는다(상태 없음, ~3줄). §7 수용 기준 3의 단언 ④가
이것을 반증 가능한 형태로 고정한다.

**중복 제거로 충분하다 — M11식 keep-only-latest는 불필요하다(3R I2, 실측).**
length 턴 사이에 다른 턴이 끼면 꼬리가 바뀌어 중복이 다시 생긴다. 그 잔여가
수용 가능한지를 3R 리뷰가 측정했다:

| 형태 | 회복문구 사본 | 실린 메시지 | 히스토리 len | `pack(10)` 후 사본 |
|---|---|---|---|---|
| **연속 length ×5** | **1** | 1 | 3 | **1** (과제 생존 ✓) |
| length/도구 교대 ×5 | 5 | 5 | 13 | **0** |

**교대 사본은 서로 다른 5개 메시지에 실리므로 `pack()`의 쌍 삭제가 걷어낸다 —
회수 경로가 존재한다.** 회수 불가능했던 것은 한 메시지에 쌓이는 연속 형태였고
(2R C2가 재현한 그것), 중복 제거가 그것을 1벌로 완전히 닫는다. **즉 수선이
회수 불가능한 형태를 정확히 겨냥하고 있으며, 남는 형태는 기존 경로가 처리한다.**
상태선과 달리 회복 문구는 "최신 1개만 유효"한 성격도 아니다.

이 근거를 여기 적어 두는 이유는 **다음 라운드가 교대 잔여를 신규 결함으로
재논쟁하지 않게** 하기 위해서다.

#### 4-2-2. B-1과의 충돌 해소 — 최종 상태를 하나로 (1R C2)

**초판은 B-1과 B-2(b)가 `agent/mod.rs:239~242`의 동일한 분기를 서로 반대로
요구한다는 사실을 한 번도 언급하지 않았다.** 발동 조건이 정확히 같고
(`finish_reason == "length"`), 파일럿의 빈 응답 4건은 전부 이 경로다.

방치하면 M13 2R가 지목한 "데이터 본 뒤 편한 쪽 고를 seam"이 된다 — 구현자가
B-2(b)를 따르면 B-1의 산출물이 사라지는데, **B-1의 수용 기준은 그래도 통과한다**
(`record_extra`로 라우팅하면 되므로).

**최종 상태를 다음 하나로 못박는다.**

> length 턴에서 절단 blob은 **히스토리에 push하지 않고**(B-2b),
> `record_extra("assistant", …)`로 **트랜스크립트에만** 남긴다.
> B-1의 `reasoning_content` 파싱은 **그 트랜스크립트 레코드의 내용**을
> `"(empty)"`에서 추론 꼬리로 바꾼다.

즉 B-1의 산출물 (a)는 "히스토리에"가 아니라 **"트랜스크립트에"**다. §7 기준 2는
그대로 유효하다 — 애초에 트랜스크립트를 보는 기준이었다.

**구현 순서를 못박는다: B-2(b)를 먼저, B-1을 그 위에**(§10).

#### 4-2-3. (a)는 착수 금지

인덱스 1 고정은 이 마일스톤에서 하지 않는다. 실측 확인: 쌍 삭제 루프

### 4-3. B-3 · `schema_fallback_count` 집계

`Report`/`TaskReport`에 `schema_fallback_count`를 더한다. 지금은 앵커 게이트가

M6이 `_count` 접미사 집계를 도입한 이유가 정확히 이것이다. 기존 키 불변의
가산 변경.

### 4-4. B-4 · 거부 경로 `!stop` 가드 회귀 테스트

**가드는 하나가 아니라 둘이다**(1R I3). `agent/mod.rs:411`(finish_nudge 억제)과
`:415`(status 억제). 1R 리뷰가 **각각 독립적으로 제거해 전체 스위트를 돌렸고 둘 다
`370 passed; 0 failed`** — 둘 다 무방비다. 초판의 "테스트 1개로 닫힌다"는 틀렸다.

- **status 가드** — 테스트로 닫는다
- **finish_nudge 가드** — **도달 불가일 가능성이 높다.** 거부 경로의 이벤트는
  둘뿐이고(`:404~406`), `MutationAttempt`는 `disarm()`을 호출해 `armed=false`로
  만들며 발동 조건이 `self.armed && …`이므로 발동할 수 없다. `Other`는 상태를
  바꾸지 않으므로 그 턴에 새로 조건이 참이 될 수 없다. **단 리뷰어가 이를 코드
  추론으로만 제시했고 실행으로 증명하지 않았다.**

  도달 불가 경로를 택하는 경우 **산출물은 주석이 아니라 실행 가능한 형태여야
  한다**(2R M3) — 주석은 실패할 수 없으므로 수용 기준이 될 수 없다. 거부 경로
  이벤트 2종(`MutationAttempt`/`Other`)에 대해 **무장 상태와 무관하게 `on_turn`이
  `None`임을 단언하는 단위 테스트**를 쓸 것. 작성 시도가 실패할 수 있음은
  §11 Q3

  구조는 다음으로 미룬다

#### 5-1-1. 타임아웃 검증 — 초판의 "(실측)"이 거짓이었다 (1R I1)

초판은 후보 문서를 그대로 옮겨 *"비숫자 값(`600s`·`10m` 같은 오타)이면 `sleep`이
즉시 실패해 상한이 조용히 사라진다(실측)"*고 썼다. **인용된 두 예시가 둘 다
틀렸다.** macOS `/bin/sleep`은 단위 접미사를 지원한다(`usage: sleep number[unit]`,
unit = `s`/`m`/`h`/`d`). 1R 리뷰가 `/bin/sh`로 직접 돌려 확인했다(macOS 26.5.2):

| 값 | 결과 |
|---|---|
| `600s` · `10m` · `1e3` | **수용됨** — 상한 유지 |
| `abc` · `600 `(후행 공백) · 빈 문자열 | 즉시 rc=1 → 상한 소멸 (`invalid time interval`) |
| `-5` | 즉시 rc=1 → 상한 소멸 — **단 사유가 다르다**: `illegal option -- 5`(getopt 단계). `case` 패턴에서 선행 `-` 처리에 주의 |

**기전 자체는 참이다** — `run_with_timeout()`의 서브셸 안 `sleep "$_rwt_timeout"`이
죽으면 `kill_tree`가 영영 안 돈다. 틀린 것은 증거다.

**그리고 진짜 결함은 초판 서술보다 나쁘다**: `10m`은 실패하지 않고 **조용히
600초로 해석된다.** 변수 이름이 `..._SECS`인데 값이 분 단위로 먹히고 아무 경고도
없다. **"상한이 사라진다"가 아니라 "상한이 60배로 조용히 바뀐다"**가 이 예시의
실제 실패 모드다.

따라서 `case`는 **실패 방지가 아니라 숫자만 허용**이어야 한다 — `600s`·`10m`·`1e3`도
거부한다. 제안된 수선 자체는 그대로 옳다.

**이 항목이 남기는 교훈**: 이 절은 바로 아래 문단에서 M13의 BSD `find` 사고를
인용하며 *"도구는 그 도구가 도는 환경에서 검증하라"*고 경고한다. **경고를 적은
그 절이 같은 실패를 저질렀다** — §12-5의 승계 규율이 스펙 자신에게 적용되지
않은 사례다. 후보 문서(`docs/m14-candidates.md` §4-2)도 같은 오류를 담고 있으므로
구현 태스크에서 함께 정정한다.

**검증 규율**: `pilot.sh`는 `/bin/sh`로 돈다. 대화형 셸의 함수·별칭·PATH가
검증을 무효화할 수 있다([[verification-environment-shadowing]]).
**`/bin/sh scripts/pilot.sh` 형태로 직접 돌릴 것.** 2R 리뷰가 `LOCO_BIN=/usr/bin/true`
스텁 + 파이프 stdin으로 **실 세션 없이 전 경로가 완주 가능함**을 실증했으므로
§7 수용 기준 5가 관측 대상을 명시한다. (부수: macOS에는 `/bin/true`가 없다.
`/usr/bin/true`를 쓸 것.)

서술한다. 아래 수치는 1R 리뷰가 `docs/baselines.md` M13과 전수 대조해 일치 확인했다.

- **경로 지정 과제가 성공 봉투다** — 파일럿 13세션 중 성공 7(감사 전 최대치),
  경로 미지정 7세션은 0/7
- **성공 봉투의 크기** — 무경합 성공 4건의 diff 추가줄은 +0~23이고, 추가줄 29
  이상인 4건 중 무경합 성공은 0건이다

**증거 강도 상한을 반드시 병기한다.** 이 축은 사후 분석이다. `task_type`은 세션
전에 수집됐지만 "파일 경로 지정 여부"라는 분해는 사전등록되지 않았고, 20과제 중
19개를 컨트롤러가 설계했다. **"이 유형이 실패한다"와 "내가 설계한 이 유형의 과제
7개가 실패한다"를 이 데이터로 구분할 수단이 없다.** n=7 vs 13. 사례로 읽어야 한다.

**분모 규칙 병기 의무**: 성공 7은 무효 Z3와 감사 강등된 F5·R5를 포함한 감사 전
최대치이고, 같은 규칙을 적용하면 무경합 기준 4/12다. 0/7 쪽은 어떤 완화도 받지
않으므로 두 수치를 나란히 쓸 때는 반드시 병기한다.

**금지 서술**: *"합성 세트에서 이미 이 축만 흔들리고 있었다"*고 쓰지 말 것 —
1차 자료가 반증한다. 축 안 29/30(96.7%) vs 축 밖 41/42(97.6%)이고, 엄격 기준으로는
축 밖이 오히려 더 나쁘다(29/30 vs 40/42). **합성 세트는 이 축을 구별하지 못하며,
대비를 만드는 것은 실레포 쪽 0/7이다.** 또한 합성 5과제 중 2개는 "기전 제공"(cargo
실행 지시)인데 파일럿 7세션에는 기전을 주는 것이 없으므로, `29/30`을 단독 인용하면
"기전 있음 vs 없음"을 "경로 있음 vs 없음"으로 읽게 된다 — `17/18`(심볼만 주는 3과제)이
더 엄밀한 대응임을 병기한다.

**게이트 4종** (전부 필수):

### 8-3. M14 비교가능성 각주 — `verify_allpass`·`verify_total`은 전후 직접 비교 불가

**§3-4-2의 규칙 4 → 규칙 5 폴백이 기존 지표의 모집단을 바꾼다**(5R I1).
`exp_metrics.py`의 검증 줄 분포는 상태선 문자열을 **리터럴로 매칭**한다
(`scripts/exp_metrics.py:33~35`) — `verify_total` = `"verification: last cargo test: "`,
`verify_allpass` = `"verification: last cargo test: all "`. 폴백은 파이프 + allpass
렌더를 규칙 5 문자열(`"verification: last command exited N"`)로 보내므로 **어느
패턴에도 안 걸린다.**

> **모델이 파이프를 쓰는 만큼 `verify_allpass`와 `verify_total`이 함께 내려간다.
> 하락은 회귀가 아니라 폴백이 작동한 증거일 수 있다.**

§8의 게이트는 M14 이전 배치(`20260719T093254Z`)와 대조하므로, 두 배치의
`verify_allpass`를 나란히 놓고 "검증이 줄었다"로 읽으면 **오독**이다.

**파생 `verify_failed`는 보존된다** — `verify_total − verify_allpass`이고 규칙 2는
폴백의 영향을 안 받으며 두 원지표가 같은 양만큼 줄기 때문이다(5R 실측 확인).
깨지는 것은 두 원지표뿐이다.

**선례**: M12 `sr_error`(검사 순서 변경으로 발동 조건 이동 — CLAUDE.md에 한 문단),
M13 T7 `verify_*`(무뮤테이션 분기 렌더로 구조적 **상향** — `docs/baselines.md` 각주).
**M14는 같은 지표를 반대 방향으로 이동시킨다.**

### 8-4. 관측 항목 (게이트 아님, 판정에 쓰지 않음)

- A-1 파이프 해제 차단 발동 횟수 · 오발동 여부(전수 확인) · **새 VERIFY_NUDGE 문구 발동 횟수**
- **FINISH_NUDGE 발동 횟수** — §3-3-3의 회귀를 배치에서도 볼 수 있게(2R C1)
- A-3 diff 첨부 횟수 · 절단 발생 횟수
- **풍선효과 감시**: `finish` 누락 스트릭 · 거짓 finish 수 · `stop_cause` 분포 ·
  **동일 파이프 명령 재실행으로 인한 반복 정지**(§3-3-1이 막으려는 것) ·
  **`REPEAT_CORRECTION` 직후 finish가 파이프 VERIFY_NUDGE로 거부된 횟수**(4R I1)

**4R I1 기록 — A-1이 `REPEAT_CORRECTION`의 탈출구 하나를 막는다.** 그 문구
(`agent/mod.rs:51`)는 *"Try a different action, **or call `finish` with your
answer**"*로 두 선택지를 준다. A-1 이후 파이프 재검증 루프에서 모델이 두 번째를

## 10. 구현 순서 (고정)

**축 간 상호작용이 주 리스크이므로 순서를 스펙에서 못박는다.**

```
B-2(c) → B-2(b)+§4-2-1 → B-1 → A-1(§3-3-1 문구 포함) → A-2 → A-3 → B-3 / B-4 → C → D
```

근거:

- **A-1의 문구 갱신(§3-3-1)은 A-1과 분리 불가** — 별도 태스크로 쪼개면 그 사이

  가장 짧은 구현을 고르면 A-2가 렌더할 값이 사라진다. 개정 3에서도 이 의존은
  그대로다(3R M3 재확인)

**A-1 태스크 브리프에 반드시 동봉할 5항목**(3R M3 — 개정 3에서 셋에서 넷으로 늘었다):

## 11. 열린 질문 (구현 태스크에서 확정)

1. **모델 대면 영문 텍스트 4종과 상호 정합·비중복**(2R I3 + 3R C1) — ① A-2의
   상태선 파이프 한정자와 "요약 없음" ② §3-3-1의 파이프 전용 VERIFY_NUDGE 문구
   ③ **§3-3-3-1의 FINISH_NUDGE 파이프 변형 문구** ④ A-3의 diff 헤더.
   M14 이후 **파이프를 언급하는 모델 대면 텍스트가 4종**이 된다(기존 M11
   `run_command.rs:92` 노트 + 위 ①②③).
   - **비중복**: 노트와 상태선은 파이프 명령 한 번에 같은 tool_result 안에 나란히
     붙으므로(둘 다 `merge_note` 경로) ①은 문장이 아니라 **짧은 한정자**여야 한다
   - **정합**: ①②③은 **같은 이벤트를 반대로 부르면 안 된다**(3R C1·4R C1의 실패).
     현행 `FINISH_NUDGE`의 `"You already ran a successful verification"`이 파이프
     경로에서 참이 아니라는 점이 출발점이고, 상태선 규칙 4의 `"all N passed"`도
     같은 문제였다(§3-4-2가 규칙 5 폴백으로 해소)
   - **비중복 논거 보강**(4R C1): **규칙 4와 규칙 5는 서로 다른 사실을 렌더하므로
     같은 한정자 규칙을 적용하지 않는다.** 개정 2·3이 둘을 묶은 것이 4R C1의 원인
   - **기존 문구 1건 동반 수정**(4R M3): 현행 `VERIFY_NUDGE`는
     *"You modified files but **never ran** a verification command"*인데,
     `mutate → cargo test(exit 101, 해제됨) → mutate → finish` 경로에서 모델은
     검증을 **돌린 적이 있다**. 상태선 규칙 1은 같은 상황을
     *"verification: none **since your last edit**"*로 정확히 렌더하므로
     **두 장치가 같은 사실을 다르게 말한다.** §3-3-1이 어차피 V 계열을 손대므로
     `"...since your last edit"` 한 구절을 추가할 것. M14가 만든 결함은 아니지만
     A-1의 사정거리 안이다
   - **발동 조건은 여기가 아니다**(4R C2): FINISH_NUDGE 변형의 **발동 조건**은
     열린 질문이 아니라 §3-3-1이 정의하는 상태다. Q1에는 **문구만** 남긴다
   - **절차 고정**(3R 리뷰어 제안): 장치를 추가·유지하는 판단을 할 때마다 **그
     장치의 모델 대면 텍스트를 함께 읽을 것.** 2R이 `finish_nudge`를 상태기계로만
     읽고 문자열을 열지 않아 C1을 놓쳤다
2. **A-3의 diff 상한 값과 절단 형태** — 상한은 15줄 안팎. `"-N lines, +M lines"`
   요약 헤더는 **필수**(2R M4)
3. **`agent/mod.rs:411` finish_nudge 가드의 도달 가능성** — 도달 불가면 산출물은
   주석이 아니라 단위 테스트(§4-4). **작성 시도가 실패할 수 있다**
4. **B-2(c)의 형태** — 하한인가 경고인가, 임계값은 얼마인가
5. **§8 임계값 최종 확정** — M12 방식(`≥33`) vs M13 방식(`앵커−4`=31). 사전등록에서
6. **§7 기준 1의 테스트 형태**(2R M2 + 3R M1) — `render()`가 비공개이므로 `on_turn`을
   통해 도는 상태 행렬이어야 한다. 단일 케이스로는 명제가 문자 그대로 만족되어
   반증력이 없다.
   - **규칙을 열거하지 말고 입력 차원을 열거할 것.** 규칙은 자유 변수가 아니라
     `(last_test_summary, last_cmd_exit)`에서 **파생**된다. 개정 2의 "두 분기 ×
     5규칙" 서술은 차원을 잘못 잡았고, 그 방식으로 쓰면 ~10케이스에 그쳐 특히
     **규칙 4의 exit 교차검증**(`allpass` + `exit 101` → 규칙 5 폴백, M12가 절단
     위장을 막으려 넣은 장치)이 누락된다
   - 실제 차원: `mutated_paths 유무 × mutated_since_verify × 요약 5형태 ×
     exit 3형태` = **60상태**. 3R 리뷰가 열거해 **현행 위반 0건**, 실행 1ms 미만,
     테스트 ~25줄임을 확인했고 4R이 재검증했다. `pending`/`has_note_channel`/
     케이던스 턴은 렌더 **여부**만 바꾸고 **형태**는 안 바꾸므로 차원에서 제외한다
   - **그 제외가 정당하려면 렌더가 실제로 일어나야 한다**(4R I3·M2) —
     **렌더 건수 == 열거한 상태 수**를 단언할 것(5R M1: **60은 매직 넘버가 아니라
     참고 실측값**이다 — 6번째 요약 형태를 추가한 더 촘촘한 테스트가 기준을 실패하면
     안 된다. 불변식의 본질은 *하나도 건너뛰지 않았다*이다). `on_turn`은
     `mutation_ok ∨ 케이던스(5/10/15/20) ∨ pacing ∨ pending`일 때만 렌더하므로,
     turn 3 같은 값을 고르면 무뮤테이션 30상태가 `None`으로 빠지고
     `else { continue }` 패턴에서 **테스트는 초록불인 채 절반이 사라진다**
   - **더 직접적인 단언을 함께 둘 것**: 들여쓰기 불변식은 실제로 중요한 성질
     (`remove_status_note()`가 전부 회수한다)의 **대리**다. 라운드트립 —
     주입 → `remove_status_note()` → 잔여 0 — 이 직접적이며 1R 프로브가 이미
     그 형태였다

## 12. 승계할 규율 (M13 §5)

이 마일스톤의 분석·리뷰에 그대로 적용한다. **1R·2R 모두에서 이 규율이 지켜지지
않은 사례가 나왔다** — 1R은 컨트롤러 쪽(§5-1-1), 2R은 리뷰어 쪽(§14-1).

1. **개입이 "궤적을 바꿨다"는 주장은 원장 판정과 diff 실물로 교차 확인한 뒤에만 쓴다**
2. **"성공"이 무엇의 성공인지 명시한다** — 도구 디스패치 Ok / 산출물 컴파일 /
   과제 충족 / 사용자 채택은 전부 다르다
3. **재계수는 귀인 오류를 못 잡는다.** 독립 리뷰 축에 "귀인"을 명시적으로 넣을 것
4. **자기가 새로 쓴 수치에 자기 규율을 적용한다** — 1R I4가 이것
5. **도구는 그 도구가 실제로 도는 환경에서 검증한다** — 1R I1이 이것
6. **"수정했다"는 그 수정에 대한 리뷰를 면제하지 않는다** — **개정 1이 실증했다**:
   1R 반영이 Critical 2건을 새로 만들었고 둘 다 리뷰어의 처방 안에 있었다.
   **이 개정 2도 예외가 아니다**
7. **검증 기준은 실패할 수 있는지도 확인한다** — 2R I1(수용 기준 5)이 이것

### 14-1. 결함의 성격이 이동했다 (2R N2)

M13 §5가 같은 이동을 기록해 다음 라운드의 리뷰 축을 정하는 데 썼다. M14도 적어 둔다.

| 라운드 | 결함의 성격 | 누가 만들었나 |
|---|---|---|
| **1R** | 코드 대조 누락 · 미실행 인용 · 계수/귀인 오류 | 컨트롤러 |
| **2R** | **처방의 2차 효과** — 반영 자체는 정확한데 처방이 다른 장치와 상호작용 | **리뷰어**(1R 처방 2건) |
| **3R** | **텍스트 층위 모순**(처방이 옳게 작동한 결과로 드러남) + **실측 시나리오를 수용 기준으로 승격할 때의 조건 유실** | **공동** — `VerifyOk` 유지는 리뷰어의 2R 권고, 문구·조건 검토 누락은 양쪽 |

2R의 Critical 2건은 리뷰어가 1R에서 제안한 해법 안에 있었고, 리뷰어가 그 사실을
스스로 명시했다("1R에서 내가 놓친 것"). 원인도 같다 — **§12-4("자기가 새로 쓴
것에 자기 규율을 적용한다")를 자기 처방에는 돌리지 않았다.**

3R의 C1도 같은 계보다. 2R이 `finish_nudge`를 **상태기계로만** 읽고
(`Other => {}`, `count`, `arm`) 그 기계가 발화하는 **문자열**을 열어보지 않았다 —
1R에서 VERIFY_NUDGE 문구를 열어 C3를 잡았으면서 자매 장치에는 같은 검사를 돌리지
않았다. 그 결과 §11 Q1이 3종→4종으로 늘었다.

| **4R** | **미갱신 소비자 1건(상태선 규칙 4) + 플래그 수명** — 전수 감사가 찾아냄 | **공동** — 한정자 처방은 리뷰어의 2R I3, 플래그 교정은 리뷰어의 3R I1 |

### 14-2. A-1의 결함은 4라운드 내내 **한 형태**였다 (4R)

4R 리뷰가 전수 감사 결과로 정리한 것이다.

> **A-1이 어떤 실행을 "검증이 아니다"로 재분류하는데, 그 실행을 이미
> "검증이다"로 전제하고 말하는 다른 장치가 갱신되지 않는다.**

| 라운드 | 갱신 안 된 소비자 |
|---|---|
| 1R C3 | `VERIFY_NUDGE` 기본 문구 |
| 2R C1 | `finish_nudge` **상태기계**(문구가 아니라 무장 로직) |
| 3R C1 | `FINISH_NUDGE` 문구 |
| 4R C1 | **상태선 규칙 4** |
| **5R I1** | **`exp_metrics.py`의 `verify_*` 매처** — 모델 대면이 아닌 **계측 소비자** |

**소비자는 두 층이다(5R M3).** 4R의 전수 감사는 **모델 대면 문자열 13종**만 봤고
그 층에서 소진을 선언했다. 그런데 상태선은 **모델도 읽고 `exp_metrics.py`도 읽는다** —
렌더 문자열을 바꾸면 지표 정의가 함께 바뀐다(M12·M13이 두 번 문서화한 사실).
5R이 그 층을 닫았다.

> **절차 확장**: *"장치의 모델 대면 텍스트 **와 그 텍스트를 매칭하는 계측기**를
> 함께 읽는다."* 다음 마일스톤은 이 검사를 두 층으로 돌릴 것.

**두 층 모두에서 소진됐다** — 모델 대면 4종은 1R~4R에서, 계측 1종은 5R에서
닫혔고, 코드베이스에서 "검증이 있었는가"를 소비하는 지점은 그 다섯이 전부다
(`report.json`은 검증 상태를 싣지 않는다).

**4R에서 13종 전수 감사를 돌려 모델 대면 층을 소진했다** — 나온 것은 C1(상태선
규칙 4)과 I1(`REPEAT_CORRECTION`, 연성)뿐이고 나머지는 정합으로 확인됐다
(M11 파이프 노트 ↔ 파이프 VERIFY_NUDGE 정합 / M12 공허 런 노트 ↔ 상태선 규칙 3은
조건 배타 / `EDIT_STRATEGY_CORRECTION`·`SR_CORRECTION`의 write_file 유도 ↔ A-3은
§11 Q2 헤더가 **필수**인 한 정합). **리뷰어 판단: 이 두 Critical이 반영되면
5R에서 수렴할 것으로 본다.**

**5R의 리뷰 축**: 개정 4가 새로 쓴 것은 §3-4-2의 규칙별 한정자 분리(규칙 4 →
규칙 5 폴백), §3-3-1의 무조건 대입 + 우선순위 + 타임아웃 표, §3-3-2의 타임아웃
동작 문단, §7 기준 6종의 유실 조건 5건, §11 Q1의 정합 조항·V 문구 수정·Q6의 렌더
건수 단언, §10 브리프 5항목, §8-3의 `REPEAT_CORRECTION` 관측이다.

4. **유실 조건 5건 삽입에서 본질적 재유실 없음** — 부작용은 M1(매직 넘버)뿐
5. **V + "since your last edit"는 항상 참** — `mutated_since_verify = true`는
   코드 전체에서 **단 한 곳**(`agent/mod.rs:475`, 뮤테이션 성공 디스패치)에서만
   세워지므로 뮤테이션 0회에서 V는 발동 불가
6. **§10 브리프·§8-4 관측·§3-5-2 헤더 근거 누락 없음**

### 17-1. 5라운드 총평 (컨트롤러 기록)

리뷰 5회 동안 **반박 0건**이다. 컨트롤러가 매 라운드 Critical을 코드로 독립
재확인했고 전부 리뷰어가 옳았다. 결함의 성격은 라운드마다 이동했다(§14-1):

```
1R 코드 대조 누락 → 2R 처방의 2차 효과 → 3R 텍스트 층위 모순
→ 4R 미갱신 소비자(모델 대면) → 5R 미갱신 소비자(계측)
```

**A-1(파이프 가드)이 1R~4R의 Critical을 전부 만들었다.** 4R 시점에 컨트롤러가
사용자에게 이 패턴을 보고하고 "A-1을 순수 알림으로 축소할 것인가"를 물었으며,
사용자가 **"4R 한 번 더"**를 선택했다. 그 판단이 옳았음이 5R에서 확인됐다 —
결합 그물의 소비자 목록이 유한했고 두 층 모두에서 닫혔다.

**이 마일스톤이 남기는 방법론 산출**: §12에 승계 규율 7개, §14-1에 결함 성격
이동표, §14-2에 "미갱신 소비자" 패턴과 **두 층 감사 절차**. 마지막 것이 가장
재사용 가치가 높다 — 하네스의 어떤 장치가 "무엇이 참인가"를 재분류할 때,
그 사실을 전제하는 소비자가 **모델 대면과 계측 양쪽에** 있다.
