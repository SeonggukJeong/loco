## 1. 배경 — 왜 이 축인가

M13은 북극성을 합성 평가에서 실사용으로 옮겼고, 그 첫 관측이 헤드라인을
뒤집었다: **합성 통과율이 실사용을 예측하지 못한다.** 동시에 파일럿은 loco
하네스 자신의 결함을 여러 형태로 드러냈다.

M12는 "테스트가 통과했는가"를 exit code 대신 파싱된 libtest 결과에 접지했다.
파일럿이 보여준 것은 그 위층이 비어 있다는 사실이다 — **"맞는 것을 테스트했는가"는
아무것도 접지하지 않는다.** `docs/m14-candidates.md` §1-3이 다섯 형태를 센다:

| 형태 | 파일럿 사례 | 기전 |
|---|---|---|
| 파이프가 exit 0 날조 | 5세션 | `has_unquoted_pipe`가 노트 채널에만 연결 |
| 컴파일 실패인데 exit 0 | 3세션 | `test result:` 줄 부재 → `parse_test_summary`가 `None` |
| `tail`이 실패 섹션 절단 | 1세션 | 통과 섹션만 남아 규칙 2·4를 모두 통과 |
| `cargo test --lib`을 bin 전용 크레이트에 | 4세션 | 실행 0건 |
| 크레이트 스코프 오지정 | 2세션 | `ran==0 ∧ filtered_out==0` — M12 공허 런 가드가 침묵 |

이 다섯을 코드 기준으로 다시 묶으면 기전은 둘로 수렴한다.

- **파이프 계열** (#1·#3, #2의 파이프 기인분) — 언어 무관하게 판별 가능
- **공허/무증거 계열** (#4·#5) — libtest 파싱에 의존

이 축을 고르는 근거는 세 가지다. ① 검증 날조는 봉투 안/밖과 무관하게 발생한다 —
과제문이 파일 경로를 줘도 모델은 exit 0을 날조한다. ② M12 스코핑에서 사용자가
정한 원칙("잘 못하는 걸 잘하게 만들려고 시간 쓰지 말고, 하네스 자신의 거짓
초록불을 고쳐라")의 직접적 연속이다. ③ 파일럿의 성공 판정 4건이 실행·대조하면
뒤집혔다는 관측(`docs/m14-candidates.md` §4)과 같은 뿌리다.

### 1-1. 이 마일스톤이 무엇을 주장하지 않는가

**효과는 `tasks/`로 실증되지 않는다.** 검증 날조는 파일럿 실레포에서 15세션분
관측됐지만 합성 세트에서는 거의 발동하지 않는다. M12가 같은 상황을 명시적으로
받아들여 회귀 게이트로 축소했고, M14도 같다. 측정은 **무회귀 확인 + 장치 발동
여부 관측**이며, **효과 확인은 다음 파일럿의 몫이다.**

이것을 성공 기준에 미리 적어 둔다(§7). 게이트 배치가 통과했다는 사실로
"검증 날조가 줄었다"를 주장하면 안 된다.

컨트롤러가 브레인스토밍 중 후보 2-5를 C축에서 A축으로 재분류했다. 문서상

접지했다"* — 같은 거짓 초록불 계열이다. 사용자가 이 재분류를 승인했다.

## 3. 축 A — 검증 접지 상위층

### 3-1. 착수점 실측

브레인스토밍 중 코드로 확인한 사실이다. 설계가 이것들에 의존하므로 구현 전에
재확인할 것.

| 사실 | 위치 | 설계 영향 |
|---|---|---|
| `has_unquoted_pipe`가 **비공개** | `src/tools/run_command.rs:16` | 가시성 승격 필요 |
| `StatusNote::record_command_result(exit, summary)`가 **명령 문자열을 안 받음** | `src/agent/status_note.rs:73` | 시그니처 변경 필요 |
| `VerifyOk` 무장 판단이 **`agent/mod.rs`의 이벤트 구성 지점**에 있음 | `src/agent/mod.rs:456~498` | `finish_nudge.rs`는 **안 건드려도 된다** — 후보 문서 추정보다 싸다 |
| `render_diff`가 **이미 `edit_file.rs`·`write_file.rs` 안에서 호출됨** | `src/tools/diff.rs:5` | A-3은 "diff를 만든다"가 아니라 "만든 것을 결과에도 붙인다" |
| `MAX_DIFF_LINES = 120` | `src/tools/diff.rs:3` | 승인 게이트용 상한. 모델 채널엔 **별도의 더 작은 상한**이 필요 |
| `ResponseMessage` = `role` + `content`뿐 | `src/llm/types.rs:39` | B-1의 수선 대상. `Delta`(`:73`)가 아니다 |

### 3-2. 왜 "긍정 증거 요구"로 전면 뒤집지 않는가

가장 단순해 보이는 설계는 검증 성립 술어를 뒤집는 것이다 — `parse_test_summary`가
`ran > 0 ∧ failed == 0`을 실제로 확인했을 때만 검증으로 친다. 이러면 다섯 형태
중 넷이 한 번에 덮인다.

**이 안을 기각한다. `parse_test_summary`는 libtest 전용이기 때문이다.**

loco는 Rust 전용 도구가 아니다. `run_command`는 `make`·`npm test`·`pytest`
무엇이든 실행하고, 그 출력에는 `test result:` 줄이 없다. 긍정 증거를 요구하면
비-Rust 레포에서 **검증이 영영 성립하지 않고** VERIFY_NUDGE가 매 런 발동한다.
M12가 `None`을 보수적 폴백으로 둔 이유가 정확히 이것이며, 그 판단은 옳았다.

따라서 거부 술어에 넣을 수 있는 것은 **언어 무관하게 판별 가능한 사실**뿐이다.
파이프가 그 조건을 만족한다.

### 3-3. A-1 · 파이프 가드를 거부 술어로

`has_unquoted_pipe`를 `pub(crate)`로 승격하고, `agent/mod.rs`의 단일 지점
(`cmd_exit`/`cmd_summary`/`empty_verify` 계산부)에서 다음 둘 **양쪽**에 가드를 건다.

- `finish_nudge::TurnEvent::VerifyOk` 무장
- `VERIFY_NUDGE` 해제

즉 **파이프가 있는 `run_command` Ok는 검증으로 치지 않는다.**

이 수선의 논거는 새 정책 도입이 아니라 **자기모순 해소**다. M11이 이미 파이프
명령의 Done 결과에 "이 exit code는 파이프라인 마지막 명령의 것"이라는 출처
안내를 붙이고 있다. 그런데 정작 하네스 자신은 같은 턴에 그 exit 0을 검증으로
세고 있었다. 모델에게 믿지 말라고 알린 신호를 하네스가 믿고 있었던 것이다.

**덮는 범위**: #1(5세션) · #3(`tail` 절단, 1세션) · #2 중 파이프 기인분.

**정당한 파이프 검증의 처리** — `cargo test 2>&1 | tail -50`은 실제 검증
의도이지만 이 가드에 걸린다. 설계상 수용하며, 근거는 셋이다. ① `VERIFY_NUDGE`는
런당 1회 래치라 finish를 영구 차단하지 않는다. ② 거부 시 모델에게 이유가
전달되고, M11 파이프 노트가 같은 방향을 이미 가리키고 있어 파이프 없이 재실행할
경로가 존재한다. ③ 파이프로 절단된 출력은 **실제로** 하네스가 검증 여부를 알 수
없는 출력이다 — 가드가 참을 말하고 있다.

**그럼에도 풍선효과는 실재하는 위험이다.** §8-1에 관측 항목으로 고정한다.

### 3-4. A-2 · 알림 전면

`StatusNote::record_command_result`가 명령 문자열을 받도록 시그니처를 넓히고,
상태선의 verification 줄이 다섯 형태를 **정직하게** 렌더한다.

원칙: **언어 무관한 사실만 적는다.** libtest 파싱이 `None`이어도 파이프 여부와
"요약이 없다"는 사실은 알 수 있다.

- **파이프** — exit code를 근거로 렌더할 때(현행 규칙 4·5) 그 exit code가 파이프라인
  마지막 명령의 것임을 표시한다
- **요약 부재** — `parse_test_summary`가 `None`이면 현행 규칙 5가 "last command
  exited N"만 렌더한다. 여기에 요약을 찾지 못했다는 사실 한 줄을 더한다.
  #2(컴파일 실패인데 exit 0)와 #4(`--lib`을 bin 전용 크레이트에)가 여기 해당
- **공허 런** — #5(`ran==0 ∧ filtered_out==0`)는 **거부하지 않는다.** M12가
  `zero_ran_zero_filtered_out_is_a_real_verification_not_a_vacuous_run` 테스트로
  못박은 거래를 반전시키지 않고, 상태선이 침묵하지 않게만 한다

**M12 거래를 왜 유지하는가**: 파일럿이 그 거래의 반증(2세션)을 제공한 것은
사실이다. 그러나 반전하면 테스트가 0건인 프로젝트·`cargo build`만으로 검증되는
과제에서 검증이 성립하지 않는다. 거래를 바꾸는 대신 **정보를 준다** — 사용자
결정 2("알림 전면, 거부는 파이프만")의 직접 적용이다.

구체적 문구는 구현 태스크에서 확정한다(모델 대면 텍스트는 영문 — §6 제약).

### 3-5. A-3 · 편집 결과를 모델에게

**결함의 실체는 "신호 부재"가 아니라 "승인자에게만 가고 모델에게는 안 감"이다.**
`render_diff`는 이미 `edit_file.rs`·`write_file.rs` 안에서 호출되어 `preview()`를
통해 승인 게이트에 뜬다. 모델이 받는 것은 성공 시 ±3줄 컨텍스트뿐이다.

파일럿에서 관측된 세 형태 전부가 이 비가시성의 산물이다.

- **조용한 삭제** — `search`가 실존 텍스트와 맞고 `replace`가 더 짧으면 정확히
  요청대로 지운다. 공개 상수 3개가 소실됐고 finish 요약은 추가만 보고했다
- **조용한 중복** — 앵커 기반 삽입은 멱등이 아니다. 재실행 시 앵커가 살아 있어
  또 1회 매치 → 테스트 사본 2개(E0428)
- **무효 편집이 진짜 편집으로** — `replace == search + "\n"`이 동일성 검사를
  통과해 `Edited (matched exact)`가 되고 상태선이 `files edited: 1`로 없던
  진척을 접지했다. 그 개행은 커밋됐다

**설계**: 성공한 `edit_file`/`write_file` 결과에 diff를 첨부한다.

- **모델 채널 전용 상한**을 새로 둔다. `MAX_DIFF_LINES = 120`은 승인 게이트용이며,
  이 값을 매 편집 턴에 붙이면 §4-2가 다루는 컨텍스트 문제를 스스로 악화시킨다.
  상한은 15줄 안팎으로 하되 정확한 값은 구현 태스크에서 확정
- **기존 ±3줄 컨텍스트를 대체한다 — 추가가 아니다.** 이것이 컨텍스트 예산
  중립성의 전제다
- **절단 시 삭제 줄을 우선 보존한다.** 조용한 삭제가 주 표적이고, 추가 줄은
  모델이 이미 알고 있는 내용이다

## 4. 축 B — 확정 결함 티켓

### 4-1. B-1 · `reasoning_content` 미파싱

`src/llm/types.rs:39`의 `ResponseMessage`는 `role`·`content` 두 필드뿐이다.
llama.cpp가 Ornith의 사고 토큰을 `reasoning_content`로 분리해 흘리면 serde가
그것을 버린다. 파일럿의 "빈 응답" 4건(각 111~122초 생성, 포착 0자)이 전부
이것이고, 그 자리에 문자열 `"(empty)"`가 모델 히스토리에 주입된다.

**수선 대상은 `Delta`가 아니라 `ResponseMessage`다.** 에이전트 루프는
`stream: false`이므로 `Delta`에 필드를 넣으면 `/chat` 경로만 바뀌고 파일럿
세션은 하나도 안 바뀐다.

**기대 효과를 낮춰 잡는다 — 이것은 교정이 아니라 진단이다.** 이 수선은 4개 턴
중 어느 것도 되살리지 못한다. `finish_reason: length` + `content` 공백은 모델이
예산을 전부 추론에 쓰고 JSON을 한 글자도 못 냈다는 뜻이라, 잘린 산문을
`parse_turn`에 넣으면 length 실패가 파싱 실패로 바뀔 뿐이다. 산출물은 좁다:
(a) `"(empty)"` 리터럴 대신 추론 꼬리를 히스토리에, (b) 추론 토큰 소비량 노출.

(b)를 위해 `ChatResponse`의 `usage` 파싱을 함께 넣는다(현재 미파싱). 계측
부채(§5-2)와 정합한다.

**수용 기준 — 반증 가능해야 한다**: `max_output_tokens`를 추론이 끝나기 전에
잘리는 값으로 강제로 낮춰 `finish_reason == "length"` ∧ `content` 공백을 **먼저
재현**하고, 그 턴의 `.loco/sessions/*.jsonl` assistant 행이 `"(empty)"`가 아님을
단언한다. **"에이전트 턴 1회를 돌려본다"만으로는 안 된다** — 정상 턴은 그 경로에
도달하지 않으므로 결함이 그대로 있어도 통과한다([[verification-criteria-must-be-falsifiable]]).

### 4-2. B-2 · `pack()` 데이터 손실

`src/session.rs`의 쌍 삭제 `drain(1..=2)`가 사용자 과제 메시지를 제거한다.
세션 1(Z1)은 예산 3686에 대해 3751(65 초과)로 축약 대상이 없어 쌍 삭제가
발동했고, **user 과제 + assistant blob이 함께** 사라졌다. 이후 과제 부재가
영구적이고 모델은 한국어 과제에 영어로 되물으며 끝났다.

**교란 요인을 먼저 적는다**: 심각도는 파일럿 자신의 `max_output_tokens=4096`
탓이다. 기본값 2048이면 예산 5529로 이 세션은 통과하고 `pack()`이 발동하지
않는다. **기본 설정에서는 재현되지 않는다.** 결함은 실재하지만 우선순위 근거를
"파일럿에서 N세션"으로 쓰면 안 된다.

**(c) — `max_output_tokens` 상향 시 `input_budget` 붕괴 방지.** 하한을 두거나
경고한다. **가장 싸고, (a)·(b)가 못 덮는 경우까지 덮는다.** 이 항목을 먼저 한다.

**(b) — `finish_reason: length` 절단 blob을 히스토리에 push하지 않는다.**
세션 1을 단독 해결한다. 문자 그대로 구현하면 부작용 2건이 **둘 다** 발생하므로
설계에 포함한다.

1. **role 교대 위반** — assistant push를 생략하면 `[system, user 과제, user 회복]`으로
   user가 연속된다. 교정 담당 `merge_adjacent_same_role`는 `pack()`의 쌍 삭제 루프
   안에서만 호출되는데 (b)의 목적이 바로 pack 미발동이라 **영영 교정되지 않는다.**
   → `push`가 아니라 `push_user_request`(꼬리가 user면 병합)를 탈 것
2. **트랜스크립트 증거 소멸** — `Session::push`는 `transcript.record`와
   `messages.push`를 함께 한다. push를 생략하면 절단 blob이 **JSONL에서도**
   사라진다. M13의 디코딩 퇴화 분석은 전부 그 blob을 읽어서 만든 것이다.
   → `record_extra()`로 트랜스크립트는 보존할 것

**(a) — 착수 금지.** 인덱스 1 고정은 이 마일스톤에서 하지 않는다. 실측 확인:
쌍 삭제 루프에는 마지막 메시지 보호가 없다(축약 루프만 `last`를 계산한다).
인덱스 1을 고정하고 드레인을 옮기면 ① 역할 술어도 함께 옮겨야 하고 안 그러면
`else` 분기의 `remove(1)`가 고정한 과제를 지운다 ② 세션 1의 형태(`len 4`)에서
인덱스 3은 **현재 요청**이라 그것을 지운다. 종료성·마지막 메시지 보호 설계가
선행돼야 한다.

**수용 기준**: `max_output_tokens`를 크게 올린 설정으로 세션 1의 형태를 재현해
**셋을 함께** 단언한다 — ① 과제 메시지가 히스토리에 살아남는다 ② 트랜스크립트에
절단 blob이 남아 있다 ③ role 교대가 깨지지 않는다.

### 4-3. B-3 · `schema_fallback_count` 집계

`Report`/`TaskReport`에 `schema_fallback_count`를 더한다. 지금은 앵커 게이트가
`tasks[].runs[].schema_fallback`을 전수 순회해야 하는데, 그 스크립트를 잘못
짜면(첫 런만 본다든지) **fail-open 위험이 Rust에서 스크립트로 이동할 뿐이다.**
M6이 `_count` 접미사 집계를 도입한 이유가 정확히 이것이다. 기존 키 불변의
가산 변경.

### 4-4. B-4 · 거부 경로 `!stop` 가드 회귀 테스트

케이던스 노트를 억제하는 `!stop` 가드는 두 곳(디스패치 경로, 게이트 거부 경로)에
있는데 회귀 테스트는 디스패치 경로만 고정한다. 거부 경로의 `!stop`을 통째로
제거해도 전체 스위트가 초록불이다(M13 T7에서 실측 확인). **테스트 1개로 닫힌다.**

## 5. 축 C — 도구·계측 부채

### 5-1. `pilot.sh` 4건

다음 파일럿에서 처음 쓰이므로 지금 고쳐 둔다. 전부 몇 줄이다.

- **cwd 전제** — `REPO="$(pwd)"`인데 `git diff "$START_REV"`는 레포 전체를 본다.
  서브디렉터리 실행 시 어긋난다. `git rev-parse --show-toplevel`로 고정
- **타임아웃 환경변수 검증** — `PILOT_BUILD_TIMEOUT_SECS`/`PILOT_TEST_TIMEOUT_SECS`에
  비숫자 값(`600s`·`10m` 같은 오타)이면 감시자의 `sleep`이 즉시 실패해 **상한이
  조용히 사라진다**(실측). `case` 한 줄
- **세션 전 `read` 4개의 EOF 내성** — 종료 코드 1이 실제 오류와 구별되지 않는다.
  한 줄 안내 + 주석에 "의도적 제외" 명시
- **INT 트랩** — 포그라운드 자식 대기 중 프로세스 그룹 SIGINT는 트랩을 실행시키지
  않는다(bash 3.2 = macOS `/bin/sh`, 실측). 결과적 동작은 바람직하므로
  **동작 변경이 아니라 문서화**로 처리하고, 세션 중 INT까지 잡는 `wait` 기반
  구조는 다음으로 미룬다

**검증 규율**: `pilot.sh`는 `/bin/sh`로 돈다. 대화형 셸의 함수·별칭·PATH가
검증을 무효화할 수 있다([[verification-environment-shadowing]] — M13에서
BSD `find` 회귀가 "실측 확인"을 통과해 커밋된 전례). **`/bin/sh scripts/pilot.sh`
형태로 직접 돌리고, 외부 명령은 `/usr/bin/`의 실제 바이너리로 확인할 것.**

### 5-2. 트랜스크립트 영속화

M13이 남긴 계측 불가 항목 중 둘을 닫는다.

  수 없어 M13은 빈-content 턴을 대리 지표로 썼다. B-1의 (b)와 같은 방향이다
- **컨텍스트 오버플로가 트랜스크립트에 안 남는다** — `AgentEvent::Notice`가
  영속되지 않아 M13 파일럿 표에서 이 행은 **0이 아니라 미측정**이었다.
  M13 스펙 리뷰의 C1이 정확히 이 공백 때문에 "배치는 정상 종료했는데 흔적 0"인

## 6. 축 D — 봉투 명시 (코드 변경 0)

후보 2-1은 수선이 아니라 범위 결정이었고, 사용자 결정은 **"명시하되 거절/경고
코드는 넣지 않는다"**이다. README 또는 `docs/`에 loco가 서비스하는 과제 봉투를
서술한다.

- **경로 지정 과제가 성공 봉투다** — 파일럿 13세션 중 성공 7(감사 전 최대치),
  경로 미지정 7세션은 0/7
- **성공 봉투의 크기** — 무경합 성공 4건의 diff 추가줄은 +0~23이고, 추가줄 29
  이상인 4건 중 무경합 성공은 0건이다

**증거 강도 상한을 반드시 병기한다.** 이 축은 사후 분석이다. `task_type`은 세션
전에 수집됐지만 "파일 경로 지정 여부"라는 분해는 사전등록되지 않았고, 20과제 중
19개를 컨트롤러가 설계했다. **"이 유형이 실패한다"와 "내가 설계한 이 유형의 과제
7개가 실패한다"를 이 데이터로 구분할 수단이 없다.** n=7 vs 13. 사례로 읽어야 한다.

**분모 규칙 병기 의무**: 성공 7은 무효 Z3와 감사 강등된 F5·R5를 포함한 감사 전
최대치이고, 같은 규칙을 적용하면 무경합 기준 4/12다. 0/7 쪽은 어떤 완화도 받지
않으므로 두 수치를 나란히 쓸 때는 반드시 병기한다.

**금지 서술**: *"합성 세트에서 이미 이 축만 흔들리고 있었다"*고 쓰지 말 것 —
1차 자료가 반증한다. 축 안 29/30(96.7%) vs 축 밖 41/42(97.6%)이고, 엄격 기준으로는
축 밖이 오히려 더 나쁘다(29/30 vs 40/42). **합성 세트는 이 축을 구별하지 못하며,
대비를 만드는 것은 실레포 쪽 0/7이다.** 또한 합성 5과제 중 2개는 "기전 제공"(cargo
실행 지시)인데 파일럿 7세션에는 기전을 주는 것이 없으므로, `29/30`을 단독 인용하면
"기전 있음 vs 없음"을 "경로 있음 vs 없음"으로 읽게 된다 — `17/18`(심볼만 주는 3과제)이
더 엄밀한 대응임을 병기한다.
