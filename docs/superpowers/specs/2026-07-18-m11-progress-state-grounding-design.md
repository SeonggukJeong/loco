## 1. 배경 — 두 번의 데이터 반전이 설계를 정했다

**출발(브레인스토밍 초안)**: 기함을 "전파 상태 접지"로 선정 — uv(update-vat-
rate)는 세율 변경을 4개 산개 지점에 전파해야 하는 과제로, M10 전 암 공통
1~3/10. 초안 메커니즘은 "편집 성공 시 대체된 옛 텍스트를 프로젝트 전역
재검색해 잔존 지점 알림"(푸터 주입)이었다.

**반전 1 — 픽스처 실물이 푸터를 반증**: uv의 4개 지점은 세율 표기가 전부
다르다 — `DEFAULT_VAT_PERCENT: u32 = 10` / `* 1.10` / `* 110 / 100` /
`* 10 / 100` (산포 함정 설계 의도, 지점별 독립 실패 요구는 M8 스펙 리뷰
산물). 어떤 편집의 옛 텍스트도 다른 지점과 텍스트 매치되지 않아 푸터는
발동 자체가 안 된다. 표기가 달라도 "같은 값"임을 아는 유일한 오라클은
`cargo test`뿐이고(지점별 독립 테스트가 실패를 열거), 그것은 모델이 직접
실행해야 얻는 정보다 — 하네스가 잔존 지점을 대신 알아낼 방법은 없다.

**반전 2 — M10 승자 배치(스탬프 152633Z) uv 10런 정독이 접지 대상을
바꿨다**: 실패는 세 덩어리로 갈린다.

| 유형 | 런 | 양상 |
|---|---|---|
| 조기 finish (스코프 상실) | 0, 5 | 1~4턴 만에 "구조 요약"·"'10' 출현 목록"을 결과물로 finish — 뮤테이션 0회 |
| 탐색 루프 | 1, 2, 7 | 성공 뮤테이션 0회로 max_turns — read/grep 위주(런 7은 실패 edit 시도 2회 포함, 이후 읽기로 회귀) |
| 편집기 턴 소진 | 3, 4, 6, 8 | 편집은 하나 "다 했는지" 몰라 새 grep 패턴을 계속 시도, 턴 고갈 |
| 성공 | 9 | grep → 편집 → `cargo test`(exit 101 목격) → 수정 → `cargo test`(exit 0) → finish, 15턴 |

10런 중 진짜 `cargo test`를 실행한 것은 성공 런 9 하나뿐. run-3은
`cargo test 2>&1 | tail -50`으로 실행해 **파이프라인의 exit code가 tail의
0으로 보고**됐다 — 모델도, "exit code: 0"을 파싱하는 finish_nudge 무장
조건도 함께 속는다.

**결론**: 모델이 잃는 상태는 "남은 지점 목록"(하네스가 알 수 없음)이 아니라
**자기 진행 상태**(하네스가 이미 정확히 앎)다 — ① 뭘 고쳤나(성공 뮤테이션
경로) ② 검증했나(뮤테이션 후 run_command 여부·exit code) ③ 턴이 얼마나
남았나. 기함을 **진행 상태 접지(progress-state grounding)**로 개명하고,
하네스가 이 세 사실을 조건부로 주입한다(§4). 전파 완주는 간접 경로로
다룬다: "verification: none …" 접지가 `cargo test` 실행을 유도하고, 실패
테스트가 남은 지점을 열거한다 — 유일한 통과 런 9의 패턴을 유도(affordance)
로 재현하는 구조다. 강요(차단)가 아닌 유도인 것은 M10 암② 풍선효과의 실측
교훈을 따른다.

## 4. 개입 — 상태선 주입 (STATUS_NOTE)

하네스가 이미 추적하는 진행 상태를 조건부로 툴 결과에 접지한다. 유도형 —
차단·거부 없음, 모델은 무시할 자유가 있다.

- **채널(핀)**: 기존 교정문과 동일한 `note` 채널(`merge_note` →
  `session.push_tool_result(tool, args, body, note)`) — `body`는 불변이므로
  ⑴ 반복 해시(`tracker.record`가 body만 해시 — mod.rs track_and_note)와
  자동 격리: 동일 반복 호출은 상태선이 있어도 RepetitionStop에 도달한다
  ⑵ 오류 스트릭 키(body 첫 문장)·finish_nudge의 exit 파싱(body 첫 줄)도
  무영향. 병합 순서: 기존 노트(salvage·교정·니지) **뒤**에 마지막으로.
  RepetitionStop 턴(stop=true)에는 주입하지 않는다(기존 우선순위 관례).
- **상태(run() 지역 — tracker·finish_nudge와 동일 수명)**:
  - `mutated_paths: Vec<String>` — 성공 뮤테이션(edit_file/write_file
    디스패치 Ok)의 `path` 인자. 렉시컬 정규화(`Path::components` 정리,
    파일시스템 조회 없음 — M10 스펙 §4가 규정했으나 main에 구현 없음,
    포팅 출처는 비병합 `m10/arm-block:src/agent/sr_block.rs`의 정규화
    함수·테스트), 중복 제거·삽입 순서 유지. `path` 부재/비문자열이면
    수집 생략(BadArgs는 애초에 실패)
  - `last_cmd_exit: Option<String>` — 마지막 run_command Ok의 body 첫 줄
    에서 `exit code: ` **접두 파싱**으로 추출한 나머지(finish_nudge의
    `== "exit code: 0"` 동등 비교와 달리 N 추출이 필요; 시그널 종료의
    `(terminated by signal)`도 그대로 실린다 — 사실 서술이므로 허용).
    **덮어쓰기 핀(스테일 방지)**: exit 줄이 없는 run_command Ok(타임
    아웃·취소 본문)는 last_cmd_exit를 **None으로 덮어쓴다** — 이전 성공
    명령의 `0`이 남아 "last command exited 0"으로 거짓 접지되는 것을
    막는다(폴백 문구 `gave no exit code`가 이 상태의 표시)
  - `mutated_since_verify`(기존, VERIFY_NUDGE)·`turns`(기존) 재사용
- **본문(영문, 경로 외 코드 유사 텍스트 금지 — M5 "줄번호 접두" 교훈:
  모델이 다음 search에 복사할 재료를 주지 않는다)**:
  - 뮤테이션 0회: `[status] files edited: none yet | turns: {t} of {max} used`
  - 뮤테이션 ≥1·검증 미실행(`mutated_since_verify == true`):
    ```
    [status] files edited: {k} ({p1}, {p2}, ... and {n} more)
             verification: none since your last edit
             turns: {t} of {max} used
    ```
    경로 목록은 5개 캡 + `and {n} more`
  - 뮤테이션 ≥1·검증 실행됨: `verification:` 줄을
    `verification: last command exited {N}`으로 (last_cmd_exit 없으면
    `verification: last command gave no exit code`). 의미론 주의: 이
    줄은 VERIFY_NUDGE와 같은 "모든 run_command 인정" 기준을 상속한다 —
    하네스는 어느 명령이 진짜 검증인지 알 수 없으므로 문구도 "last
    command"라는 사실 서술로 한정(테스트 통과를 주장하지 않는다). §2
    기준 2 ②의 지표는 이보다 강한 cargo 기준(지표 측 강화)
- **발동(턴당 최대 1회 — 아래 어느 하나라도 참이면 주입)**:
  1. 이번 턴이 성공 뮤테이션 디스패치 (목록 즉시 접지 — 편집기 런 겨냥)
  2. `mutated_paths`가 빈 상태로 turns가 5·10·15·20 도달 (탐색 루프 런
     1·2·7 겨냥 — "files edited: none yet"이 착수 압력)
  3. turns 15·20 도달 (무조건 — 소진 런의 페이싱)
  "턴"은 기존 `turns` 카운터 값. 현행 증가 의미론(실측): length 턴·
  summary 없는 finish 턴·게이트 거부 턴·VERIFY_NUDGE 반려 턴은 증가,
  **파싱 재시도는 턴 내부 루프라 비증가**(mod.rs — "max_turns에 계상
  안 됨"). 도달 판정 시점(증가 전/후)은 플랜에서 핀 고정한다. **이월
  핀**: 조건 2·3의 임계 도달 턴이 note 채널이 없는 턴(finish 오류·
  VERIFY_NUDGE 반려·**length 턴** — 모두 `push_tool_result`가 아닌
  `session.push` 경로)이면 그 턴에는 주입하지 못한다 — pending
  플래그로 이월해 **다음 툴 결과 턴에 1회** 주입한다(턴당 1회 캡
  유지; 세 경로 모두 이월로 통일, 리뷰 2R M-4′). 게이트 거부 턴은
  뮤테이션이 아니므로 조건 1 비발동(조건 2·3은 발동 가능 — 거부 결과도
  `push_tool_result` 경로의 툴 결과 턴이다).
- **관측성**: note는 트랜스크립트에 별도 `kind:"user"` 이벤트로 기록된다
  (session.rs 실측) — exp_metrics(§6)가 `[status]` 발동 수·발동 후 행동
  전환을 집계한다. REPL Notice는 내지 않는다(교정이 아닌 상시 접지 —
  소음 방지).
- **예산과 누적 방지(리뷰 1R I-1 반영)**: pack()의 본문 생략 단계는
  `</tool_result>` **뒤 접미(=병합된 노트)를 명시적으로 보존**한다
  (session.rs — 교정 노트 보존이 목적). 즉 옛 상태선은 축약으로 회수되지
  않고 쌍 제거 단계에서만 사라진다 — 방치하면 최악 25회 × ~35토큰 ≈
  850토큰이 회수 불가로 누적되고, `turns: 5 of 25`와 `turns: 20 of 25`가
  문맥에 공존하는 모순 접지가 생긴다. 설계: **최신만 유지** — 새 상태선
  주입 시 저장 히스토리에서 직전 `[status]` 블록을 제거한다(마커로
  구분되는 접미 블록만 제거, 같은 메시지에 병합된 교정 노트는 보존;
  pack()과 같은 제자리 변형이며 트랜스크립트는 원본 무손실 — 관측성
  §6은 트랜스크립트 기준이라 무영향). 문맥에는 항상 상태선이 최대
  1개 → 상시 비용 ~35토큰. **블록 경계 핀(리뷰 2R)**: 제거 대상 탐색은
  메시지의 **마지막 `</tool_result>` 이후**로 한정한다(툴 body가
  `[status]`를 포함할 수 있다 — loco 자기 소스 grep 도그푸딩 실재) —
  블록 범위는 마커 줄 + 들여쓴 연속 줄의 구조로 판정하며, MaxTurns 후
  REPL 후속 요청이 같은 메시지 꼬리에 병합된 경우(session.rs 실측)
  블록 **뒤**의 병합 텍스트를 보존해야 한다(truncate-to-end 금지).

## 5. 부속 — 파이프 exit code 안내

`run_command` 결과 본문(tools/run_command.rs) 조립 시, 명령 문자열에
**따옴표 밖** `|`가 있으면 끝에 한 줄을 덧붙인다:

> `note: this command is a pipeline - the exit code reflects only the last command in the pipe`

- 따옴표 인지 스캐너(single/double quote 상태 추적, ~20줄) — 모델들이
  grep 패턴 내 `\|`를 상용하므로(4개 배치 run_command 130건 실측: 실제
  파이프 52건, 따옴표 안 `\|` 포함 6건 — 그중 실파이프 없는 순수 오발
  대상 2건) naive `contains('|')`는 grep의 상당수에 오발한다. `||`(OR 연산자)는 파이프가 아니므로 스킵(실측 0건이나
  스캐너 한 줄로 족하다). 이스케이프 엣지 케이스의 잔여 오발은 허용
  오차(정보 한 줄, 무해). 노트는 **Done 분기에만** 붙인다 —
  TimedOut/Cancelled 본문은 첫 줄이 exit code가 아니라서 노트의 전제
  자체가 없다.
- body에 들어가므로 반복 해시에 포함되지만 결정론적(같은 명령 → 같은
  노트)이라 반복 탐지 무영향. 첫 줄(`exit code: N`) 불변 —
  finish_nudge·VERIFY_NUDGE 파싱 무영향.
- pipefail 강제는 하지 않는다(§2 비목표): POSIX sh에 pipefail이 없고
  cmd /C와의 대칭도 깨진다. 진실을 강제하는 대신 위장을 **알려준다**.
