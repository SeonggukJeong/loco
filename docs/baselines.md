## 측정의 한계 (해석 시 주의)

1. **채점이 의도적으로 관대하다**: outcome과 무관하게 check를 실행하므로 "작업은 됐지만 finish를 못 부른" 실행도 통과다. qwen의 통과 12건 중 6건이 이 경우(MaxTurns 3, RepetitionStop 3). "Finished만 인정" 기준으로 환산하면 qwen 16.7%(6/36), gemma 11.1%(4/36 — 전부 Finished)로 격차가 3배→1.5배로 줄어든다. 종료 규율을 따로 보려면 outcome 분포를 함께 읽을 것.
2. **행동 기반 판정**: `cargo test` 종료 코드가 유일한 기준 — 수정의 최소성·품질·"진짜 리네임인지(별칭 추가로도 통과 가능)"는 측정하지 않는다.
3. **모델+스캐폴딩 시스템 측정**: 순수 모델 능력이 아니다. 예: edit-crlf-file 통과에는 loco의 EOL 자동 보존(M3)이 기여. 스캐폴딩을 바꾸면 같은 모델도 수치가 변한다 — 그게 이 하네스의 목적이다.
4. **알려진 판정 우회(이론)**: protected가 `tests`/`Cargo.toml`만 커버 — `.cargo/config.toml`에 가짜 러너를 쓰면 대부분 과제가 가짜 통과 가능(M5 수정 예정). 협조적 4B 모델에선 실위험 근접 0이고, 실행별 트랜스크립트로 감사 가능. 이번 통과 건들은 해당 없음 확인.
5. **표본 3회, 신뢰구간 없음**: 과제당 1건 차이가 ±33%p. 시드 3개(0-2) 고정이라 재현은 되지만 분산 추정은 안 된다. 모델 간 비교는 전체 통과율(n=36)로만.
6. **환경 의존**: report.json이 유효 config(`max_output_tokens=4096` 등 로컬 오버라이드)를 스냅샷하지 않는다(M5 항목) — 다른 머신·설정과의 수치 비교는 이 문서의 측정 조건을 수동 대조할 것. `command_timeout_secs`(60s)는 `--timeout-scale`의 영향을 받지 않는다(스펙 의도).
7. **판정기 자체의 협소함**: find-definition은 정답 형식이 좁고(후행 슬래시·따옴표 불인정), fix-off-by-one의 zero 테스트는 버그 상태에서도 통과하는 비변별 케이스(3중 2개만 변별). 과제 개편은 기준선 무효화를 수반하므로 M5에서 일괄 검토.

### 행동 지표 추출 레시피

```python
import json, sys, glob, os
# usage: python3 m9_metrics.py .loco/eval/<stamp>
MARKS = {
    "sr_error": "search and replace are identical",
    "sr_correction": "Write the MODIFIED code in `replace`",
    "finish_missing": "finish requires a string `summary`",
    "finish_args_corr": "Do not call finish with empty args again",
    "finish_nudge": "do not re-verify what you have already confirmed",
    "repeat_corr": "repeating the same tool call",
}
for path in sorted(glob.glob(os.path.join(sys.argv[1], "run-*.jsonl"))):
    counts = dict.fromkeys(MARKS, 0)
    with open(path) as f:
        for line in f:
            e = json.loads(line)
            if e.get("kind") == "assistant":
                continue  # 모델이 인용한 문구는 세지 않는다
            c = e.get("content", "") or ""
            for k, m in MARKS.items():
                counts[k] += c.count(m)
    print(os.path.basename(path), counts)
```

## M12 — 지표 비교 주의 각주 (정직한 하네스, 2026-07-18)

M12(스펙 `2026-07-18-m12-honest-harness-design.md`, 커밋 fc560f8)는 새 효과
입증 실험(다암 비교·32K 측정)을 만들지 않는다(스코핑 결정 §0-3 — 회귀
게이트로 축소). **경량 사전등록 회귀 게이트 측정 자체는 수행했다** — 결과는
아래 "M12 실험 — 회귀 게이트" 절 참고. 아래 세 각주는 M12 이후 배치를 M11
이전 배치와 비교할 때 반드시 참조할 것.

- **sr_error 비교 주의:** T1(`tools/edit_file.rs`)이 `search==replace` 동일성
  검사를 `apply_edit`(매치 확정) **뒤로** 옮겼다. 그 결과 M12 이후 배치의
  `sr_error`/`sr_correction`(그리고 `exp_metrics.py`의 동명 컬럼)은 "매치가
  실재하는 진짜 S/R"만 센다 — 이전에는 이 카운터에 섞여 있던 두 오형이 각각
  분리된 오류로 이동했다: 파일에 없는 환각 코드(0매치)는 "not found"로,
  `replace_all` 없이 2곳 이상 매치하는 모호한 S/R은 "N locations"로. 즉 M12
  이후 배치의 `sr_error` 수는 M11 이전 배치보다 구조적으로 **낮게** 나올 수
  있다(오류 종류가 재분류된 것이지 개입 효과가 아니다) — 직접 비교하지 말 것.
- **normalize 표기 영향:** `agent/status_note.rs::normalize`가 M12 전
  `Path::new("/src/a.rs").components()`의 `RootDir`를 빈 파츠로 흘려 보내
  `join("/")`가 `"//src/a.rs"`(슬래시 중복)를 만드는 버그가 있었다(T4가 수선,
  `absolute` 플래그로 `"/"` 1개만 합성). 절대경로를 인자로 받는 편집 후에는
  상태선의 파일 목록 표기가 M12 이후 `//src/a.rs` → `/src/a.rs`로 바뀐다 —
  화면 표기만의 변화이며, `exp_metrics.py`의 `sr_files` 컬럼은 `os.path.basename`
  기준이라 이 표기 변화에 무관(지표 자체는 영향 없음).
- **sr_corr_total 반사실(counterfactual) 각주:** (a) `exp_metrics.py`의
  `sr_corr_total`을 pre-M12 배치(34개, 747행)에 돌린 값은 **관측이 아니라
  반사실 재현**이다 — 위 `sr_error` 각주와 성격이 다르다: `sr_error`는 같은
  궤적을 다르게 재분류하는 것(그 런은 실제로 일어났고, 오류를 어느 통에
  담을지만 바뀐다)인 반면, `sr_corr_total`은 **그 런들이 실제로 밟지 않은
  궤적**을 가정한다 — 그 히스토리 런들은 파일별 SR_CORRECTION(M12 §4-1)을
  한 번도 받지 못했으므로, 재현이 "발화했을 것"이라 표시하는 지점에서
  실제 루프가 같은 방식으로 계속됐으리라는 보장이 없다. 편향의 방향은
  뚜렷하다: 교정 피드백이 없었던 만큼 히스토리 루프는 더 오래 돌며 파일별
  히트를 더 쌓았을 것이므로 **55는 과대추정 쪽으로 치우친 상한 추정치**다.
  따라서 이 컬럼은 (b) M12 비교의 **베이스라인 축(arm)이 아니다** — 55는
  실현가능성/상한 추정일 뿐, "M12 코드가 동등한 작업량에서 낼 값"의 기준선이
  아니다. (c) 규칙과 실제 궤적이 일치하는 **M12 이후 배치에서만 충실
  (faithful)**하다. (d) 실패 모드가 뒤집혀 있다는 점도 주의: M12 이후 배치의
  파일별 발화 수가 pre-M12 반사실값보다 *더 적게* 나오면 "확대 트리거가
  잘 안 걸린다"로 오독하기 쉽지만, 실제로는 "교정이 제대로 작동해 루프를
  일찍 끊었다"는 정반대 결론일 수 있다 — 이 마일스톤 자체의 워치독 지표에서
  나타나는 역설이므로, M12 이후 `sr_corr_total`의 증감은 단독으로 해석하지
  말고 반드시 `sr_recovered`/분모(회복률)와 stop_cause(sr 정지 비율)를
  함께 봐야 "트리거 미달"과 "루프 단축"을 구분할 수 있다.

### ★ 합성 평가 통과율 ↔ 실사용 채택률의 괴리

스펙 §7 기준 4는 *"채택률이 낮게 나오는 것도 결과이며, 그 경우 '합성 평가
통과율과 실사용 채택률의 괴리'가 M14의 가장 중요한 입력이 된다"*로 이 사안을
**사전에 등록**했다. 채택률은 낮게 나왔으므로 조건이 발동했다.

**총합끼리의 대조는 통제되지 않는다** — 과제 구성·채점자(자동 `check` vs 사용자
판정)·반복수(3회 vs 1회)가 전부 다르다. 따라서 인과 진술이 아니라 **타당도
진술**로만 쓴다: *"게이트 35/36이 실사용에 대해 무엇을 말해 주는가"*의 답이 좁다는 것.

**유형을 맞추면 훨씬 선명하다.** 파일럿에서 과제문이 파일 경로를 지정하지 않아
모델이 어디를 볼지 스스로 골라야 했던 과제는 **7세션**이다. `tasks/`에서 같은
형태를 12개 `task.toml` 전건 정독으로 분류하면 **5과제**이고, 그중 심볼명만 주는
(기전조차 안 주는) 것은 **3과제**다:

| `tasks/` 과제 | 형태 | 앵커 | 게이트 |
|---|---|---|---|
| `find-definition` | 심볼만 (*"이 프로젝트에서 `area` 함수가 정의된 파일을 찾아…"*) | 3/3 | 3/3 |

### 8-3. M14 비교가능성 각주 — `verify_allpass`·`verify_total`은 전후 직접 비교 불가

(스펙 §8-3 전문, 원문 그대로 옮김)

**§3-4-2의 규칙 4 → 규칙 5 폴백이 기존 지표의 모집단을 바꾼다**(5R I1).
`exp_metrics.py`의 검증 줄 분포는 상태선 문자열을 **리터럴로 매칭**한다
(`scripts/exp_metrics.py:33~35`) — `verify_total` = `"verification: last cargo test: "`,
`verify_allpass` = `"verification: last cargo test: all "`. 폴백은 파이프 + allpass
렌더를 규칙 5 문자열(`"verification: last command exited N"`)로 보내므로 **어느
패턴에도 안 걸린다.**

> **모델이 파이프를 쓰는 만큼 `verify_allpass`와 `verify_total`이 함께 내려간다.
> 하락은 회귀가 아니라 폴백이 작동한 증거일 수 있다.**

§8의 게이트는 M14 이전 배치(`20260719T093254Z`)와 대조하므로, 두 배치의
`verify_allpass`를 나란히 놓고 "검증이 줄었다"로 읽으면 **오독**이다.

**파생 `verify_failed`는 보존된다** — `verify_total − verify_allpass`이고 규칙 2는
폴백의 영향을 안 받으며 두 원지표가 같은 양만큼 줄기 때문이다(5R 실측 확인).
깨지는 것은 두 원지표뿐이다.

**선례**: M12 `sr_error`(검사 순서 변경으로 발동 조건 이동 — CLAUDE.md에 한 문단),
M13 T7 `verify_*`(무뮤테이션 분기 렌더로 구조적 **상향** — `docs/baselines.md` 각주).
**M14는 같은 지표를 반대 방향으로 이동시킨다.**

### 관측 항목(사전등록 4종, 판정 아님) — 답

1. **A-1 파이프 해제 차단**: `pipe_unreleased`·`verify_nudge_pipe` 양
   배치 0. 두 배치의 모든 `run_command` 호출(M14 54건·대조 58건)을 전수
   추출한 결과 `|` 포함 명령이 **0건**이었다(고유 명령 집합에 `&&`/`2>&1`
   뿐, 파이프 없음). 배선·단위 테스트 5종은 소스에서 확인 —
   **트리거 조건 미발생. "불필요"도 "고장"도 아니다**(확신도 높음; M13
   `ARGS_TOOL_SWITCH_NOTE` 20세션 0회와 동형 — 효과는 `tasks/`로
   실증되지 않는다고 사전등록이 미리 인정했다). 이 배치에서 신뢰 범위를
   가진 유일한 신규 장치는 A-3뿐이다.
2. **FINISH_NUDGE 발동**: `finish_nudge_total` 양 배치 0. 무장 조건은
   양 배치 36/36 런에서 충족됐으나 27~28개 런이 무장 직후 바로 `finish`를
   호출해 K=4 반복 창이 형성되지 않는다. **대조군도 0이라 이 지표는

   되지 않았다가 아니다). 그 회귀는 §7 기준 6의 단위 테스트가 전담한다.
3. **A-3 diff 첨부·절단**: 첨부 56회(`edit_file` 44 + `write_file` 12,

   "REPEAT_CORRECTION 직후 파이프 거부" 두 watch 항목은 트리거 대상이
   없어 정의상 0).

### 법의학 — `sr_error` 31→35 상승은 M14에 귀속되지 않는다 (확신도 높음, 직접 반증)

우려한 가설(Task 6의 diff 렌더 본문을 모델이 다음 `search`로 복사한다)은
**반증되었다.** 실행별 델타는 양방향(+9/−5, 순 +4)이고, 회복 실패 증분
4건은 서로 다른 4개 사건이 아니라 `rename-function-2` **한 실행 안에서
동일 호출이 4회 반복**된 것(실효 표본 n=1). 그 실패 `search` 문자열
(`format!("total: {}", total_price(items))`, search==replace 축퇴)의
출처는 loco가 렌더한 diff가 아니라 **rustc 진단**(양 배치 동일 텍스트)
이었고, 배치 전수 스캔(M14 실패 45건·대조 37건)에서 diff 마커(`-`/`+`/`@@`)를
포함한 `search`는 **0건**이었다. 결정적으로, **바이트 단위로 동일한
실패**가 diff 장치가 없는 **대조군에도** 존재한다(`run-rename-function-1`
rec 15 = M14 `run-rename-function-2` rec 21). 두 실행의 차이는 실패
발생이 아니라 회복 경로(대조: `read_file` 2회로 2트라이 회복 / M14:
4회 반복 후 SR_CORRECTION·REPEAT_CORRECTION 주입 → `write_file` 전환으로
성공)뿐이다. "결과 본문을 `search`로 복사"하는 패턴은 오히려 **대조군
1회 vs M14 0회**로 가설과 반대 방향이다. **판정하지 못한 것**: 약한
프라이밍 가설(diff 노출이 축퇴 `search` 확률을 형식만으로 밀어올릴 수
있는가)은 n=1·반복 없음에서 원리적으로 판정 불가능하다.

기타 이동은 전부 양방향 churn이거나(`length_retry`, `cargo_after_mut` —
둘 다 중간 확신도, n=1) 신규 한정자의 정상 발화(`status_no_summary` —
4건 전수 확인, 전부 exit-101 컴파일 실패라 libtest 요약이 애초에 없는
참인 사례, 오탐 0, 확신도 높음)로 귀속됐다. `args_tool_switch` 1→2는
M13이 20세션에서 "미관측"이라 기록한 규칙의 실발화다 — 법의학이 분석한
2개 배치(대조 `20260719T093254Z` 1회 + 이 배치 2회 = 3회)에 더해
`20260718T222824Z`(M12 게이트 배치, 이 법의학의 별도 분석 대상 아님)의
1회를 합치면 세 배치 총 4회이며, "불필요"가 아니었음을 갱신한다.

전 항목 **팔당 n=1, 반복 없음, 사후 분석.** 전체 분석: `docs/experiments/2026-07-20-honest-verification-ii/report.md`,
법의학 원본 `.superpowers/sdd/m14-forensics.md`.
