## 1. 최우선 가설 — cold start 온보딩 부재

### 1-1. [설계] 레포 온보딩 하네스: 탐색 → 요약 컨텍스트 → 작업

**문제 진술**

`tasks-real` 프롬프트 = **이슈 본문 only** (M15 스펙: 손대지 않음). 모델은 낯선
실레포에서 항해·이슈↔코드 매핑·패치·검증을 **한 세션·이슈 텍스트만**으로 수행한다.

M15 부분 관측과 정합:

| 관측 | 해석 |
|---|---|
| 통과 ≈ 0 | 가혹 조건 바닥 — 베이스라인으로서 유효 |
| zero-mut 다수 | 수선 단계 진입 전 턴/시간 소진 |
| nav 후에도 fix_hit 낮음 | “파일은 보는데 작업 기억으로 못 고정” |
| pack↑ & zero-mut (일부 rg) | 컨텍스트는 차는데 구조화 메모가 없음 |
| 1× finished = 산문 설명 | 과제 유형 오인 + 온보딩 없이 추측 finish |
| M13 경로 미지정 0/7 | 동일 축의 실사용 증거 |

**제품 직관** (큰 모델 포함): 처음 보는 레포는 **우선 탐색 → 요약된 레포 메모 → 작업**.
지금 평가는 그 1단계를 허용·강제하지 않는다. 그건 UX 결함이지 “모델만의 실패”가
전부가 아니다.

**M15 베이스라인과의 관계**

- M15 0점은 **온보딩 없는 바닥 좌표**로 남긴다 (폐기하지 않음).
- M16은 그 좌표 **위**에 온보딩 장치를 올려 *들어 올리는지*를 잰다.
- 이슈 본문을 사람이 풀어 쓰는 방식은 **금지** (항해 누설·변별력 붕괴).

## 2. Harness 형태 후보 (비교)

목표가 “모델이 레포를 이해하게 돕는다”일 때, **누가 요약을 쓰는지**와
**언제 쓰는지**로 가른다.

### 형태 A — 프롬프트 유도만 (최소)

| | |
|---|---|
| **무엇** | SYSTEM/첫 user에 “먼저 탐색하고 메모한 뒤 수정” 문구 |
| **구현** | 문자열만. 툴·상태기계 없음 |
| **장점** | 1 PR, 회귀 위험 최소 |
| **단점** | 9B가 무시·조기 edit 가능. **강제력 0** — M15가 보여 준 실패 모드를 구조적으로 안 막음 |
| **측정** | first_mut_turn 지연·notes 유무를 트랜스크립트 휴리스틱으로만 |
| **성숙도** | [티켓] 가능하나 **효과 기대 낮음** — A/B의 sham arm 또는 1단 스모크용 |

### 형태 B — 전용 메모 툴 + 작업 게이트 (권장 논의 중심)

| | |
|---|---|
| **무엇** | 툴 예: `update_repo_notes` / `repo_notes` (append 또는 replace). 본문은 모델이 씀 |
| **게이트** | 성공 mutation(`edit_file`/`write_file` on non-notes) 전 **notes가 비어 있지 않을 것** — 1회 거부 노트 후 재시도 (VERIFY_NUDGE류 once-latch) |
| **노출** | notes 전문 또는 말미 N줄을 상태선/`[repo_notes]` 블록으로 매 턴 접지 (M11 status 패턴) |
| **pack** | notes 블록은 **시스템 인접 고정** 또는 elide 최하위 우선순위 (과제 메시지 보호와 유사 논의 필요) |
| **장점** | 강제력 있음. 트랜스크립트에 notes 이벤트 → 계측 용이. 제품 루프와 동일 형태 |
| **단점** | 잘못된 notes 고착. 게이트가 탐색 루프를 늘려 Timeout↑ 가능 (풍선). 툴 스키마 1개 추가 |
| **성숙도** | **[설계]** — 게이트 조건·notes 길이 캡·보호 경로가 미정 |

**게이트 변형 (B1–B3)**

| 변형 | 조건 | 위험 |
|---|---|---|
| B1 soft | 문구만 “notes 권장”, 거부 없음 | A와 동일 약함 |
| B2 hard | mut 전 notes 필수 | 탐색 턴 강제 소비 |
| B3 phased | 턴 1..K 탐색 전용( mut 툴 비활성 ), K 이후 mut 허용 ∧ notes 필수 | 가장 명확, max_turns 예산 재설계 필요 |

### 형태 C — 하네스 주도 온보딩 (모델 부담↓)

| | |
|---|---|
| **무엇** | 세션 시작 시 하네스가 `list_files` / 루트 README / `Cargo.toml` 등을 **자동 실행**하고 요약을 user 또는 system 접미사로 주입 |
| **장점** | 9B 탐색 품질에 덜 의존. 첫 턴부터 지도 있음 |
| **단점** | 주입 토큰 고정비 (32K 예산). 요약 알고리즘 설계 = 사실상 새 제품. **이슈별 스포일러 금지** 규칙이 까다로움 |
| **성숙도** | [설계]–[질문] — “무엇을 자동으로 넣나”가 열린 문제 |

### 형태 D — 평가 전용 사전 아티팩트 (레포 단위 메모)

| | |
|---|---|
| **무엇** | `tasks-real`과 별도로, 레포당 한 번 사람 또는 강모델이 쓴 `REPO_MAP.md`를 샌드박스에 넣고 프롬프트가 “이 파일을 먼저 읽어라”고 지시 |
| **장점** | 실험 암 구성이 쉬움. 에이전트 코드 최소 |
| **단점** | **제품 경로와 괴리**. 메모 품질·누설(정답 파일 언급) 감사 비용. M15 표본과 공정 비교 시 메모 유무가 교락 |
| **성숙도** | [설계] — 측정 암으로만 쓰고 제품 기본값으로 올리지 말 것 |

### 형태 E — 외부 인덱서 / RAG

| | |
|---|---|
| **무엇** | 심볼·파일 임베딩 검색 툴 |
| **장점** | 대형 레포 항해 |
| **단점** | 의존성·인덱스 생애주기·폐쇄망. 스펙 의존성 동결 규율과 충돌 가능. **M16 1차 범위 과대** |
| **성숙도** | [질문] — M16 본선 비권고, 백로그 |

### 한눈에 비교

| 형태 | 강제력 | 구현비 | 누설 위험 | 제품 정합 | M16 1차 후보 |

## 3. 형태 B를 고를 때 못박을 설계 질문

### 3-1. notes에 **넣어도 되는 것** / **넣으면 안 되는 것**

| 허용 (레포 지도) | 금지 (과제 누설) |
|---|---|
| crate/모듈 트리, 바이너리 진입점 | “이 이슈의 정답은 `foo.rs`” |
| 테스트 레이아웃 (`tests/`, integration) | 실패 테스트 이름만 박아 항해 단축 (check 지목과 동일 축) |
| 코딩 관례 (error 타입, clap 사용) | solution diff 요약 |
| “CLI 플래그는 보통 `src/app.rs`/`args.rs`” 수준의 **관례** | 오라클 파일 목록 복사 |

평가 시: notes 본문 ∩ `oracle_files` 비공집합이면 **누설 런으로 라벨** (탈락이 아니라 분석 층) — 정책은 사전등록에서 확정.

### 3-2. 길이·갱신·pack

- 캡: 예) 2–4KB 또는 N줄. 초과 시 tail keep / 모델에 truncate 통지.
- 갱신: replace vs append. append는 모순 누적, replace는 실수 시 지도 소실.
- pack: notes를 elide하면 온보딩이 증발 → **과제 메시지와 동급 보호** 검토 (M13 pack-과제삭제 교훈).

### 3-3. 턴 예산

M15 `max_turns=25`·`timeout_secs=600`에서 온보딩 K턴을 빼면 수선 예산이 더 줄 수 있다.

| 옵션 | 내용 |
|---|---|
| T+ | 실레포만 `max_turns` 상향 (예 40) + 온보딩 K=5–8 |
| split | “온보딩 세션”과 “작업 세션”을 하네스가 이어 붙임 (상태 파일 notes) — 구현 큼 |
| timeout | 600s 재검토 (M15 Timeout 다수) — 온보딩과 **별개 레버**, 한 실험에 섞지 말 것 |

**권고:** 온보딩 실험에서 턴/시간 상향을 쓸 경우 **암 표에 명시**하고, M15 바닥과 비교할 때
“장치 효과 vs 예산 효과”를 섞지 말 것 (2×2 또는 예산 고정 암).

### 3-4. 성공 지표 (사전등록 초안 스케치)

| 역할 | 지표 |
|---|---|
| 주 | `passed` (과제 수준 평균 + 부트스트랩, M15 §6-4-7 승계) |
| 보조 | `passed_strict`, false_finish |
| 기전 | `notes_chars` / notes 갱신 횟수, `first_mut_turn`, nav_hit/fix_hit, zero-mut 비율 |
| 감시 | Timeout·MaxTurns 분율 (풍선), pack_fired, protected_edits |
| 실격 | M15와 동일 대역 또는 “온보딩 암만의 최소 들어 올림” 임계 (예: 통과 ≥1 과제 또는 과제 평균 ≥ ε) — **데이터 전 확정** |

대조: **M15 동결 표본·동일 로드/운용점** 위 암:

1. control = M15와 동일 (이슈 only, 온보딩 없음) — 재측정 또는 M15 스탬프 인용(코드 동일성 조건)
2. treatment = B2/B3

## 7. 세션 논의 요약 (2026-07-21)

- 전패 베이스라인도 **의미 있음** (바닥 좌표). 테스트 전면 결함 가설은 verify·spot으로 약함.
- 제품적으로는 **탐색→요약 문서→작업**이 맞고, 현재 평가는 cold start 가혹 조건.
- 큰 모델도 레포 맵 없이 실레포 cold start는 흔들림 → 장치 정당화는 “소형만의 특혜”가 아님.
- M16 후보는 그 장치를 **하네스로 강제·계측 가능**하게 만드는 것.
