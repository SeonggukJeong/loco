## Commands
- `cargo test` ŌĆö full suite (fast, <1s)
- `cargo clippy --all-targets -- -D warnings` ŌĆö required gate per task; `--all-targets` matters (also lints test code)
- `cargo run` ŌĆö agent REPL (default input runs the tool loop; /chat for plain streaming chat); `cargo run -- -p "question"` ŌĆö one-shot agent; summary to stdout, progress to stderr, exit codes 0/1/2
- `cargo run -- --auto` (or `-p ... --auto`) ŌĆö auto-approve the confirmation gate (still blocked by `auto_deny_patterns`); without it, mutating actions prompt `ņĀüņÜ®ĒĢĀĻ╣īņÜö? [y/N]` interactively, or are denied outright in `-p` non-interactive mode
- `cargo run -- eval <tasks-dir> [--repeats N] [--seed N] [--timeout-scale F]` ŌĆö eval harness (e.g. `cargo run -- eval tasks/ --repeats 3`); report table to stdout + `./.loco/eval/<stamp>/report.json` (+ per-run transcripts `run-<task>-<repeat>.jsonl`); exit 0 on normal completion regardless of pass rate, 1 on harness errors (server down, bad task defs) or a Ctrl+C partial report. Table now carries a "ņŚäĻ▓®" (strict) column and a "Ļ▒░ņ¦ō ņä▒Ļ│Ą finish" count (M6)
- `cargo run -- eval <tasks-dir> --verify` ŌĆö judge meta-test gate (no LLM/server): per task, `check` must FAIL on the pristine fixture (discriminability) and PASS after overlaying `solution/` (solvability); exit 0 only when all tasks pass both, else 1. Mutually exclusive with `--repeats`/`--seed`; `--timeout-scale` still applies. Run after ANY change to `tasks/` (M6 ┬¦4)
- `cargo run -- eval <tasks-dir> --filter <name>` (repeatable) ŌĆö exact-match task-directory filter for `eval`/`--verify`; each filter value must match Ōēź1 task or the harness errors (exit 1) ŌĆö a typo can't silently shrink a batch (M10)
- `python3 scripts/exp_metrics.py <stamp-dir>...` ŌĆö behavioral-metrics extractor for experiment eval batches (S/R errors, 2-try recovery, block/perturb-trigger counts, finish-missing streaks, stop-cause classification, merges report.json pass/strict/false-finish; M11 adds status/pipe-note marker counts, `first_mut_turn`/`cargo_after_mut`, a zero-mut outcome breakdown in the summary, S/R file attribution `sr_files`, and `status_in_args` copy-contamination detection ŌĆö the "turn" in `first_mut_turn` is the tool_result 1-based ordinal, not the agent's turns counter; M12 adds `empty_test_note`, the status-line verification-rule distribution `verify_zero`/`verify_allpass`/`verify_failed` (`verify_failed` derived as `verify_total - verify_allpass`, `verify_total` itself not a distinct signal ŌĆö it matches both the rule-2 and rule-4 renders), `sr_corr_total` (SR_CORRECTION firings ŌĆö numerically identical to the pre-existing `sr_correction` marker since per-file-vs-consecutive attribution isn't recoverable from the transcript alone), and `perturb_turns_ext` (temperature-perturbation reach under the M12-widened trigger ŌĆö a from-scratch state-machine replay of `sr_by_file`/`badargs_streak`, same necessity as the pre-existing `perturb_turns`: the temperature override itself never appears in transcript text); `--selftest` self-checks against a built-in fixture transcript ŌĆö stdlib only, its own gate outside `cargo` (M10)
- Session transcripts: one JSONL file per run at `./.loco/sessions/*.jsonl` (best-effort; `.loco/.gitignore` keeps them out of commits)
- Tunable config (defaults): `max_turns`=25, `max_output_tokens`=2048, `context_tokens`=8192, `command_timeout_secs`=60, `auto_deny_patterns`; unknown keys rejected (`deny_unknown_fields`)
- Server-down smoke: temp dir with `.loco/config.toml` pointing `base_url` at `http://127.0.0.1:1/v1` (port 1 is reliably closed; ephemeral ports may be occupied)

## Hard constraints (from spec)
- Edition 2024. Dependency list is fixed by the spec ŌĆö ask the user before adding any crate
- reqwest stays `default-features = false, features = ["json", "stream", "rustls-no-provider"]`; TLS crypto provider is rustls+ring (direct dep, user-approved) ŌĆö no OpenSSL and no aws-lc-sys in the graph (Windows offline builds need no cmake/NASM). `main()` installs the ring provider at startup
- Network calls only to the configured endpoint; the HTTP client uses `.no_proxy()` (corporate proxies must not capture localhost LLM traffic)
- User-facing CLI messages in Korean; identifiers and SYSTEM_PROMPT in English
- Errors: `thiserror` in `llm` module, `anyhow` at app level
- Conventional commits (subjects may be Korean)

## Architecture
- lib + thin bin: modules declared in `src/lib.rs` (`config`, `llm`, `ui`); `src/main.rs` is wiring only
- config: layered TOML ŌĆö global config_dir then `./.loco/config.toml`, later file wins, `deny_unknown_fields` rejects typos
- llm: hand-rolled SSE line parser (`llm/sse.rs`); chat retry = 3 total attempts with 200/400ms backoff, 5xx retried, 4xx immediate; `list_models` deliberately has NO retry (startup check, fail fast ŌĆö spec ┬¦9 exception)
- REPL history: index 0 is the system prompt (`/clear` truncates to 1); failed or empty streams pop the user turn so it can be retyped
- tools: `Registry::guided()` adds write_file/edit_file/run_command to the M2 three (6-tool set; `finish` stays loop-handled, not in the registry); `path::confine_for_write` is the write variant of `confine` ŌĆö target need not exist, but the deepest *existing* ancestor is canonicalized and root-checked (blocks escape via a symlinked ancestor). `edit_file`'s `search` match is a 3-stage ladder (exact ŌåÆ ignore-trailing-whitespace ŌåÆ indent-shift); 0 matches errors with the closest actual text quoted (first-line substring then char-bigram fallback, Ōēż10 lines), Ōēź2 matches errors listing up to 5 locations unless `replace_all: true` (non-overlapping greedy at stage 2, per-location indent at stage 3); success returns ┬▒3 lines of post-edit context (line numbers in the header only ŌĆö models copy body prefixes into their next `search`), and search==replace is an error; write_file/edit_file preserve the file's dominant EOL style (CRLF in, CRLF out, `tools/eol.rs`). Model-facing error feedback (M5): `Registry::dispatch` echoes the tool signature + received keys on serde BadArgs ("missing field"/"invalid type" only); read_file translates directory paths to a list_files hint; grep falls back to literal search on invalid regex (header carries the reason, also on 0 matches). `run_command` runs `sh -c`/`cmd /C` from project root in its own process group so timeout/cancel kills the whole tree (`kill -9 -<pgid>` / `taskkill /T /F`); UTF-8 with CP949 lossy fallback for Korean Windows consoles; output middle-truncated past 8000 bytes ŌĆö the exec mechanics live in `tools/exec.rs`, shared with eval's `check` execution. M11: Done results append one English provenance line when the command contains a real unquoted pipe (`has_unquoted_pipe` ŌĆö quote/escape-aware, `||` is not a pipe) so models don't read a pipeline's exit 0 (last command's) as test success; timeout/cancel paths get no note
- agent: confirmation gate lives inside the turn loop via the `Approver` trait (`TtyApprover`/`AutoApprover`/`NonInteractiveApprover`) ŌĆö fires only for `is_mutating()` tools with a successful `preview()`; denial feeds back to the model as a tool result. Turn parsing is salvage-tolerant (`protocol.rs` builds turns via `serde_json::Value`): fields outside `args` (action-level scalars, `args_2`, top-level non-reserved keys) are normalized into `args` (objects overwrite, scalars insert-if-absent; BTreeMap key order makes it deterministic), `ModelTurn.salvaged` triggers a one-line correction note merged into the tool result ŌĆö this drove gemma's M5 jump (285 salvages/36 runs). Repetition detection (`agent/repetition.rs`, M5): a `(tool|args, result-hash)` 8-turn window counted AFTER dispatch (denied results and summary-less `finish` included) ŌĆö 3rd identical hit injects one correction, 5th returns `AgentOutcome::RepetitionStop` (exit 2, the tool result is still pushed first); a separate same-error streak (first sentence, 3 consecutive) injects one strategy correction (edit_file/write_file ŌåÆ "rewrite with write_file", else generic). Both corrections latch once per run. M9 scaffolding: edit_file's identical-search/replace rejection now appends a prescription sentence (current code in `search`, changed code in `replace`); a dedicated 2-consecutive S/R streak correction (`SR_CORRECTION`, own latch, `agent/repetition.rs`) handles that error key exclusively ŌĆö the generic 3-streak correction is excluded for it, while the window-based `REPEAT_CORRECTION` still stacks at the 3rd identical call (intended escalation). finish discipline: a missing-summary finish 2-consecutive streak injects `FINISH_ARGS_CORRECTION` once per run (no-action turns preserve the streak; dispatched or gate-denied actions reset it); `agent/finish_nudge.rs` is an event-driven state machine ŌĆö "mutation" = successful edit_file/write_file dispatch, arms on a post-mutation `exit code: 0` run_command, counts armed-state read/grep/list_files and re-verification run_command turns in a K=4 window requiring Ōēź1 repeated call (`RepetitionTracker::seen_key`, queried before record), injects `FINISH_NUDGE` once per run; RepetitionStop takes priority on the same turn. A summary-carrying `finish` after unverified mutations is rejected once per run (`VERIFY_NUDGE`; a `run_command` Ok clears the flag regardless of exit code except an empty test run ŌĆö see M12 below). M10 adds a decoding-layer intervention alongside the M9 text corrections: an S/R streak Ōēź2 (same trigger point as `SR_CORRECTION`) overrides the request's `temperature` to 0.7 until the streak breaks (`SR_PERTURB_TEMPERATURE`, run()-scoped like `tracker`/`finish_nudge` ŌĆö no config key, no latch, re-arms if the streak re-reaches 2) ŌĆö winner of a pre-registered 3-arm experiment against a rejected alternative that blocked `edit_file` on a file after 3 cumulative S/R errors (`m10/arm-block`, preserved unmerged: it eliminated S/R-caused RepetitionStops but transferred the failure to finish discipline, a "balloon effect"); measured numbers with stamps in `docs/baselines.md` "M10 ņŗżĒŚś 1". M11 progress-state grounding: a run()-scoped `StatusNote` (`agent/status_note.rs`, pure state machine) injects a `[status]` line as the LAST note merged into tool results ŌĆö fires immediately on a successful edit_file/write_file dispatch, on a zero-mutation cadence (turns 5/10/15/20), and as unconditional pacing (15/20); renders the edited-file list (lexical dedup, cap 5 + "and N more"), verification state (last run_command exit code, overwritten to a no-exit-line notice so a stale 0 can't ground falsely), and turns used (turn = turns+1, 1-based). `session.remove_status_note()` strips the previous block before each new injection (keep-only-latest ŌĆö pack()'s elision preserves post-`</tool_result>` suffixes, so old notes are not reclaimed by shrinking; a block = the marker line + 9-space-indented continuation lines, searched only in the post-close-tag suffix so fake markers inside tool bodies are untouched, and text after the block such as a merged follow-up user request survives). Channel-less turns (length / summary-less finish / VERIFY_NUDGE rejection) carry a hit threshold over via a one-shot pending flag; RepetitionStop turns are excluded (`!stop` guard). The marker contract ŌĆö `"[status] "` prefix, 9-space continuation indent ŌĆö is shared verbatim by status_note.rs, session.rs and scripts/exp_metrics.py. Tool dispatch runs via `spawn_blocking` (synchronous tools don't block the async runtime) with a shared `cancel: AtomicBool` the REPL flips on Ctrl+C. M12 "honest harness" grounds the verification signal in parsed libtest output instead of exit code alone, and widens two M9/M10 intervention triggers ŌĆö no new experiment batch, gated by the existing `cargo test`/clippy/`eval --verify` gates. A pure `src/test_summary.rs::parse_test_summary` sums every `test result: ŌĆ” P passed; F failed; ŌĆ”; K filtered out ŌĆ”` section of a (possibly multi-crate/workspace) run body ŌĆö line-start-anchored so a catted log can't spoof it, `ran = passed + failed` (ignored excluded), up to 4 failed test names in encounter order ŌĆö and returns `None` when no summary line is present, the conservative fallback every consumer honors. It is parsed exactly once at the run_command dispatch site in `agent/mod.rs` and shared by three consumers: (1) `tools/run_command.rs` appends an English one-line note on an exit-0 body when `ran==0 Ōł¦ filtered_out>0` (`note: 0 tests ran (N filtered out) - cargo test filters by test NAME, not file name; this exit 0 did not verify anything` ŌĆö same append point/timeout-cancel exemption as the M11 pipe note); (2) `StatusNote`'s verification line grows from exit-code-only to 5 rules: rule 1 (`mutated_since_verify`) still wins outright; rule 2 renders up to 2 failed test names + "and N more" whenever `failed>0`, exit-code-independent (catches `cargo test 2>&1 | tail`-style masking); rule 3 renders "ran 0 tests (filter matched nothing)" on the same empty-run condition as the run_command note (different wording, non-overlapping text); rule 4 ("all P passed") additionally cross-checks `last_cmd_exit=="0"` against the same parse (blocks a truncated exit-101 body's surviving pass section from misreading as a clean pass, falling back to rule 5 otherwise); rule 5 keeps the pre-M12 `last command exited N`/`gave no exit code` text when nothing parsed; (3) `finish_nudge`'s `VerifyOk` and the `VERIFY_NUDGE` release predicate both gain `Ōł¦ ┬¼(ran==0 Ōł¦ filtered_out>0)` ŌĆö an empty run no longer arms FINISH_NUDGE nor releases VERIFY_NUDGE (both previously false-greened on a 0-test exit 0, the flagship failure motivating this milestone). `RepetitionTracker` (`agent/repetition.rs`) gains a per-file cumulative S/R counter (`sr_by_file`, keyed by `status_note::normalize`) alongside the pre-existing consecutive streak ŌĆö `SR_CORRECTION` now fires on `error_streak>=2 Ōł© cum>=2`, latched per file rather than per run (so a second file's loop still gets corrected) and capped at `MAX_SR_CORRECTIONS`=3 total per run (balloon-effect guard, per the M10 arm-block precedent); a successful edit_file/write_file dispatch (`record_mutation_ok`) clears that file's counter and latch, so a loop recurring after a real fix counts as a new incident. A parallel `badargs_streak` counts consecutive `Error: invalid arguments: missing field` bodies only (deliberately not expanded to other error kinds). `SR_PERTURB_TEMPERATURE`'s trigger widens to `sr_streak()>=2 Ōł© sr_file_streak()>=2 Ōł© badargs_streak()>=2` (mechanism/latch/lifetime unchanged ŌĆö M10's decoding-layer fix now reaching more of the loops it targeted). Dispatch (`agent/mod.rs`, before the approval gate/preview so rule 2's tool swap is judged under the resulting tool) also gains a salvage reverse rule for a model-emitted `args.tool` key: the key is always stripped before dispatch ŌĆö value equal to the action's own `tool`, or an unregistered name (`finish` included, since it's loop-handled and never registry-resident) ŌåÆ key dropped with `ARGS_TOOL_KEY_NOTE`; value naming a different *registered* tool ŌåÆ the action is redispatched as that tool with `ARGS_TOOL_SWITCH_NOTE` (the model's args apparently expressed its real intent). Both notes are independent of and stack with the pre-existing `SALVAGE_NOTE` channel; `turn.salvaged` itself is never set by this rule ŌĆö the spec's original wording read as if it should be, which self-contradicted the very next paragraph (forbidding reuse of the `salvaged`-triggered `SALVAGE_NOTE` for this malformation); the spec text has been clarified to say so explicitly. `edit_file`'s dry run (`tools/edit_file.rs`) also swapped check order: `apply_edit` (which resolves the match ŌĆö 0 found, Ōēź2 locations, or exactly 1) now runs *before* the `search==replace` identical-text check, so a hallucinated (0-match) or ambiguous (Ōēź2-match, no `replace_all`) S/R-identical edit surfaces as "not found"/"N locations" rather than masquerading as the identical-text error ŌĆö `sr_error`/`SR_CORRECTION` (and its `exp_metrics.py` counterpart) now fire only when the search text genuinely exists exactly once, so M12-onward `sr_error` counts aren't directly comparable to pre-M12 batches (see `docs/baselines.md` "M12")
- session: `Session` (src/session.rs) owns chat history + the JSONL transcript together ŌĆö every push records both. `pack()` is the spec ┬¦6 budget: elide oldest tool-result bodies first, then drop oldest user+assistant pairs atomically (system prompt and final message preserved), mutating stored history in place (transcript keeps unabridged originals). Context-overflow 400 (heuristic: body contains "context") packs at a shrinking budget and retries, up to 2 shrinks per `run()` call before propagating ŌĆö takes priority over the blind 400 fallback ladder, which doesn't fire on overflow
- `--auto`: `AutoApprover` approves everything except `run_command` matching `auto_deny_patterns` (cross-platform default list ŌĆö `rm -rf`, `sudo`, `git push`, etc., configurable in TOML), blocked only in auto mode; interactive mode shows a warning line instead
- `eval` (`src/eval/`): in-process harness ŌĆö per task├Śrepeat: fixture ŌåÆ temp sandbox ŌåÆ `Agent::run` with `AutoApprover` + seed(base_seed+repeat) ŌåÆ protected paths re-synced from the fixture (deletes agent-added files ŌĆö anti reward-hack; `.cargo` is implicitly protected; the tripwire's effective check is `temp_dir/.cargo` only (the sandbox's parent IS temp_dir ŌĆö detection only, no cleanup: remove a leftover `${TMPDIR}/.cargo` manually). M7 adds change-detection snapshots (`eval/integrity.rs`): `$CARGO_HOME` config.toml/config plus `.cargo/config*` in every ancestor above canonicalized temp_dir are recorded at harness start and re-compared before each check ŌĆö any state transition aborts the harness. Still out of scope (backlog): pre-existing poisoned configs (CARGO_HOME isolation) and cargo-binary replacement (`$CARGO_HOME/bin`, `~/.rustup` ŌĆö PATH pinning/hashing)) ŌåÆ the `check` command's exit code decides pass; `check` always runs, regardless of outcome. Timeout math is clamped (`scaled_timeout`: saturating, cap 3600s ŌĆö huge `timeout_secs`/non-finite scale can't panic `from_secs_f64`); report.json snapshots the effective config (`effective_config`: base_url, temperature, context/output tokens, max_turns, command timeout, loco version) for cross-machine comparison. An LLM error aborts the whole harness; Ctrl+C writes a partial report (`interrupted: true`, exit 1). Timeouts: `agent::bounded::run_bounded` sets `ToolCtx.cancel` then waits `cancel_grace` so `run_command` can kill its process group before giving up (no orphans) ŌĆö the same helper wires `-p`'s Ctrl+C handling (exit 2). `AgentOutcome::Cancelled` is a loop-top cancel check, keeping the grace window fast. `report.json` also aggregates `passed_count`/`passed_strict_count` (passed Ōł¦ `outcome==finished`)/`false_finish_count` (`outcome==finished` Ōł¦ !passed) per task and at the top level ŌĆö all `_count`-suffixed additions, existing keys (`pass_rate`, `total_pass_rate`, per-run `passed`) unchanged (M6 ┬¦5), plus top-level `avg_duration_secs` (run-weighted mean of per-run agent durations ŌĆö excludes check/sandbox overhead, so it differs from wall-clock `duration_secs`/runs; M7 ┬¦4). `verify` (`src/eval/verify.rs`, M6 ┬¦4) is a separate LLM-free gate: each task dir is `task.toml` + `fixture/` + `solution/`, where `solution/` is a file-level overlay of only the changed/added files (no delete/rename expression); `overlay_tree` and the eval `sync_protected` restore both do read+write (NOT `fs::copy`, whose mtime preservation on macOS causes stale test-binary judging); verify rejects a `solution/` that overlaps any protected path (incl. implicit `.cargo`), and `cargo_tripwire` runs in verify too
- `tasks/`: 12 zero-dependency cargo-crate fixtures (one dir each ŌĆö `task.toml` + `fixture/` + `solution/`); all currently use `check = "cargo test"`, `protected = ["tests", "Cargo.toml"]`; `tasks/.gitattributes` pins the CRLF fixture+solution byte-exact; every fixture MUST carry a `/target` `.gitignore` ŌĆö the walker honors .gitignore without git (`require_git(false)`), and without it a post-`cargo test` bare `list_files` returns ~14KB of target/ paths as an unshrinkable final message ŌåÆ context-overflow 400. Every task carries a `solution/` reference overlay (the `--verify` gate's solvability half); answer-format judges (find-definition, count-usages) normalize before comparing ŌĆö trim ŌåÆ strip matched surrounding quotes ŌåÆ path/int canonicalize ŌĆö and self-test that ladder with fixed accept/reject cases living in the same `tests/*.rs` file (so `check` exercises the ladder on both eval and verify runs, M6 ┬¦3)
- `tasks-large/` (M8): 3 eval tasks on a shared 5-crate `inv-*` workspace fixture (10-20K LOC) ŌĆö `find-definition-large` measures locating a definition buried past the system-prompt tree's depth cap; `fix-monthly-total` measures picking the right one of two duplicate summation functions (v1/v2) to patch amid false comments/FIXMEs; `update-vat-rate` measures propagating a rate constant across 4 scattered call sites without missing one. Run `cargo run -- eval tasks-large --verify` after ANY tasks-large change (3/3 expected ŌĆö same discipline as `tasks/`). `tasks-large/README.md` sits outside `fixture/` (never copied into the sandbox, so it's model-invisible) and holds the 11-trap ledger plus the drift procedure: the base is `find-definition-large/fixture`, with `fix-monthly-total`/`update-vat-rate` maintained as `rsync` copies of it (task-specific bug/change layered on top) ŌĆö any base edit must be re-synced to both per that procedure. ## Notes
- `.superpowers/` is git-ignored scratch (subagent-driven-development progress ledger lives there); `git clean -fdx` destroys it
- Remote: `origin` = github.com/SeonggukJeong/loco.git ŌĆö push only when the user asks
- Live smoke/eval needs LM Studio with a model loaded at context length Ōēź `context_tokens` (8192) ŌĆö smaller loaded ctx makes LM Studio 400 ("n_keep > context length") mid-eval, aborting the harness. `model = ""` auto-selects the FIRST `/v1/models` entry; LM Studio sorts loaded models first, so unload the old model when switching for comparison runs. `curl localhost:1234/api/v0/models` shows loaded state + context length
- Eval measurement protocol: a full `eval tasks --repeats 3` run takes ~45-75 min per model; do NOT run cargo build/test concurrently with a live measurement ŌĆö CPU contention skews the timing-sensitive judgments. report.json nests runs per task (`tasks[].runs[]` with passed/outcome per run), not a flat run list; per-batch metric grep recipes are in docs/baselines.md "M5 Ļ▓ĮĻ│╝"
- `TtyApprover` is intentionally synchronous-blocking on `readline()`, not polled/async ŌĆö an async stdin reader left running during the prompt would steal the Ctrl+C keystroke meant for the REPL's `select!`, orphaning the reader task
