# loco — 폐쇄망 소형모델 코딩 CLI

로컬에서 서빙되는 소형 LLM(OpenAI 호환 API)으로 코딩을 지원하는 CLI.
설계 문서: `docs/superpowers/specs/2026-07-02-loco-design.md`

## 프로젝트 상태: M12 완료 · 정직한 하네스 — 검증 실질 접지 (2026-07-19)

### 무엇이 문제였나

소형 모델은 `cargo test <파일명>` 같은 명령을 자주 쓴다. 그런데 cargo는
파일명이 아니라 **테스트 이름**으로 필터링하므로, 이 명령은 테스트를 한 개도
실행하지 않고 **exit 0**을 돌려준다. 하네스는 그 exit 0을 "검증 성공"으로
읽어 모델에게 전달했고, 모델은 그것을 근거로 "테스트가 통과했습니다"라며
작업을 종료했다. 즉 **하네스 자신이 거짓 초록불을 만들어 모델에게 먹이고
있었다.** M12는 새 기능을 늘리는 마일스톤이 아니라 이 결함을 수선하는
**회귀 게이트**다(모델 통과율 개선은 목표가 아니다).

### 무엇을 고쳤나

**축 1 — 검증 신호를 테스트 실질에 접지.** libtest 요약 줄을 실제로 파싱하는
순수 모듈을 만들고(요약이 없으면 `None` → 기존 동작 유지), 그 결과를 세 곳에
흘린다: ① 0개 테스트가 실행된 명령에는 "이 exit 0은 아무것도 검증하지
않았다"는 노트를 붙이고 ② 상태선의 verification 행을 exit code 하나가 아니라
5규칙(실패한 테스트 이름 인용, 0-테스트 무효화, exit 교차검증 등)으로 렌더하며
③ FINISH_NUDGE/VERIFY_NUDGE가 공허한 실행을 "검증됨"으로 오인해 무장/해제하던
경로를 차단했다.

**축 2 — 이미 입증된 개입의 발화 사각 해소.** S/R 교정 카운터를 "연속 2회"
전용에서 **파일별 누적**으로 확장하고(중간에 다른 오류가 끼어도 리셋되지 않음,
대신 런당 발화 상한 3회로 풍선효과 방지), 온도 섭동 트리거에 BadArgs
missing-field 연속 스트릭을 추가했으며, `args` 안에 잘못 들어간 `tool` 키를
디스패치 직전에 정규화하는 salvage 역방향 규칙을 신설했다.

### 측정 결과

| | 결과 | 대조군 | 판정 |
|---|---|---|---|
| **회귀 게이트** (`tasks/` 36런) | **33/36** (엄격 32) | 33/36 (엄격 31) | 기준 `≥33/36` 충족 — **임계값에 정확히, 재측정 없이 1회 통과** |
| 관찰 (`tasks-large` 20런) | 11/20 (엄격 9) | 10/20 (엄격 8) | 게이트 아님 — 관찰 기록만 |

게이트 통과에 **여유가 없었다**는 사실을 그대로 기록해둔다. 기준값·재측정
규칙·종결 규칙은 데이터를 보기 전에 사전등록 문서에 못박았고, 게이트가 통과한
뒤에도 규칙을 되돌아보며 완화하지 않았다.

**신규 장치가 실제로 발동했다**(과거 배치에는 기능 자체가 없어 전부 0):
salvage 역방향 규칙 118회(두 배치 합계, **오발동 0건** — 전수 확인),
0-테스트 무효화 노트 2회, verification 실질 렌더 5회.

**기함이 실전에서 일한 사례**: `update-vat-rate-8` 런에서 모델이
`cargo test --package inv-report check_vat_report`를 실행해 `exit 0 / 0 tests
ran / 15 filtered out`을 받자 새 노트가 붙었고, **모델이 다음 턴에 필터를
고쳤다.** 최종 리뷰에서 평가 하네스 밖 실제 크레이트로도 재현했다.

### 정직 기록 — 확인된 것과 열어둔 것

배치 후 이상 징후 3건이 관측되어 전용 법의학으로 전수 귀속 분석했고, **셋 다
M12 신규 장치의 실패가 아님**을 확인했다:

- 신규 `stop_cause=sr` 반복정지 1건 → 새 실패가 아니라 **라벨만 이동**(대조군의
  같은 시드 런도 동일하게 실패했다). 다만 궤적이 갈린 원인은 M12가 맞다.
- S/R 회복률 하락(76.7%→64.0%) → 분포 전반의 회귀가 아니라 **위 한 런에 갇힌
  왜곡**. 그 런을 양쪽에서 빼면 대조 75.0% vs 배치 80.0%로 오히려 역전된다.
- 거짓 성공 finish 1건 → **뮤테이션 0회 상태의 자신 있는 finish**로, M12
  이전부터 있던 별개 문제. 상태선의 0-뮤테이션 발동 주기(5/10/15/20턴)를 한 턴
  차이로 비껴갔다.

**열어둔 항목 2가지**(해결했다고 적지 않는다):

1. **뮤테이션 0회 거짓 finish 갭** — 두 배치의 거짓 finish 4건 중 3건이 이
   유형이다. 실재하며 M12가 건드리지 않았고, 스펙이 이미 다음 마일스톤으로
   이월해 둔 항목이다.
2. **S/R 오류문에 추가한 문장의 역효과 가설** — "파일은 수정되지 않았고 네
   search 텍스트가 그대로 남아 있다"는 확인이, 오히려 모델에게 "내 search는
   맞다"는 확신을 줘 같은 호출 재제출을 강화했을 가능성. **n=1이라 입증되지
   않았으므로 가설로만 기록**하고 관찰을 계속한다.

상세는 `docs/baselines.md` "M12 실험 — 회귀 게이트" 절과
`docs/experiments/2026-07-18-honest-harness/report.md`(사전등록 문서는 같은
디렉토리의 `pre-registration.md`).

## 시작하기

1. LM Studio(또는 Ollama, llama.cpp server 등)에서 모델을 로드하고 서버 시작
   - LM Studio 기본 주소 `http://localhost:1234/v1` 는 설정 없이 바로 동작
2. 실행:

   ```
   cargo run                 # 대화형 에이전트 REPL
   cargo run -- -p "질문"    # 단발 실행 (답변만 stdout, 종료코드 0/1/2)
   cargo run -- --auto       # 확인 게이트를 자동 승인 (REPL/-p 모두 사용 가능)
   ```

## 사용법

REPL에 입력한 내용은 에이전트가 처리한다 — 모델이 read_file/list_files/grep로
프로젝트를 조사하고 write_file/edit_file/run_command로 직접 수정·실행까지 한다.
답은 마지막에 한 번에 출력된다 (`finish`). 진행 중 Ctrl+C 로 취소할 수 있다.

파일 수정이나 명령 실행처럼 상태를 바꾸는 동작은 실행 전에 미리보기(diff 또는
명령어)를 보여주고 `적용할까요? [y/N]` 확인을 거친다. `--auto`를 주면 이 확인을
전부 건너뛰고 자동 승인하되, `rm -rf`·`sudo`·`git push` 등 위험한 명령 패턴은
자동 모드에서만 별도로 차단된다(대화형에서는 경고만 표시하고 계속 진행).
`-p` 단발 실행은 `--auto` 없이는 확인을 띄우지 않고 그냥 거부한다(비대화형이므로).

대화 내용은 실행마다 `.loco/sessions/*.jsonl`에 한 줄씩 기록된다(최선 노력 —
기록 실패가 에이전트를 멈추지 않는다). 히스토리가 컨텍스트 예산을 넘으면
자동으로 절삭한다(오래된 툴 결과 생략 → 오래된 질문/답 쌍 제거 순서) — 더는
`/clear`로 직접 비울 필요가 없다.

- `/chat <메시지>` — 에이전트 없이 모델과 바로 스트리밍 대화 (빠른 질문용)
- `/clear` — 히스토리 초기화(수동으로 비우고 싶을 때만; 컨텍스트 관리 목적으로는
  불필요 — 위 자동 절삭 참고)
- `/config`, `/help`, `/quit`

`-p` 모드 종료 코드: `0` 정상(finish), `1` 에러(연결 실패·파싱 실패),
`2` 최대 턴 도달·같은 툴 호출 반복으로 조기 종료, 또는 Ctrl+C로 중단(실행 중이던
명령의 자식 프로세스까지 정리한 뒤 종료). 진행 표시는 stderr로 가므로 stdout만
파이프하면 답변만 남는다.

## 평가 하네스 (eval)

과제 세트를 돌려 모델의 코딩 성공률을 측정한다. 세트는 둘이다 — `tasks/`(소형
크레이트 12과제)와 `tasks-large/`(M8 신설, 5크레이트 워크스페이스 3과제 — 함정
대장·드리프트 절차는 `tasks-large/README.md`):

```
cargo run -- eval tasks/                          # 과제당 1회
cargo run -- eval tasks/ --repeats 3 --seed 0      # 과제당 3회 반복 (시드 0/1/2)
cargo run -- eval tasks/ --timeout-scale 2.0       # 느린 머신 — 모든 타임아웃 ×2
cargo run -- eval tasks/ --verify                  # 판정기 게이트 (LLM 없이) — 아래 참고
cargo run -- eval tasks-large/ --repeats 3         # 대형 저장소 트랙도 동일 사용법
cargo run -- eval tasks-large/ --filter fix-monthly-total --repeats 10   # 과제 필터 (M10)
```

`--filter`는 이름 정확 일치로 과제를 골라 돌린다(반복 지정 가능, `--verify`와 조합
가능). 전체 정의를 로드·검증한 **뒤** 선별하며, 어느 값이든 일치하는 과제가 없으면
오류다 — 오타가 배치를 조용히 축소한 채 완주하는 사고를 막는다(사전등록 실험 규율).

`--verify`는 모델 없이 판정기 자체를 검증하는 메타테스트다(M6): 과제마다 원본 픽스처에서
`check`가 **실패**(변별성)하고 `solution/`을 덮은 뒤 **통과**(해결가능성)하는지 확인해, 둘 다
만족할 때만 종료 코드 0. `tasks/`를 손댈 때마다 돌린다. `--repeats`/`--seed`와는 배타.

과제마다: 픽스처를 임시 샌드박스에 복사 → `--auto`와 같은 권한으로 에이전트 실행
→ `protected`로 지정한 경로를 픽스처 원본으로 되돌리고(에이전트가 건드렸다면
그 흔적까지 지움 — 보상 해킹 방지) → `check` 명령의 종료 코드로 통과/실패를
가른다. `check`는 에이전트 실행이 중간에 실패했어도 항상 돈다.

결과는 통과율 표로 stdout에 출력되고, `./.loco/eval/<타임스탬프>/report.json`
(전체 리포트)과 과제×반복별 실행 기록 `run-<과제>-<반복>.jsonl`로도 저장된다.
행동 지표(S/R 오류·교정 발동·회복률·정지 원인 등)는
`python3 scripts/exp_metrics.py .loco/eval/<타임스탬프>`로 추출한다(M10, 표준
라이브러리만 사용 — `--selftest`로 자기검증).

종료 코드: `0` 정상 완료(통과율과 무관), `1` 하네스 에러(서버 다운, 잘못된 과제
정의) 또는 Ctrl+C로 중단되어 부분 리포트만 남긴 경우.

## 빌드 노트

TLS는 rustls+ring 고정 — OpenSSL도 aws-lc-sys(cmake/NASM)도 그래프에 없어
Windows 폐쇄망에서 `cargo vendor` 후 Rust 툴체인만으로 빌드된다.

## 설정 (선택)

`./.loco/config.toml` (프로젝트) 또는 전역 설정 파일. 전역 경로는 OS마다 다르며
(macOS는 `~/Library/Application Support/dev.loco.loco/config.toml`, Linux는
`~/.config/loco/config.toml`) REPL의 `/config` 명령으로 확인할 수 있다:

```toml
base_url = "http://localhost:1234/v1"
model = ""            # 비우면 서버의 첫 모델 자동 선택
temperature = 0.1
context_tokens = 8192
max_output_tokens = 2048
max_turns = 25
command_timeout_secs = 60
```
